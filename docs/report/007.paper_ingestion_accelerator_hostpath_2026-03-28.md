## DSA-2LM: A CPU-Free Tiered Memory Architecture with Intel DSA

- **Venue / Year**: USENIX ATC 2025
- **Local PDF**: `papers/top_tier_pdfs/dsa2lm_atc25.pdf`
- **Source URL**: https://www.usenix.org/system/files/atc25-liu-ruili.pdf

### Repo summary

DSA-2LM is one of the closest direct systems precedents for this repo because it embeds Intel DSA into a higher-level memory policy rather than treating DSA as a standalone primitive. Its main value here is to show that DSA can produce end-to-end system benefit when the surrounding control policy is designed around it.

### Technique

- **Direct in-kernel DSA page migration**: bypass the generic DMA interface and invoke DSA directly for page migration.
- **Adaptable concurrent migration**: handle both 4 KB and 2 MB migration in one mixed-page path.
- **Batch + multi-work-queue scheduling**: batch small-page copies and split huge pages across multiple DSA work queues.
- **Threshold-driven aggressive migration loop**: shorten migration intervals because copy work is offloaded.

### Why it matters here

It helps the repo argue that DSA work should be evaluated above microbenchmarks. That aligns with the repo's move from `dsa-stdexec` measurements toward higher-level deployment questions such as RPC and memory movement policy.

### Gap relative to this repo

DSA-2LM is about tiered-memory page migration, not composable async abstractions or per-operation framework overhead. It is a direct accelerator paper, but not yet a direct framework paper.

### RQ tags

- **RQ1**: strong evidence that DSA matters in a real higher-level path
- **RQ2**: medium evidence that policy and framework structure must surround the hardware
- **RQ5**: medium support for deployment-facing accelerator integration

### Extracted opening text

> “With the advent of Intel Data Streaming Accelerator (DSA), a CPU-free hardware to move data between memory regions, data movement can be up to 4× faster than a single CPU core.”

> “We have developed DSA-2LM, a new tiered memory system that adaptively integrates DSA into page migration.”

> “Experimental results show ... DSA-2LM can achieve 20%, 30% and 16% performance improvement under real-world applications.”

## How to Copy Memory? Coordinated Asynchronous Copy as a First-Class OS Service

- **Venue / Year**: SOSP 2025
- **Local PDF**: `papers/top_tier_pdfs/copier_sosp25_preprint.pdf`
- **Source URL**: https://ipads.se.sjtu.edu.cn/_media/pub/sosp25-copier-preprint.pdf

### Repo summary

Copier treats copy as a first-class asynchronous OS service with overlap, hardware-capability use, and global optimization opportunities. In this repo's literature stack, it is a strong abstraction precedent for elevating copy offload above a narrow device API.

### Technique

- **Copy as a first-class OS service**: make copy a coordinated system service rather than just a library call.
- **Asynchronous copy with `amemcpy` / `csync`**: overlap computation with copy while preserving use-point synchronization semantics.
- **Queue-based segmented copy pipeline**: track per-segment completion and allow early consumption of ready pieces.
- **Piggybacked heterogeneous dispatcher + copy absorption**: coordinate AVX and DMA copy units and eliminate redundant intermediate copies.

### Why it matters here

It directly supports the repo's instinct that copy should be surfaced as a structured service or abstraction layer, not only as individual accelerator submissions. That is especially relevant to `accel-rpc`, where copy is one stage in a broader pipeline.

### Gap relative to this repo

Copier is OS-service-centric and copy-specific. It does not address the broader multi-stage async pipeline of copy, checksum, compression, and transport integration that the repo is after.

### RQ tags

- **RQ1**: medium support for improving the practical DSA path
- **RQ2**: strong support for designing higher-level async copy abstractions
- **RQ3**: medium relevance where copy dominates RPC paths
- **RQ5**: medium support for integrating copy into larger deployment-facing systems

### Extracted opening text

> “This paper argues for copy as a first-class OS service, offering three key benefits: with the asynchronous copy abstraction provided by the service, applications can overlap their execution with copy; the service can effectively utilize hardware capabilities to enhance copy performance; the service’s global view of copies further enables holistic optimization.”

> “We introduce Copier, a new OS service of coordinated asynchronous copy, to serve both user-mode applications and OS services.”

> “Evaluations show that Copier achieves up to a 1.8× speedup for real-world applications like Redis and a 1.6× improvement over zIO.”

## Understanding the Host Network

- **Venue / Year**: SIGCOMM 2024
- **Local PDF**: `papers/top_tier_pdfs/understanding_host_network_sigcomm24.pdf`
- **Source URL**: https://www.cs.cornell.edu/~ragarwal/pubs/understanding-the-host-network.pdf

### Repo summary

Understanding the Host Network reframes the host as a network of contention domains and gives a credit-based way to reason about data movement inside the machine. For this repo, it is the main external justification for treating the host-to-accelerator path as a first-class systems bottleneck rather than just a local implementation detail.

### Technique

- **Host-network abstraction**: model the host as a network of processor, memory, and peripheral interconnects.
- **Domain-by-domain credit-based flow control**: analyze the host through contention domains with their own credits and latency.
- **Blue/red contention regime analysis**: distinguish cases where host contention harms different traffic classes differently.
- **Latency-throughput domain model**: explain host-path limits using flow-control style reasoning instead of only raw bandwidth numbers.

### Why it matters here

The repo's positioning repeatedly claims that prior work emphasizes the NIC-to-host path while this work emphasizes the host-to-accelerator path. This paper gives the clearest external framework for that argument.

### Gap relative to this repo

It is primarily a characterization and conceptual-model paper. It does not directly propose a DSA-oriented framework or RPC data path.

### RQ tags

- **RQ1**: medium support for why host-path bottlenecks matter to measured DSA throughput
- **RQ4**: strong support for the repo's host-path generalization and positioning

### Extracted opening text

> “The host network integrates processor, memory, and peripheral interconnects to enable data transfer within the host.”

> “We present domain-by-domain credit-based flow control, a conceptual abstraction to study the host network.”

> “The host network performs flow control over different domains ... different applications may traverse different domains, and may thus observe different performance degradation upon contention within the host network.”

## Host Congestion Control

- **Venue / Year**: SIGCOMM 2023
- **Local PDF**: `papers/top_tier_pdfs/hostcc_sigcomm23.pdf`
- **Source URL**: https://www.cs.cornell.edu/~ragarwal/pubs/hostcc.pdf

### Repo summary

hostCC argues that congestion is not only a network-fabric phenomenon and introduces host-local congestion signals plus a host-local response policy. In this repo, it matters as further evidence that control-path policy inside the host can dominate observed performance.

### Technique

- **Host-local congestion signals**: measure congestion inside the host network, not just in the external fabric.
- **Sub-RTT host-local response**: react faster than RTT-scale network control when host resources are congested.
- **Two-level control architecture**: combine host-resource allocation with network-resource allocation.
- **Protocol-compatible integration**: insert host-congestion handling into the Linux stack without changing applications or NIC hardware.

### Why it matters here

It complements the repo's batching-regime thesis by showing that once operations become frequent enough, host-local contention and control response need to be treated explicitly.

### Gap relative to this repo

hostCC is about congestion response rather than accelerator abstraction or framework cost. It is therefore a positioning and systems-context paper more than a direct mechanism source.

### RQ tags

- **RQ1**: medium support for host-path bottleneck awareness
- **RQ4**: medium support for the repo's broader host-to-accelerator framing

### Extracted opening text

> “The conventional wisdom in systems and networking communities is that congestion happens primarily within the network fabric. However ... high-bandwidth access links and relatively stagnant technology trends for resources within hosts have led to emergence of host congestion.”

> “hostCC collects host congestion signals that capture the precise time, location, and reason for host congestion.”

> “hostCC introduces a sub-RTT granularity host-local congestion response that uses congestion signals to allocate host resources between network traffic and host-local traffic.”

## High-throughput and Flexible Host Networking for Accelerated Computing (ZeroNIC)

- **Venue / Year**: OSDI 2024
- **Local PDF**: `papers/top_tier_pdfs/zeronic_osdi24.pdf`
- **Source URL**: https://www.usenix.org/system/files/osdi24-skiadopoulos.pdf

### Repo summary

ZeroNIC separates data and control paths to deliver both throughput and flexibility in host networking for accelerated computing. For this repo, the paper is valuable less as a direct DSA reference than as an architectural analogy for keeping the hot data path lean while reserving richer control logic for a colder path.

### Technique

- **Physical separation of data and control paths**: keep payload transfer and protocol logic on different paths.
- **Header/payload split-merge pipeline**: DMA payloads directly while processing headers separately.
- **Per-flow memory-segment bookkeeping**: maintain enough mapping state to place data correctly under reordering and retransmission.
- **Protocol-agnostic zero-copy endpoint design**: keep the fast path independent of whether buffers live in CPU, GPU, FPGA, or another memory domain.

### Why it matters here

That separation maps well onto the repo's own concern that the fast path should stay minimal while scheduling, policy, and integration logic may live elsewhere. It is a useful deployment-facing comparison point.

### Gap relative to this repo

ZeroNIC uses custom NIC co-design rather than commodity on-die accelerators. It is therefore adjacent architecture rather than a direct implementation template.

### RQ tags

- **RQ2**: medium support for separating hot-path and control-path responsibilities
- **RQ3**: medium relevance to accelerator-aware RPC deployment
- **RQ5**: strong architectural analogue for deployment-facing accelerator transports

### Extracted opening text

> “Modern network hardware is able to meet the stringent bandwidth demands of applications like GPU-accelerated AI. However, existing host network stacks offer a hard tradeoff between performance and flexibility.”

> “The key idea in our design is the physical separation of the data path ... and the control path.”

> “ZeroNIC achieves RDMA-like throughput while maintaining the benefits of robust protocols like TCP under various network perturbations.”

## My CXL Pool Obviates Your PCIe Switch

- **Venue / Year**: HotOS 2025
- **Local PDF**: `papers/top_tier_pdfs/cxl_nic_candidate_arxiv2503.23611.pdf`
- **Source URL**: https://arxiv.org/pdf/2503.23611
- **Note**: best-matching public candidate for the earlier “CXL-NIC” thread, but not literally titled “CXL-NIC”.

### Repo summary

This paper broadens the story from accelerator micro-mechanisms to interconnect organization by arguing that PCIe-device pooling can be implemented in software using CXL memory pools. In the repo's literature review, it serves as a bridge to the MMIO, interconnect, and disaggregation side of the argument.

### Technique

- **Software PCIe pooling on top of CXL memory pools**: use a CXL pod as the substrate for PCIe device pooling.
- **CXL-resident I/O buffers**: place TX/RX or device I/O buffers in shared CXL memory.
- **Software coherence for non-coherent shared CXL memory**: manage coherence explicitly because current pooled memory is not cross-host coherent.
- **Pooling orchestrator + agents**: use a control plane to map devices to hosts, monitor health, and rebalance or fail over usage.

### Why it matters here

The repo already treats hardware improvements to the host path as complementary to its batching-based software story. This paper gives that broader deployment and interconnect perspective.

### Gap relative to this repo

It is not directly a DSA, RPC, or accelerator-framework paper, and the title remains only a best-match candidate for the earlier CXL-NIC thread. Its role here is therefore contextual rather than central.

### RQ tags

- **RQ4**: medium support for the broader generalization and positioning story
- **RQ5**: medium relevance to deployment paths involving disaggregated or pooled devices

### Extracted opening text

> “Pooling PCIe devices across multiple hosts offers a promising solution to mitigate stranded I/O resources, enhance device utilization, address device failures, and reduce total cost of ownership.”

> “This paper argues that PCIe device pooling can be effectively implemented in software using CXL memory pools.”

> “Once CXL pools are in place, they can serve as a building block for pooling any kind of PCIe device.”

## Why these papers were grouped together

These papers anchor the repo’s broader host-to-accelerator and intra-host positioning. DSA-2LM and Copier are the closest direct matches for CPU-free or asynchronous data movement. Understanding the Host Network and hostCC explain why the host path itself is a first-class bottleneck. ZeroNIC shows how host networking can be redesigned by separating data and control paths. The CXL pool paper broadens the MMIO / interconnect / device-disaggregation story that the repo already connects to its batching-regime thesis.
