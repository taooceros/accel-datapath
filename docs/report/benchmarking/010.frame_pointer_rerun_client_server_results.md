# Frame-pointer rerun with client/server profiling

**Date**: 2026-04-08

## Scope

Reran the representative profiling lane from `docs/report/benchmarking/009.tonic_profile_bounded_matrix_results.md` after confirming frame pointers are enabled in `.cargo/config.toml`:

```toml
[build]
rustflags = ["-C", "target-cpu=native", "-C", "force-frame-pointers=yes"]
```

Unlike the earlier capture set, this rerun records **both client-side and server-side** perf/flamegraph artifacts for each representative point.

## Setup

- transport: loopback
- run mode: split `server` + `client`
- server core: `0`
- client core: `1`
- warmup: `3000 ms`
- measure window: `10000 ms`
- output dir: `results/tonic/2026-04-08-frameptr/`

Representative points rerun:

1. `small_struct_off_single` — `256 B`, structured, compression `off`, concurrency `1`, runtime `single`
2. `medium_struct_off_multi` — `4 KiB`, structured, compression `off`, concurrency `128`, runtime `multi`
3. `large_struct_off_multi` — `1 MiB`, structured, compression `off`, concurrency `512`, runtime `multi`
4. `large_struct_on_multi` — `64 KiB`, structured, compression `on`, concurrency `128`, runtime `multi`

Artifacts captured per point:

- client perf stat + JSON summary
- client perf record + perf report
- client flamegraph
- server perf stat
- server perf record + perf report
- server flamegraph

`taskset: Interrupt` appears in the server perf-stat stderr sidecars because the profiled server was terminated with `SIGINT` after the matching client finished; the perf-stat outputs were still written successfully.

## Client rerun vs. prior loop2 capture

The frame-pointer rerun preserves the same client-side regimes reported in `results/tonic/2026-04-01-loop2/`.

| point | prior IPC | rerun IPC | prior key bottleneck | rerun key bottleneck |
| --- | ---: | ---: | --- | --- |
| `small_struct_off_single` | `2.1` | `2.2` | frontend-bound tiny-stack control path | same; `memmove` + H2/HPACK helpers remain hottest self entries |
| `medium_struct_off_multi` | `1.8` | `1.8` | copy/allocator + memory pressure | same; `memmove`, `_int_malloc`, free/allocator paths still lead |
| `large_struct_off_multi` | `0.3` | `0.3` | movement dominated | same; `__memmove_avx512_unaligned_erms` stays dominant |
| `large_struct_on_multi` | `4.5` | `4.6` | compression dominated | same; `miniz_oxide::deflate::core::compress_inner` stays dominant |

The main conclusion from report 009 therefore survives the frame-pointer-enabled rerun: enabling frame pointers did not change the substantive client-side attribution story.

## Findings

1. **Server-side profiles reinforce the same regime split already visible on the client.** Small unary server work is still mostly control-path/frontend pressure with modest copy/header costs.
2. **Medium uncompressed server work is still allocator/copy dominated.** The server report stays led by `memmove` plus malloc/free paths, matching the client-side “buffer lifecycle first” conclusion.
3. **Large uncompressed runs remain fundamentally movement dominated on both sides.** The client keeps `memmove` at `71.17%` self time; the server still shows `56.79%` self time in the same routine, with extra buffer-drop and mutex costs layered on top.
4. **Compression-heavy structured runs remain CPU-transform dominated on both sides.** The client and server both spend about two-thirds of self time in `miniz_oxide::deflate::core::compress_inner`, so the compression-offload story is unchanged.
5. **The frame-pointer-enabled rerun successfully produced readable client and server perf reports plus flamegraphs.** This is sufficient to carry forward the matched-comparison / internal-timer work with symmetric profile coverage.
