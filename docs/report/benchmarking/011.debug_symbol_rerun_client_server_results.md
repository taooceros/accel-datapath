# Debug-symbol rerun with client/server profiling

**Date**: 2026-04-08

## Scope

Repeated the representative profiling rerun from `docs/report/benchmarking/010.frame_pointer_rerun_client_server_results.md`, but rebuilt the release binary with debuginfo enabled while keeping frame pointers on.

Build mode used:

- frame pointers from `.cargo/config.toml`
- release debuginfo via `CARGO_PROFILE_RELEASE_DEBUG=true`

Binary verification:

- `readelf -S accel-rpc/target/release/tonic-profile` shows `.debug_info`, `.debug_abbrev`, `.debug_line`, and `.debug_str`

## Setup

- transport: loopback
- run mode: split `server` + `client`
- server core: `0`
- client core: `1`
- warmup: `3000 ms`
- measure window: `10000 ms`
- output dir: `results/tonic/2026-04-08-frameptr-debuginfo/`

Representative points rerun:

1. `small_struct_off_single` — `256 B`, structured, compression `off`, concurrency `1`, runtime `single`
2. `medium_struct_off_multi` — `4 KiB`, structured, compression `off`, concurrency `128`, runtime `multi`
3. `large_struct_off_multi` — `1 MiB`, structured, compression `off`, concurrency `512`, runtime `multi`
4. `large_struct_on_multi` — `64 KiB`, structured, compression `on`, concurrency `128`, runtime `multi`

Artifacts captured per point:

- client perf stat + JSON summary
- client perf record + perf report
- client flamegraph
- server perf stat
- server perf record + perf report
- server flamegraph

As in the previous rerun, `taskset: Interrupt` appears in server perf-stat stderr sidecars because the profiled server is terminated after the matching client workload completes; the stat outputs were still produced successfully.

## What improved with debuginfo

The primary gain is **profile readability**, not a changed bottleneck conclusion.

### Flamegraph richness

Flamegraph SVGs became substantially larger, consistent with richer symbolization:

### More specific hot symbols surfaced

Compared with the frame-pointer-only rerun, the debuginfo build surfaces a few more concrete Rust-side helpers near the hot path, for example:

- `small_struct_off_single` client: `<bytes::bytes_mut::BytesMut as bytes::buf::buf_mut::BufMut>::put_slice`
- `medium_struct_off_multi` server: `<h2::server::Peer as h2::proto::peer::Peer>::convert_poll_message`
- `large_struct_off_multi` client: `tonic_profile::make_payload`
- `large_struct_on_multi` client: `miniz_oxide::deflate::core::HuffmanOxide::optimize_table`

These do not overturn the high-level attribution, but they make the profiles more useful for slide-level and implementation-level discussion.

## Findings

1. **Release debuginfo improves symbol richness materially.** The flamegraphs are much denser and show more specific Rust-side helpers and instantiated frames.
2. **The underlying attribution story is stable.** Adding debuginfo did not move the experiment into a different bottleneck regime.
3. **Client/server symmetry remains intact.** Both sides still tell the same broad story for each representative point.
4. **The debuginfo rerun is the better artifact set for future narrative and implementation work.** If a new slide or deep-dive report needs representative flamegraphs, prefer `results/tonic/2026-04-08-frameptr-debuginfo/` over the frame-pointer-only set.
