## Findings

### 1. Small unary RPCs still favor the single-thread runtime, but the gap is modest under strict split-core pinning

At `256 B` and concurrency `1`:

- single-thread: `27.48k rps`, p50 `35 us`
- multi-thread: `20.88k rps`, p50 `47 us`

Single-thread remains best for tiny localhost RPCs, but the split-core two-process setup removes the extreme multi-thread collapse seen in the earlier looser setup.

### 2. Medium payloads clearly benefit from multi-thread execution

At `4 KiB` and concurrency `128`:

- throughput: `66.46k rps` vs `34.27k rps` for single-thread (**~1.94x**)
- p50 latency: `1.61 ms` vs `3.71 ms`
- p99 latency: `4.50 ms` vs `6.77 ms`

This is the clearest stable benefit in the matrix. The medium workload has enough outstanding work to amortize runtime overhead.

### 3. Medium and large uncompressed runs are dominated by memory movement and allocation behavior

The medium multi-thread perf report is led by `memmove`, malloc/free, raw-vector growth, `BytesMut` reserve paths, and mutex contention inside h2 polling. The large uncompressed cases are strongly backend- and memory-bound.

That means the current hot path is not just "Tokio overhead" or "prost cost". Buffer growth, copy volume, and allocator churn are first-order contributors in the uncompressed path.

### 4. Under split-core pinning, software compression becomes the dominant CPU cost and hurts end-to-end throughput

Direct `64 KiB repeated` comparison at concurrency `128`, runtime `multi`:

- compression **off**: `12.74k rps`, `796.42 MiB/s`, p99 `23.11 ms`
- compression **on**: `7.29k rps`, `455.82 MiB/s`, p99 `27.44 ms`

Relative to compression-off, compression-on changed by:

- throughput: **-42.8%**
- bandwidth: **-42.8%**
- p99 latency: **+18.7%**

Perf explains the reversal: the compression-on case is overwhelmingly dominated by `miniz_oxide::deflate::core::compress_inner`, with additional copy and checksum work behind it. Once the client is restricted to a single core, compression no longer hides transport or movement cost; it becomes the bottleneck.

### 5. Large-payload tail latency remains significant, especially for structured payloads

At `64 KiB structured`, concurrency `512`, runtime `multi`:

- throughput: `768.23 MiB/s`
- p95: `74.20 ms`
- p99: `106.18 ms`
- max: `430.59 ms`

The repeated uncompressed payload is materially better in both median and tail latency, which suggests payload content and buffering behavior matter even before compression is enabled.

## Updated shortlist for next-step deep profiling

1. **buffer growth / buffer lifecycle / copies** in medium and large uncompressed runs
2. **allocator behavior** along the tonic/h2/bytes path
3. **runtime topology effects** for tiny RPCs and for medium concurrency saturation
4. **compression placement and implementation** only after accounting for the strong single-core bottleneck seen here

## Limitations

- `tonic-profile` still reports end-to-end metrics only; internal encode/decode/compress timers are not implemented yet.
- The required subtraction variants beyond compression on/off are not yet available, so allocation and copy attribution still relies on perf evidence rather than controlled variants.
- Flamegraphs were generated without release debuginfo, so symbol fidelity can be improved in a follow-up run.
- `cargo flamegraph` reported some lost chunks during capture, so the SVGs are useful for shape, not exact percentages.

## Conclusion

With client and server split into two localhost processes pinned to separate cores, the current first-pass tonic profile is now reproducible and materially different from the earlier looser run:

- single-thread still wins for tiny unary RPCs,
- multi-thread strongly helps the medium uncompressed workload,
- large uncompressed workloads are dominated by memory movement and allocation,
- and software compression is no longer an obvious win once the client is constrained to one core.

For the next phase, buffer/copy/allocation analysis is the strongest follow-up lane, while compression should be revisited only with better placement control or more available CPU.
