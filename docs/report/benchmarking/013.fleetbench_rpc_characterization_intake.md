# FleetBench RPC characterization intake

**Date**: 2026-04-12
**Type**: source inspection and incorporation recommendation

## Question

The next characterization-design question was whether Google FleetBench's RPC benchmark mechanism should influence the repo's Tonic characterization mechanism.

Short answer: **yes — and the repo should explicitly do both levels of characterization.** FleetBench is useful for lower-level CPU/code-path understanding: realistic protobuf message shape, closed-loop outstanding-RPC generation, request/response delay distributions, peer/connection topology, and representative gRPC/protobuf instruction behavior. The repo's Tonic stage attribution remains useful for higher-level decomposition: encode/decode, copy/buffer lifecycle, compression, framing/transport, runtime/queueing, and tail behavior.

The right incorporation is therefore **not** "FleetBench instead of decomposition". It is a two-level mechanism:

1. **CPU/code-path characterization**: FleetBench-style realistic message programs and load shaping, evaluated with `perf stat`, flamegraphs, IPC/topdown counters, cache/memory indicators, and throughput/latency.
2. **Stage decomposition**: Tonic-specific timers and software controls to map observed CPU behavior back to actionable RPC buckets and offload-readiness decisions.

## What FleetBench RPC is doing

FleetBench describes its RPC workload as instruction-focused with a strong data-driven aspect and built on gRPC. The current suite also emphasizes production-derived distributions and realistic concurrent workloads.

Mechanically, the RPC benchmark has two related entry points:

1. `rpc_benchmark`, a Google Benchmark wrapper. It registers one benchmark per `program` in `kPrograms`, defaults the suite filter to `BM_RPC_Fleet`, starts a local gRPC server on an OS-selected port, and runs one outstanding unary RPC stream against it.
2. `rpcperf_main`, a CLI-style driver that exposes more workload knobs: ports, workers, max outstanding RPCs, peers, max peers, connections per peer, request/response delay distributions, runtime duration, program selection, logging, and gRPC TCP Tx zerocopy flags.

The core request path is closed-loop:

- the client pre-generates a vector of request messages for the selected `program`,
- each outstanding stream sends one async unary RPC,
- when the completion callback fires, it records latency, optionally sleeps according to a request-delay distribution, and sends the next RPC if `keep_running()` is true,
- the server returns a randomly selected pre-generated response message and optionally sleeps according to a response-delay distribution.

## Refined framing: two-level characterization

The repo should treat FleetBench-style CPU characterization and Tonic stage decomposition as complementary, not competing:

| Level | Question | Mechanism | Output |
|---|---|---|---|
| CPU/code-path | What does realistic gRPC/protobuf execution look like on this CPU? | FleetBench-like proto programs, closed-loop outstanding RPCs, connection topology, delay distributions, `perf stat`, flamegraphs | IPC/topdown, memory/cache behavior, hot symbols, realistic protobuf-shape sensitivity |
| Stage decomposition | Which RPC stage is responsible and is it offload-relevant? | Tonic codec/body timers, instrumentation-off controls, pooled/copy-minimized variants, compression controls | encode/decode/copy/buffer/compression/runtime attribution and offload-readiness ranking |

This means a future `tonic-profile` run should usually have two paired forms:

1. **CPU run**: instrumentation off, realistic proto-shape mode, external `perf`/flamegraph on representative points.
2. **Decomposition run**: instrumentation on, matched configuration, lower-overhead counters where possible, used to map CPU hot paths to stage buckets.

## Required bridge: mapping high-level buckets to low-level CPU evidence

The two levels should not remain separate reports. Each high-level decomposition bucket should have a corresponding low-level CPU evidence plan, so the repo can say both **where in the RPC stack** the time belongs and **why the CPU is spending time there**.

| High-level Tonic bucket | Low-level CPU question | Evidence to collect | Expected low-level signatures | Interpretation rule |
|---|---|---|---|---|
| Serialization / deserialization | Is prost walking many fields, varints, nested messages, or string/bytes fields? | stage timers for encode/decode; flamegraph symbols under prost/prost-derive generated encode/decode; instruction count; IPC; branch miss rate | higher instructions/RPC; retiring-heavy or branch-sensitive profiles; hot generated message encode/decode frames | If encode/decode timers rise with realistic proto shape but not bytes payload size, treat as schema traversal, not copy offload. |
| Copy / buffer lifecycle | Is time spent moving bytes or growing buffers? | body/buffer timers; `perf topdown` memory-bound/backend-bound; flamegraph `memmove`, allocator, `BytesMut`, `RawVec`; LLC/cache metrics where available | low IPC, high backend/memory-bound, hot `memmove`/allocator/grow paths | If CPU evidence is memory-bound and pooling/copy-min controls reduce it, classify as software buffer-policy before hardware offload. |
| Compression / decompression | Is compression compute dominating or hidden by memory movement? | compression/decompression timers; flamegraph compression library frames; IPC/topdown retiring vs memory-bound; compression ratio | high retiring and hot gzip/miniz/zstd frames for compute-bound compression; random payloads show large compression time with poor ratio | Only treat as offload candidate for compressible regimes where transform remains large after software controls. |
| Framing / transport glue | Is cost in gRPC/h2/hyper framing rather than payload transforms? | frame-header/body-handoff timers; flamegraph h2/hyper/tonic body frames; instructions/RPC at small payloads | fixed per-RPC instruction overhead; hot h2/tonic state-machine frames; weak scaling with payload size | If cost is fixed per RPC and not payload-size sensitive, classify as control-path/runtime work, not DSA/IAX byte-path work. |
| Runtime / queueing / scheduling | Is tail/throughput dominated by async scheduling and load shape? | instrumentation-off throughput; p50/p95/p99; async-bench; flamegraph tokio scheduler/waker frames; context switches/migrations; closed-loop outstanding-RPC metadata | high tail variance; hot scheduler/waker frames; sensitivity to runtime mode, outstanding RPCs, channel count, delay distributions | If changing load shape/runtime shifts winner without changing stage timers, report it as queueing/runtime policy. |
| Application/service delay | Is observed latency intentionally induced by server/client delay distributions? | metadata for client/server delay distributions; queue depth/outstanding RPCs; latency histograms | p99 tracks configured service-time distribution; stage timers remain small | Do not attribute configured delay to transport or payload-path cost. |
| Request/response asymmetry | Is one endpoint or direction responsible? | split client/server stage snapshots; request/response serialized sizes; endpoint flamegraphs | server response-heavy profiles for Fleet/P10-like response; client request-heavy profiles for P3/P7-like request | Require endpoint split before claiming a direction-specific offload target. |

Practically, every important run should produce a **paired interpretation row** like:

```text
High-level bucket: copy / buffer lifecycle
Low-level evidence: 0.5 IPC, 63% memory-bound, memmove/BytesMut growth hot, pooling changes throughput
Conclusion: memory movement and buffer policy dominate; improve software buffer lifecycle before claiming DSA offload.
```

This mapping is the bridge from decomposition to CPU characterization: stage timers identify the bucket, while CPU counters/flamegraphs explain the underlying machine behavior.

## The most relevant mechanisms

### 1. Program-based realistic protobuf shape

FleetBench does not just send a flat byte vector. It uses generated protobuf programs:

- `RequestMessage` and `ResponseMessage` are top-level `oneof` wrappers over `P0` ... `P10` program-specific messages.
- `combo.h` maps program names `0` ... `9` and `Fleet` to program indices; `Fleet` maps to `P10`.
- each program has request and response sides;
- each side has three message variants;
- each message variant has four generated setter variants;
- the client/server pre-generate `kMaxMessagesPerProgram * kMaxSettersPerMessage = 3 * 4 = 12` request or response buffers per program.

A quick static summary of the generated proto files shows that the programs vary meaningfully:

| program | request fields | request strings | request repeated | response fields | response strings | response repeated |
|---|---:|---:|---:|---:|---:|---:|
| P0 | 208 | 13 | 8 | 348 | 36 | 20 |
| P1 | 73 | 16 | 5 | 122 | 24 | 12 |
| P2 | 338 | 19 | 51 | 183 | 42 | 14 |
| P3 | 2396 | 529 | 229 | 303 | 13 | 39 |
| P4 | 223 | 63 | 21 | 9 | 0 | 1 |
| P5 | 895 | 24 | 92 | 249 | 18 | 25 |
| P6 | 38 | 5 | 11 | 39 | 7 | 1 |
| P7 | 2062 | 737 | 167 | 156 | 25 | 36 |
| P8 | 395 | 138 | 22 | 541 | 71 | 37 |
| P9 | 226 | 1 | 13 | 117 | 9 | 9 |
| P10 / Fleet | 153 | 30 | 16 | 2418 | 248 | 272 |

This is the strongest mechanism to borrow. Our current `tonic-profile` `EchoRequest { payload: Vec<u8> }` path exercises payload bytes, compression, copy, and buffer movement, but it does **not** exercise realistic protobuf field layout. Passing a FleetBench-like serialized blob inside `bytes payload` would still mostly measure a bytes field; it would not reproduce realistic prost encode/decode behavior. To borrow the mechanism correctly, Tonic needs a separate protobuf-shaped RPC mode.

### 2. Closed-loop concurrency and connection topology

FleetBench's client starts `max_outstanding_rpcs` per stub buffer, and stub buffers are created for `peers * connections_per_peer`. After each completion callback, it optionally delays and recursively sends the next RPC. This is closer to a configurable closed-loop load generator than our current `tonic-profile` loop, which mainly varies task concurrency.

Mechanism to borrow:

- separate `concurrency` from `max_outstanding_rpcs`, `connections_per_peer`, and `peer_count`,
- record whether a run is single-channel, multi-channel, or multi-peer,
- preserve a closed-loop mode, but make the mode explicit in metadata.

### 3. Request and response delay distributions

FleetBench supports distribution arguments for request/inter-RPC delay and response delay. Distribution types include constant, uniform, empirical, exponential, gaussian, and poisson. The parser accepts inline textproto or a file containing a textproto.

Mechanism to borrow:

- add `--client-delay-dist` and `--server-delay-dist` to `tonic-profile`, with at least `constant`, `uniform`, `empirical`, and `poisson` initially,
- report sampled mean and distribution description in JSON metadata,
- use this to distinguish saturation runs from service-time/queueing characterization runs.

### 4. Server response is not echo-only

FleetBench's server ignores the request payload for application semantics and returns a randomly selected pre-generated response for the chosen program. That means request and response shape can differ. This matters because local Tonic results currently conflate request and response into the same payload shape.

Mechanism to borrow:

- add independent request and response shape selection,
- support asymmetric request/response sizes and schemas,
- report request/response serialized sizes separately.

### 5. Tx zerocopy threshold as an explicit transport/copy control

FleetBench exposes gRPC TCP Tx zerocopy flags and a byte threshold. Rust Tonic/hyper will not map directly to C++ gRPC's channel arguments, but the idea is still useful: make large-message copy-control thresholds explicit.

Mechanism to borrow conceptually:

- add metadata/control fields for large-message copy policy thresholds,
- do not claim C++ gRPC Tx zerocopy equivalence in Tonic unless we implement a real hyper/Tokio equivalent.

## Mechanisms not worth directly importing

1. **Do not import the default `rpc_benchmark` wrapper as-is as the only benchmark.** Its default wrapper uses one worker, one connection, one outstanding RPC, no compression, no checksum, and local loopback. It is useful as a CPU/code-path baseline, but it is narrower than our regime-based characterization matrix.
2. **Do not treat FleetBench as stage attribution by itself.** It records client completion latencies and server query counts when logging is enabled, but it does not decompose encode/decode/buffer/copy/compression/framing/runtime costs. Use it as the CPU half of a paired CPU+decomposition workflow.
3. **Do not rely on its `compress` or `checksum` flags without checking implementation.** In the inspected RPC path, those options are stored in client/server option structs, but the code search found no gRPC compression algorithm setup or checksum work in the RPC source. The meaningful implemented transport/copy knob I found is TCP Tx zerocopy.

## Incorporation recommendation

Add a **FleetBench-inspired protobuf workload mode** to `tonic-profile`, but keep it as a Tonic-native mechanism that supports both CPU characterization and stage decomposition:

### Phase 1: schema-shape support

Add a second unary RPC mode beside the current byte-echo mode:

```text
--rpc unary-bytes        # current EchoRequest { bytes payload }
--rpc unary-proto-shape  # new generated nested protobuf request/response mode
--proto-program fleet | p0 | p3 | p7 | p10 | custom
--proto-message-policy cycle | random
--response-program same | fleet | p10 | custom
```

Start with a small subset rather than copying all FleetBench generated protos:

- `p6` or another small/light program for fixed overhead,
- `p0` or `p2` for moderate nested/repeated shape,
- `p3`/`p7` for string-heavy request-side shape,
- `p10/Fleet` for response-heavy production-derived shape.

### Phase 2: distribution and load-shaping support

Add:

```text
--client-delay-dist 'constant:0'
--server-delay-dist 'constant:0'
--load-mode closed-loop | fixed-window
--max-outstanding-rpcs N
--connections-per-peer N
--peer-count N
```

For the first implementation, the dist syntax can be simpler than FleetBench textproto, for example:

```text
constant=0
uniform=0:50
poisson=20:max=200
empirical=0,0,10,100
```

### Phase 3: report integration

Extend report tables so each run includes:

- RPC mode: bytes vs proto-shape,
- program / request shape / response shape,
- serialized request bytes and serialized response bytes,
- field-count class: flat, nested, repeated-heavy, string-heavy,
- load mode and delay distributions,
- existing buckets: encode/decode, buffer reserve/grow, body accumulation, compression, frame header.

## How it maps to our current buckets

| Current Tonic bucket | FleetBench mechanism to borrow | Expected value |
|---|---|---|
| serialization / deserialization | generated nested protobuf program shapes | exposes real prost field traversal rather than bytes-field-only encode/decode |
| copy / buffer lifecycle | asymmetric request/response serialized sizes; Tx-zerocopy-inspired threshold concept | separates byte movement from schema traversal and size asymmetry |
| runtime / queueing | closed-loop max outstanding RPCs, peers, connections per peer | avoids conflating task count with connection/load shape |
| compression | realistic string-heavy/repeated-heavy schemas | more realistic compressibility than only random/structured/repeated byte arrays |
| tail latency | client/server delay distributions | lets us model queueing and service-time variance explicitly |
| framing / transport | gRPC unary service with multi-channel peers | keeps an RPC-preserving path while varying connection topology |

## Concrete next repo task

The next implementation task should be:

> Add a `unary-proto-shape` mode to `accel-rpc/tonic-profile` with a small FleetBench-inspired schema subset and metadata reporting for request/response serialized sizes, then rerun paired CPU and decomposition comparisons against the existing `unary` bytes mode.

Do this before further DSA/IAX ranking, because the current bytes-only harness can overstate copy/buffer importance and understate realistic prost serialization cost for nested/string-heavy messages.

## Proposed first experiment after implementation

Keep it small:

| axis | values |
|---|---|
| RPC mode | `unary-bytes`, `unary-proto-shape` |
| proto shape | small/light, repeated-heavy, string-heavy, Fleet/P10 response-heavy |
| concurrency | `1`, `32`, `128` |
| runtime | `single`, `multi` |
| compression | off, on for string-heavy/Fleet only |
| buffer policy | default, pooled |
| instrumentation | off for throughput, on for attribution repeats |

Expected question:

> Does realistic protobuf shape change the offload-readiness ranking from “copy/buffer first” toward “serialization/schema traversal matters in some regimes,” or does copy/buffer remain dominant after shape realism is added?
