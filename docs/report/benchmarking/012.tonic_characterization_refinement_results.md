## Method

This run used `tonic-profile --mode selftest` rather than split client/server processes. That makes the stage timers process-local and captures both client and server codec/body work together, which is useful for first-pass attribution but not a replacement for later split-process endpoint-specific accounting.

Common settings:

- command wrapper: `taskset -c 0,1`
- warmup: `300 ms`
- measure window: `1000 ms`
- repeats: `3` per configuration
- RPC mode: unary
- transport: `127.0.0.1` loopback
- multi-thread runtime: harness `worker_threads(2)`

Matched unary regimes:

| regime label | payload | concurrency | runtime | compression | variants |
|---|---:|---:|---|---|---|
| tiny | `256 B structured` | `1` | `single` | off | default inst on/off, pooled, copy-minimized |
| medium | `4 KiB structured` | `128` | `multi` | off | default inst on/off, pooled, copy-minimized |
| large-64K | `64 KiB structured` | `32` | `multi` | off | default inst on/off, pooled, copy-minimized |
| large-1M | `1 MiB structured` | `32` | `single` | off | default inst on/off, pooled, copy-minimized |
| compression | `64 KiB structured/random` | `128` | `multi` | off/on | default inst on/off for compression-on |

Unless stated otherwise, the tables below use the median across three repeats. I use medians because a few short high-concurrency selftest runs had severe tail outliers.

## Baseline stage attribution

Stage time is reported as microseconds per completed unary round trip, with client and server process-local stage counters combined in selftest mode.

| regime | throughput | p99 | stage us/RPC | top stage shares |
|---|---:|---:|---:|---|
| 256 B structured/off c1 single | 24,248.3 rps | 47 us | 0.69 | decode 41.5% (0.29 us), encode 19.9% (0.14 us), buffer_reserve 15.3% (0.11 us) |
| 4 KiB structured/off c128 multi | 7,993.4 rps | 259,455 us | 9.97 | buffer_reserve 44.4% (4.42 us), encode 23.2% (2.31 us), decode 20.4% (2.04 us) |
| 64 KiB structured/off c32 multi | 2,333.6 rps | 259,583 us | 75.34 | body_accum 36.2% (27.24 us), encode 30.9% (23.27 us), decode 22.3% (16.82 us) |
| 1 MiB structured/off c32 single | 176.4 rps | 570,879 us | 706.52 | body_accum 35.0% (247.36 us), encode 31.7% (223.88 us), decode 30.5% (215.33 us) |

Interpretation:

1. The timer buckets follow the expected regime shift. Tiny RPCs are dominated by small fixed encode/decode/control work, while large messages move toward body accumulation and encode/decode byte movement.
2. The `4 KiB` matched point is the clearest buffer-policy regime in this run: `buffer_reserve` is the largest instrumented bucket.
3. The `64 KiB` and `1 MiB` uncompressed regimes stay movement-dominated. `body_accum`, `encode`, and `decode` collectively dominate the instrumented stage sum.

## Instrumentation overhead control

| regime | inst-on rps | inst-off rps | on/off | p99 on | p99 off |
|---|---:|---:|---:|---:|---:|
| 256 B c1 single | 24,248.3 | 24,934.8 | 0.972 | 47 | 44 |
| 4 KiB c128 multi | 7,993.4 | 19,739.1 | 0.405 | 259,455 | 10,159 |
| 64 KiB c32 multi | 2,333.6 | 4,730.2 | 0.493 | 259,583 | 20,911 |
| 1 MiB c32 single | 176.4 | 454.2 | 0.388 | 570,879 | 134,399 |

Interpretation:

- Timer overhead is acceptable for the tiny single-thread point but is **not negligible** for high-concurrency and large-message regimes.
- The likely cause is not the cost of one timer alone; it is repeated `Instant::now` plus shared atomic counter updates on hot codec/body paths, combined with selftest client/server contention.
- Therefore, this instrumentation should be treated as a **diagnostic attribution mode**, not as the throughput baseline. Throughput claims should prefer instrumentation-off runs plus external `perf` evidence.

## Buffer-policy controls

Ratios compare each variant against the default / instrumentation-on baseline for the same matched regime.

| regime | variant | rps ratio vs baseline | stage-us/RPC ratio | buffer-reserve us/RPC ratio | p99 |
|---|---|---:|---:|---:|---:|
| 256 B c1 single | pooled | 0.996 | 1.012 | 0.990 | 46 |
| 256 B c1 single | copy-min | 0.977 | 2.200 | 4.468 | 48 |
| 4 KiB c128 multi | pooled | 2.513 | 0.725 | 0.679 | 9,607 |
| 4 KiB c128 multi | copy-min | 0.338 | 1.839 | 1.284 | 270,335 |
| 64 KiB c32 multi | pooled | 1.384 | 1.004 | 3.807 | 219,263 |
| 64 KiB c32 multi | copy-min | 1.562 | 1.067 | 2.581 | 27,695 |
| 1 MiB c32 single | pooled | 0.845 | 2.007 | 77.718 | 701,439 |
| 1 MiB c32 single | copy-min | 2.156 | 1.630 | 48.669 | 184,575 |

Interpretation:

- `pooled` is promising for the `4 KiB` regime: throughput improves by `2.51x`, stage time per RPC falls, and buffer-reserve timer cost falls.
- `copy-minimized` is not a uniformly good setting. It is harmful at `4 KiB`, but improves throughput and p99 at `64 KiB` and especially `1 MiB` in this selftest run.
- The `buffer_reserve` timer does **not** directly mean “reallocation cost only.” It measures reserve calls and requested reserve sizes. Large pre-sizing policies can increase the measured reserve bucket even when throughput improves. This is why buffer-policy conclusions should be based on end-to-end deltas plus `perf`, not reserve-time counters alone.

## Compression controls

Matched point: `64 KiB`, concurrency `128`, multi-thread runtime, default buffer policy.

| payload | off rps | on rps | on/off | p99 off | p99 on | compress us/RPC | decompress us/RPC | on inst-on/off |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| 64 KiB structured c128 multi | 5,935.4 | 3,474.4 | 0.585 | 45,919 | 55,519 | 388.90 | 92.13 | 0.686 |
| 64 KiB random c128 multi | 3,754.3 | 421.2 | 0.112 | 297,471 | 626,687 | 4,389.76 | 46.80 | 0.986 |

Interpretation:

- Compression remains payload-regime-sensitive. Structured payload throughput falls to `0.585x` of uncompressed; random payload throughput collapses to `0.112x`.
- The instrumentation tells a useful directional story: compression dominates random payload cost, while structured payload compression is expensive but less catastrophic.
- Compression-on instrumentation overhead is large for structured payloads (`0.686x` inst-on/off), but small for random payloads (`0.986x`) because the gzip compute cost dominates the timer overhead.

## Representative `perf stat` observations

| point | IPC | backend-bound | memory-bound | retiring |
|---|---:|---:|---:|---:|
| `tiny_c1_single_default_inst_on` | 2.2 | 33.7% | 17.3% | 36.9% |
| `medium_c128_multi_default_inst_on` | 1.3 | 40.6% | 25.6% | 24.8% |
| `large64_c32_multi_default_inst_on` | 0.7 | 47.3% | 33.7% | 18.0% |
| `large1m_c32_single_default_inst_on` | 0.5 | 74.2% | 63.7% | 10.9% |
| `comp64_structured_c128_multi_default_inst_on` | 4.3 | 20.3% | 15.2% | 69.0% |

Interpretation:

- The uncompressed path moves from moderate IPC at tiny payloads toward memory/backend-bound behavior as payload size grows.
- The `1 MiB` uncompressed point is strongly memory-bound, matching the earlier `memmove`/buffer-movement conclusion from report 009.
- The compressed structured point is retiring-heavy with high IPC, matching the interpretation that software gzip work dominates more than memory stalls in that regime.

## Findings

1. **The new bucketed characterization framing is paying off.** The internal timers separate tiny fixed codec work, medium buffer-reserve sensitivity, large body/encode/decode movement, and compression transform work more cleanly than the old end-to-end-only matrix.
2. **Instrumentation must be controlled, not left on by default.** It is low-overhead only for the tiny single-thread point. For larger/high-concurrency regimes it changes throughput and tails enough that future reports should pair instrumentation-on attribution with instrumentation-off throughput.
3. **Buffer policy is now the strongest software-control lane.** `pooled` materially improves the `4 KiB` matched regime, and `copy-minimized` improves the larger regimes in selftest, but the effects are not monotonic across size.
4. **Compression is still not a generic offload target.** It is destructive for random payloads and only conditionally useful for structured payloads; compression has to remain compressibility-aware and tail-aware.
5. **Large uncompressed messages remain the clearest movement-bound DSA follow-up candidate after software controls.** However, the next step should first separate client/server sides in split-process mode and reduce timer overhead before making an offload-readiness threshold claim.

## Limitations

- This run used selftest mode, so client and server instrumentation are combined in one process.
- The short `1000 ms` windows and high concurrency produced some tail outliers; medians are used for report tables.
- The current timer implementation uses hot-path shared atomics and `Instant::now`, which is too intrusive for final throughput measurement.
- No flamegraphs were collected in this pass; `perf stat -d` was collected for representative points, while call-stack evidence should reuse or extend reports 010/011 in the next pass.
- `async-bench` expansion was not run here; it remains Phase B work.

## Next steps

1. Add lower-overhead per-worker or endpoint-local timer accumulation, then flush counters at phase end.
2. Run split client/server mode and emit a server-side snapshot so encode/decode/body costs can be attributed per endpoint.
3. Rerun the `4 KiB` pooled and `64 KiB`/`1 MiB` copy-minimized controls with longer windows.
4. Add representative flamegraphs for the same matched points.
5. Only after those controls, rank residual copy/buffer/compression buckets against DSA/IAX hardware lower bounds.
