## Findings

### 1. Small unary RPCs are runtime-dominated

At `256 B` and concurrency `1`, the single-thread runtime is dramatically better than multi-thread:

- throughput: `27.1k rps` vs `2.2k rps`
- p50 latency: `36 us` vs `425 us`

For tiny messages on localhost, extra Tokio scheduling is more expensive than useful. This matches the plan hypothesis that small-message performance is dominated by framework overhead rather than payload work.

### 2. Multi-thread runtime helps once payload work and concurrency are non-trivial

At `4 KiB` and concurrency `128`, multi-thread beats single-thread on average:

- throughput: `39.9k rps` vs `34.8k rps` (+14.5%)
- p50 latency: `3.31 ms` vs `3.70 ms`
- p99 latency: `4.79 ms` vs `5.07 ms`

The single-thread configuration also showed much larger run-to-run variance. Once there is enough outstanding work, multi-thread scheduling overhead is amortized.

### 3. Large payloads expose severe tail behavior even on loopback

At `64 KiB`, `structured`, concurrency `512`, runtime `multi`:

- throughput: `795.8 MiB/s`
- p95: `76.2 ms`
- p99: `113.9 ms`
- worst observed max: `493.6 ms`

So the large-payload regime is not just throughput-bound; it is also highly tail-sensitive. Buffering, queue depth, and runtime handoff are likely contributing materially here.

### 4. Compression is a strong candidate for deeper follow-up on compressible payloads

Direct `64 KiB repeated` comparison at concurrency `128`, runtime `multi`:

- compression **off**: `8.49k rps`, `530.5 MiB/s`, p99 `114.4 ms`
- compression **on**: `12.54k rps`, `783.7 MiB/s`, p99 `17.8 ms`

Relative to compression-off, compression-on improved:

- throughput by **~47.7%**
- bandwidth by **~47.7%**
- p99 latency by **~6.4x**

On this highly compressible payload, reduced bytes-on-wire and downstream movement dominate the software compression cost. This keeps compression/decompression at the top of the follow-up list.

### 5. CPU profile shape changes with compression-heavy runs

The representative `time -v` runs showed:

- medium no-compression case: `124%` CPU, `10.2 MiB` RSS, `350k` voluntary context switches
- large compression-on case: `182%` CPU, `34.2 MiB` RSS, `61k` voluntary context switches

This suggests the compression-heavy case shifts work from scheduler churn toward actual CPU-bound transform work plus larger in-process buffering.

## Shortlist for next-step deep profiling

1. **compression / decompression** on compressible medium-to-large payloads
2. **buffer movement / buffer lifecycle** in large-payload, high-concurrency runs
3. **Tokio runtime overhead** for tiny unary RPCs where scheduler cost dominates total latency

## Limitations

- `perf` was unavailable on this machine, so no `perf stat` or `perf record` artifacts were collected in this run.
- `cargo flamegraph` was not installed, so no flamegraph SVGs were generated.
- `tonic-profile` currently reports end-to-end metrics only; internal encode/decode/compress phase timers are not yet implemented.
- Large-payload runs showed noticeable variance, so deeper repetition and/or tighter machine isolation would help before making fine-grained conclusions.

## Conclusion

The split-localhost tonic experiment supports the current repo hypotheses:

- small unary RPCs are dominated by runtime/framework overhead,
- medium workloads benefit from multi-thread execution once concurrency is high enough,
- and compression remains the clearest next candidate for deeper tonic-path profiling because it stays material in end-to-end results.
