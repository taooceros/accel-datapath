## Findings

1. **The runtime crossover is workload-dependent, not monotonic.** Tiny low-concurrency RPCs still prefer the single-thread runtime, medium payloads with meaningful concurrency prefer multi-thread, and very large payloads fall back to movement-dominated behavior where extra runtime parallelism on a pinned core does not help.
2. **The strict bounded matrix confirms the active plan concern about confounded comparisons.** The best-performing runtime/concurrency choice changes across sizes, so future attribution claims should compare matched regimes rather than one "small/medium/large" point per category.
3. **Medium and large uncompressed runs are still ruled by copy/allocation/buffer growth.** The perf reports stay led by `memmove`, allocator paths, and `BytesMut`/`RawVec` growth rather than protobuf or scheduler work.
4. **Compression remains highly payload-sensitive.** It is disastrous for incompressible payloads, while structured payloads show a more nuanced trade-off: lower throughput but sometimes better tail latency.
5. **The strongest next-step deep-profiling lane is still buffer lifecycle and copy behavior.** Compression is worth revisiting only with tighter placement/control variants and explicit compressibility-aware comparisons.

## Limitations

- `tonic-profile` still reports end-to-end metrics only; internal encode/decode/compress/buffer timers are not implemented yet.
- The pooled-buffer, copy-minimized, and instrumentation-off control variants from the plan are still absent.
- The current split-core setup pins each process to a single CPU, so the multi-thread runtime comparison reflects scheduler topology on a constrained core budget, not unrestricted scaling.

## Conclusion

The plan's bounded first-pass tonic experiment is now executed end to end. The resulting matrix strengthens the existing repo hypothesis: once payloads become medium or large, the dominant software costs are buffer movement, allocation, and copy behavior; compression only helps in carefully chosen payload regimes and is actively harmful on incompressible data.
