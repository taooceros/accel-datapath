# IDXD + Tonic same-repo claim package

**Date**: 2026-04-24  
**Type**: comparative claim package / narrow conclusion

## Question

For the curated unary Tonic workloads in this repo, does the current same-repo ordinary-versus-IDXD package support the async-framework-mismatch claim, narrow it, or falsify it for this specific IDXD + Tonic seam?

## Evidence package

This report is grounded in the stable S04 outputs:

- `accel-rpc/target/s04-claim-package/latest/summary/comparison_summary.json`
- `accel-rpc/target/s04-claim-package/latest/summary/ordinary_vs_idxd.csv`
- `accel-rpc/target/s04-claim-package/latest/summary/claim_table.md`

The package currently combines:

- a fresh S02 software rerun under `accel-rpc/target/s04-claim-package/latest/software/`
- the current control-floor export at `accel-rpc/target/control-floor/async_control_floor_summary.json`
- a seeded `latest/idxd/` subtree copied from the frozen S04 fixture because the live S03 preflight is blocked on this host

## Host-preparation note

A live IDXD rerun is still blocked here because `verify_s03_idxd_path.sh` reports that `tools/build/dsa_launcher` lacks `cap_sys_rawio+eip`. The S04 workflow now makes that failure explicit and, to keep the comparison/reporting boundary inspectable, seeds `accel-rpc/target/s04-claim-package/latest/idxd/` from the frozen fixture artifacts when that preflight is unavailable.

That gives the repo a stable claim-package tree and report-ready outputs, but it also means the current `latest/` comparison is **mixed provenance** rather than a fully fresh same-host ordinary-versus-IDXD rerun.

## What the current tables actually show

The current `latest/` tables do **not** show an IDXD win:

- bytes/client: IDXD is ~0.005x the software baseline throughput ratio
- bytes/server: IDXD is ~0.653x the software baseline throughput ratio
- proto/client: IDXD is ~0.003x the software baseline throughput ratio
- proto/server: IDXD is ~0.425x the software baseline throughput ratio

The stage totals are also mixed rather than uniformly better:

- proto/client keeps a lower IDXD-attribution stage total than software attribution
- the other rows show higher stage totals on the seeded IDXD side than on the fresh software side

## Narrow conclusion

For the evidence currently present in `latest/`, the package **does not support** a positive same-repo claim that the IDXD path beats the ordinary path for these workloads.

More specifically:

1. The repo now has the **mechanical comparison package** S04 promised: stable run tree, JSON summary, CSV, markdown claim table, and a tracked report.
2. The current `latest/` numbers **falsify any broad claim of clear IDXD advantage** on their face.
3. Because `latest/` is mixed provenance on this host, the package should **not** be used to make a stronger scientific statement about the live accelerator/runtime interaction than the data supports.

So the correct repo-local statement today is:

> The S04 package is complete as a comparison/reporting workflow, but the current host cannot refresh the accelerated subtree from live hardware. The currently generated `latest/` tables do not support a positive IDXD win claim, and they should be treated as a narrow blocking/falsifying outcome until a launcher-capable prepared host reruns the same workflow end to end.

## Interpretation

### What is supported

- The repo can now assemble ordinary-path evidence, accelerated-path evidence, and control-floor reference data into one durable claim-package tree.
- The summary contract is stable enough that downstream readers can inspect one JSON file, one CSV, and one markdown table instead of scraping ad hoc outputs.
- Operational failures are explicit: missing launcher capability is surfaced as a phase-labeled preflight blocker, not as a silent comparison gap.

### What is not supported

- A broad claim that Tonic async-framework overhead is the main explanation for fast-accelerator mismatch.
- A broad claim that the current IDXD path is a clear winner for the curated workloads.
- Any statement that depends on the seeded fixture artifacts being equivalent to a fresh prepared-host rerun.

## Why the async mismatch claim remains narrow

The control-floor numbers remain useful as a lower bound, but the current package does not show them dominating the end-to-end rows in a way that would justify blaming the outcome primarily on Tokio scheduler mechanics. At the same time, the mixed-provenance `latest/` package prevents a clean positive acceleration claim.

That leaves the milestone with a deliberately narrow result:

- the **workflow and artifact boundary are proven**
- the **current generated comparison is a blocking/falsifying package rather than a positive win package**
- the async-framework-mismatch claim remains **unproven beyond this narrow repo-local scope**
