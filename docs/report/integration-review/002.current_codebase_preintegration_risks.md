## Executive summary

S04 can safely consume `dsa-ffi`'s public async owner/handle surface from a downstream `tonic-profile` binary or test because the current code exports the right symbols, preserves typed async versus memmove failures, and already proves clone composition and shutdown behavior inside `dsa-ffi`.

S04 should not rewrite or force async behavior into `tonic-profile/src/custom_codec.rs`. That file still implements tonic's synchronous `Encoder::encode()` path with a cached synchronous `DsaSession`, and the surrounding runner/claim-package metadata explicitly labels the existing accelerated proof as `seam=codec_body` and `accelerated_lane=codec_memmove`, not as a Tokio async-handle consumer.

No plan-invalidating blocker was found. The main pre-integration cleanup recommendation is to keep S04's first downstream proof as a separate, narrow `tonic-profile` binary/artifact contract rather than coupling it to the existing codec benchmark runner or turning `tonic-profile` into a public library.

## Blocking risks

No current code seam blocks S04 from proving downstream Tokio reuse through a new `tonic-profile` binary that imports `dsa_ffi::{AsyncDsaSession, AsyncDsaHandle, AsyncMemmoveRequest, AsyncMemmoveError}`.

The following would become blockers only if S04's implementation plan chose the wrong seam:

1. **Codec rewrite path:** `custom_codec.rs` is synchronous by trait shape. Trying to await `AsyncDsaHandle::memmove()` inside `Encoder::encode()` would require a runtime/blocking workaround or a larger tonic seam redesign. That would violate the review boundary inherited from T01.
2. **Public library path:** `tonic-profile` is organized as a binary-first crate with private sibling modules. If S04 requires reusable public `tonic-profile` APIs, that is a different design slice and should be approved explicitly.
3. **Claim-package path:** Existing S03/S04 scripts are tied to `codec_body`/`codec_memmove` evidence. If S04 tries to publish async-handle evidence as the same claim without new metadata, future agents will not be able to distinguish codec proof from async proof.

## Acceptable constraints

These constraints are real but acceptable if S04 stays on the recommended downstream-binary path:

- `AsyncDsaHandle` serializes requests through one worker-owned session. This is a feature of the current proof contract, not a performance parallelism promise.
- `AsyncDsaSession::shutdown()` is synchronous and joins a worker thread. The tests prove drain-then-stop behavior; S04 should call it explicitly after awaited work and report its outcome.
- `AsyncMemmoveRequest` owns bytes and validates zero length / destination sizing before dispatch. That keeps the cross-thread boundary safe and should not be bypassed.
- Existing hardware verification still depends on `/dev/dsa/wq*`, `devenv`, `launch`, and `tools/build/dsa_launcher` capability checks. Unsupported-host outcomes remain evidence, not generic failures.
- `tonic-profile`'s existing `accelerated_mode_contract.rs` may pass either by successful IDXD metadata or by explicit queue-open failure without bogus artifacts. This is aligned with inherited failure-classification discipline.

## S04 safe path

The safest S04 implementation path remains:

1. Add a new `accel-rpc/tonic-profile/src/bin/<async-downstream-proof>.rs` binary.
2. Import only public `dsa_ffi` async symbols from `dsa-ffi/src/lib.rs`.
3. Open one `AsyncDsaSession`, clone `session.handle()`, and drive at least two requests through ordinary Tokio composition (`tokio::join!` or spawned tasks).
4. Explicitly call `session.shutdown()` and record the shutdown result.
5. Emit a small JSON artifact that identifies the proof as downstream async-handle evidence, not codec-body evidence.
6. Add deterministic integration tests around the binary using test scenarios or `spawn_with_factory` patterns analogous to `await_memmove.rs` and `dsa-ffi` contract tests.
7. Add a new hardware verifier script only after the artifact contract is stable, mirroring launcher/preflight patterns from `verify_async_memmove.sh` and `verify_s03_idxd_path.sh`.

## Pre-integration cleanup recommendation

S04 does not need cleanup of the core `dsa-ffi` async API before starting. It does need naming and artifact-boundary cleanup in the design before implementation:

- Name the new proof boundary distinctly, for example `downstream_async_handle` or `tonic_async_handle`, so it cannot be confused with `codec_body` / `codec_memmove`.
- Keep `custom_codec.rs` unchanged unless a comment or report reference is needed.
- Do not extend `run_s03_idxd_evidence.py` for the async proof; create a separate verifier so current S03 evidence remains historically stable.
- Keep deterministic test scenarios small and focused on cross-crate use, not on re-proving every `dsa-ffi` contract case.
