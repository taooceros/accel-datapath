## Executive summary

The current active Rust workspace has **not yet** reached the S04 target layout. It still exposes `accel-rpc/dsa-ffi` as the active safe Rust/async proof crate and `dsa-bindings` as the raw low-level crate named `idxd-bindings`. `accel-rpc/tonic-profile` actively depends on `dsa-ffi`, and the active verifier scripts still build/package-test `dsa-ffi` rather than `idxd-rust`.

The S04 consolidation target from decisions `D023` and `D024` is therefore a rename/migration, not a no-op:

- create or migrate the raw C/UAPI/MMIO layer to canonical package/path `idxd-sys`;
- create or migrate the safe Rust binding plus Tokio-facing owner/handle API to canonical package/path `idxd-rust`;
- remove `dsa-ffi`, `idxd-bindings`, and top-level wrapper copies from active workspace/verifier/downstream consumption, or leave them only as clearly de-scoped compatibility/historical artifacts;
- keep `iax-ffi` separate because it is an IAX/non-IDXD package surface and is not the S04 canonical DSA binding target.

## Naming hazards found

1. **`dsa-ffi` is both an active package and a top-level wrapper directory.** The active package is `accel-rpc/dsa-ffi`; the top-level `dsa-ffi/` directory only contains wrapper scripts. S04 should avoid leaving both visible as equivalent-looking active entrypoints.
2. **`dsa-bindings` path contains the `idxd-bindings` package.** The directory name and package name differ, and both are superseded by the requested canonical raw package name `idxd-sys`.
3. **`idxd-sys` and `idxd-rust` are absent today.** Any downstream S05 work that imports those names before T02 lands will fail. This is expected and should become the consolidation proof target.
4. **`iax-ffi` is active but not a DSA/IDXD consolidation target.** It should remain classified as unrelated IAX/non-IDXD so duplicate-package checks do not falsely require its removal.
5. **Historical S01-S03 reports and decisions intentionally mention stale names.** Completed summaries and decisions should not be rewritten. Active manifests, active verifier scripts, and active S04/S05 planning docs are the important stale-reference failure surfaces.

## S04 implementation guidance

- Treat `accel-rpc/dsa-ffi` as the source of the safe API and tests to preserve, not as the final package name.
- Treat `dsa-bindings` / package `idxd-bindings` as the source of raw MMIO/UAPI pieces to preserve, not as the final package name.
- Keep `DsaSession`, validation error taxonomy, async owner/handle semantics, verifier preflight checks, and artifact-validation behavior intact through the rename.
- Ensure `tonic-profile` and any other downstream consumer imports the canonical safe crate (`idxd-rust`) only after T02/T03.
- Leave completed S01-S03 summaries unchanged; use this report and later active docs to explain the supersession.
