## Executive summary

M003 may safely inherit that the repository has a real Rust DSA proof substrate, a retained Tonic ordinary-versus-IDXD evidence workflow, and explicit failure-classification conventions. It must not inherit any broad claim that the current repo already demonstrates a fresh prepared-host IDXD performance win or that integration should continue automatically after M005.

The safest interpretation for S04 is:

1. Reuse `accel-rpc/dsa-ffi` as the canonical Rust DSA owner and async proof substrate.
2. Keep `accel-rpc/tonic-profile` as a downstream proof host and evidence/reporting culture, not as proof that the synchronous codec seam is already suitable for Tokio-native integration.
3. Preserve explicit failure categories and phase-labeled verifier/report failures.
4. Treat M005 as an intentional pause point; S04 remains blocked on human review of design choices, not merely on executor availability.

## M001 inherited assumptions and caveats

### Inherited assumptions

- **Proof boundary:** M001 established `accel-rpc/tonic-profile` and the Tonic codec/body payload seam as the repo-local ordinary-versus-IDXD proving ground. M003 can assume this harness is the established comparison context for curated unary Tonic workloads.
- **Measurement discipline:** M001 deliberately separated ordinary software evidence, instrumentation/trustworthiness work, real IDXD-aware validation, and final claim-package publication. M003 should preserve that staged evidence discipline rather than collapsing proof, implementation, and reporting into one path.
- **Failure classification:** M001 accepted negative or falsifying accelerator results as successful milestone evidence if the same-repo comparison path is real, comparable, and conclusive for its narrow scope. M003 should continue treating clear non-wins or blocked-preflight results as useful evidence, not as reasons to hide or reframe failures.
- **Workload realism:** M001’s ordinary-path realism was intentionally compact and FleetBench-inspired rather than a full FleetBench import. Future integration should not assume broader workload coverage than the curated workload matrix actually supplies.

### Caveats

- **Canonical interpretation moved to report 014:** The M001 summary explicitly says it records proof-boundary delivery, not the current comparison conclusion. Report 014 is the durable post-cleanup interpretation.
- **No broad acceleration claim:** The current claim package does not support a casual statement that IDXD clearly beats ordinary execution for the curated workloads.
- **Prepared-host gap remains:** Any stronger claim still requires rerunning the retained S03/S04 accelerated path on a prepared host so the accelerated subtree is refreshed from live hardware rather than mixed-provenance fixture-backed artifacts.
- **Synchronous seam is not async approval:** The proven Tonic codec/body seam is valuable as a comparison seam, but it does not by itself approve forcing Tokio-native async behavior into tonic’s current synchronous codec API.

## M002 inherited assumptions and caveats

### Inherited assumptions

- **Proof boundary:** M002 made `accel-rpc/dsa-ffi` the canonical Rust-facing DSA proof surface. M003 should build on that crate rather than bypassing it or treating older C++/stdexec work as the mandatory first bridge.
- **Minimal operation surface:** The credible operation surface remains memmove. M003 should avoid broadening into a general accelerator runtime or additional operation families unless a later milestone explicitly changes scope.
- **Failure classification:** M002’s verifier and wrapper conventions distinguish hardware-path `MemmoveError` from worker/structural async failures and unsupported-environment outcomes. M003 should preserve those typed distinctions across any Tokio-facing or downstream boundary.
- **Async ownership:** M002 exposed async behavior through a worker-thread-backed wrapper around one live `DsaSession`. M003 can inherit the model that the worker owns the live session while callers await operations through a future-shaped API.

### Caveats

- **Binding proof is not system proof:** M002 proves the Rust binding and validation surfaces, not a fresh same-host ordinary-versus-IDXD performance result.
- **Single-session ownership matters:** The async wrapper’s credibility depends on preserving the worker-owned single-session boundary. Downstream integration should not blur ownership by making raw session state casually shareable or by disguising blocking behavior as async.
- **Unsupported environments are expected evidence:** Prepared-hardware failures, launcher/preflight issues, and unsupported-host outcomes must stay diagnosable rather than being flattened into generic test failures.
- **Post-M005 pause applies:** The M002 summary states that post-M005 work is intentionally paused until a human explicitly chooses whether and how to proceed into M003.

## M005 inherited assumptions and caveats

### Inherited assumptions

- **Proof-boundary consolidation:** M005 consolidated the retained M001/M002 proof boundary around one software verifier, one IDXD-aware verifier, and one optional comparison/report workflow with explicit phase-labeled failures.
- **Code organization:** `tonic-profile` is intentionally binary-first and split into private sibling modules (`cli`, `workload`, `service`, `runtime`, `report`) rather than a new public library surface. M003 should avoid treating that harness as a newly generalized public API.
- **Canonical written claim:** M005 established `docs/report/benchmarking/014.idxd_tonic_same_repo_claim_package.md` as the single canonical written claim for the surviving M001/M002 proof boundary.
- **Pause state:** M005 closed with an explicit pause-after-M005 state. The current S03 review is therefore part of unpausing responsibly, not a continuation of automatic implementation.

### Caveats

- **Report 014 is narrow by design:** The current stable package is useful as an inspectable proof/reporting boundary but remains limited by the prepared-host and mixed-provenance caveats.
- **Verifier center of gravity changed:** M005 notes that repo-root-safe verification now sits more directly on Python runners such as `run_s02_evidence.py`, `run_s03_idxd_evidence.py`, and `run_s04_claim_package.py` than on older shell-wrapper ceremony.
- **Cleanup did not authorize new integration:** M005 repaired and consolidated existing surfaces; it did not decide the S04 async downstream integration shape.
- **Future design choice remains open:** M005 explicitly says future work should first decide whether M003 should start at all and whether stronger same-host/live-runtime claims are needed.

## Implications for S04 design review

S04 should proceed only after the human review answers the integration-shape question. Based on the inherited assumptions above, the strongest default path is to prove cross-crate Tokio reuse in a small downstream `tonic-profile` binary or equivalent repo-local consumer while leaving `custom_codec.rs` synchronous. That path reuses M002’s async ownership model and M001/M005’s evidence culture without overclaiming that the current tonic codec seam has become async-ready.

Before any implementation-oriented slice starts, the design-choice report should ask the human to approve or reject at least these boundaries:

- whether downstream proof should be a separate Tokio consumer rather than a codec rewrite;
- whether the public async shape should remain owner/session plus cloneable handle;
- whether report 014’s narrow claim boundary remains the canonical baseline for any integration narrative;
- whether any cleanup/API adjustment is required before S04 consumes the async surface.
