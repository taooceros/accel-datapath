## Question for the human reviewer

Which integration path should unblock S04?

- **A. Recommended:** Add a distinct downstream Tokio async-handle proof in `accel-rpc/tonic-profile` that consumes the public `dsa-ffi` async API and leaves the synchronous codec seam unchanged.
- **B. Cleanup first:** Insert a cleanup/API-shape slice before S04 to tighten naming, artifact schemas, and test-only/public-boundary documentation, then implement the downstream proof.
- **C. Larger redesign:** Defer S04 and redesign the Tonic integration boundary, potentially including a public `tonic-profile` library surface or a codec/runtime refactor.

The review should explicitly approve one of these paths, or state a different path with the same level of scope and proof-boundary detail.

## Alternative A â€” separate downstream Tokio async-handle proof

### Shape

S04 adds a small, distinct downstream proof surface under `accel-rpc/tonic-profile`, likely a new binary and verifier/artifact contract, that imports only public `dsa-ffi` async symbols. The proof should open one `AsyncDsaSession`, clone `session.handle()`, drive multiple requests through ordinary Tokio composition such as `tokio::join!` or spawned tasks, explicitly call `shutdown()`, and emit a JSON artifact labeled as downstream async-handle evidence.

The existing `custom_codec.rs` synchronous codec-body path remains unchanged and continues to represent the older `codec_body` / `codec_memmove` proof lane.

### Tradeoffs

**Pros:**

- Directly follows `D017` and `D018`.
- Reuses the real `dsa-ffi` async owner/handle API without broadening the low-level DSA proof boundary.
- Avoids forcing async behavior into tonic's synchronous codec trait.
- Keeps M001/M005 codec-body evidence historically stable.
- Produces a narrow artifact future agents can distinguish from the existing claim package.

**Cons:**

- Does not prove async acceleration inside the actual `ProfileEncoder::encode()` body.
- Adds another verifier/artifact surface that must be named carefully to avoid claim-package confusion.
- May require a later slice if the project ultimately wants codec-integrated async behavior.

### S04 impact

S04 can proceed immediately as an implementation slice with a narrow scope:

1. Add a downstream async-handle binary or equivalent proof consumer in `tonic-profile`.
2. Add deterministic tests around cross-crate async-handle use and failure classification.
3. Add a separate verifier/artifact contract with metadata such as `proof_seam=downstream_async_handle`, not `codec_body`.
4. Preserve explicit lifecycle, worker, validation, and memmove error categories.

### Codebase improvements before S04

No blocking cleanup is required before this path. The implementation should still make these local improvements as part of S04:

- Use distinct proof names such as `downstream_async_handle` or `tonic_async_handle`.
- Avoid mutating `run_s03_idxd_evidence.py` or the existing S03 metadata meanings.
- Document in the new artifact/report that `custom_codec.rs` is intentionally not the async proof seam.
- Prefer a minimal artifact schema over reusing the broader benchmark claim-package schema.

### Human decision needed

Approve S04 to implement a separate downstream Tokio async-handle proof in `tonic-profile`, leaving the synchronous codec seam and existing S03/S04 claim-package meanings unchanged.

## Alternative B â€” cleanup/API-shape slice before S04

### Shape

Before downstream integration, insert a short cleanup slice that tightens artifact naming, documents public-versus-test-only async surfaces, and optionally adds small comments or docs around the synchronous codec constraint. S04 would then implement Alternative A on a cleaner naming and documentation base.

Candidate cleanup work:

- Clarify that `dsa-ffi`'s hidden `spawn_with_factory` / worker test facilities are contract-test support, not the downstream production seam.
- Add or update docs explaining that `custom_codec.rs` remains the synchronous `codec_body` / `codec_memmove` lane.
- Reserve names for future S04 metadata, for example `downstream_async_handle`, `async_handle_consumer`, or `tonic_async_handle`.
- Define a minimal JSON schema for downstream async-handle proof artifacts before writing the binary.

### Tradeoffs

**Pros:**

- Reduces ambiguity before S04 implementation.
- Makes review artifacts and future claim packages easier to interpret.
- Lowers the risk that implementation accidentally reuses codec-body names for async-handle evidence.

**Cons:**

- Delays the first downstream proof despite no hard code blocker.
- May spend a slice on documentation/naming work that S04 could handle inline.
- Does not produce new runtime evidence by itself.

### S04 impact

S04 becomes simpler and more mechanical after cleanup because the names, schema, and non-goals are already approved. However, the milestone timeline grows by one cleanup slice, and S04 remains blocked until that cleanup lands.

### Codebase improvements before S04

This alternative explicitly requires cleanup before S04:

- Add a tracked design/schema note for the downstream async-handle artifact.
- Add comments or docs preserving the distinction between codec-body evidence and async-handle evidence.
- Document which `dsa-ffi` async APIs are intended for downstream consumption versus tests.

### Human decision needed

Approve inserting a cleanup/API-shape slice before S04, with Alternative A as the intended implementation path after cleanup.

## Alternative C â€” larger Tonic boundary redesign

### Shape

Defer the downstream async-handle proof and redesign the integration target itself. This could mean turning parts of `tonic-profile` into a public library surface, redesigning the benchmark runtime to host async DSA work more directly, or investigating whether a tonic codec/body replacement can honestly expose async behavior without blocking inside `Encoder::encode()`.

### Tradeoffs

**Pros:**

- Could align future proof work more closely with the actual Tonic request/response path.
- May produce a cleaner long-term architecture if the project wants a reusable Tonic accelerator integration layer.
- Gives the human reviewer room to reject the narrow downstream-proof framing.

**Cons:**

- Conflicts with M005's binary-first `tonic-profile` cleanup unless explicitly superseded.
- Broadens M003 from Tokio reuse proof into crate-layout or Tonic seam redesign.
- Risks hiding the already-proven `dsa-ffi` async question behind larger framework work.
- Likely requires new planning before any implementation can be credible.

### S04 impact

S04 should not proceed under the current downstream-consumer task shape. The roadmap should be reassessed and S04 should be replaced or preceded by a redesign slice that defines the new public API, codec/runtime boundary, verifier metadata, and proof standard.

### Codebase improvements before S04

This alternative requires substantial pre-S04 work:

- Decide whether `tonic-profile` remains binary-first or gains a public `src/lib.rs` surface.
- Decide whether codec-body evidence remains separate from async runtime evidence.
- Define how async DSA work can be represented honestly at a tonic seam whose current codec API is synchronous.
- Replan verifier and claim-package metadata so future reports cannot confuse old codec evidence with new async integration evidence.

### Human decision needed

Reject the current S04 downstream-consumer direction and approve a redesign/replanning step before any integration code changes.

## Recommended decision

Choose **Alternative A**.

The current codebase does not show a blocker to downstream async-handle proof: `dsa-ffi` already exposes a public future-shaped owner/handle API, and `tonic-profile` already has the dependencies needed to consume it. The main risk is evidence confusion, not missing runtime capability. A separate downstream proof preserves the worker-owned DSA boundary from M002, the binary-first `tonic-profile` organization from M005, and the S03 review gate from `D021`/`D022` while producing the smallest useful S04 implementation.

Alternative B is acceptable if the human reviewer wants maximum naming/schema clarity before code changes. Alternative C should be chosen only if the project goal has shifted from proving downstream Tokio reuse to redesigning the Tonic integration architecture.
