## MegaPipe: A New Programming Interface for Scalable Network I/O

- **Venue / Year**: OSDI 2012
- **Paper folder**: [`papers/megapipe-a-new-programming-interface-for-scalable-network-i-o/README.md`](papers/megapipe-a-new-programming-interface-for-scalable-network-i-o/README.md)
- **Source URL**: https://www.usenix.org/system/files/conference/osdi12/osdi12-final-40.pdf

### Repo summary

MegaPipe redesigns network I/O around a per-core channel abstraction and explicitly makes partitioning, lightweight endpoints, and batching first-class for message-oriented workloads. In this repo's terms, it is an early API-level argument that submission and completion structure can be a dominant systems bottleneck, not just a low-level implementation detail.

### Technique

- **Per-core channel abstraction**: use a per-core bidirectional kernel↔user-space channel for both requests and event notifications.
- **Partitioning**: assign work and connection ownership explicitly across cores to preserve locality.
- **Lightweight socket (`lwsocket`)**: use a cheaper endpoint abstraction than a traditional socket for message-oriented workloads.
- **Batching**: group request submission and event delivery to amortize per-operation overhead.

### Why it matters here

It supports the repo's batching-regime thesis from the API side: once workloads are message-oriented and per-operation costs matter, batching and per-core ownership need to be built into the interface rather than left as accidental optimizations. It is especially relevant to the repo's questions about event aggregation and submission/completion policy.

### Gap relative to this repo

MegaPipe is still a network-I/O API paper rather than an accelerator paper. It does not address on-die accelerators, composable async abstractions on top of batched hardware, or the nanosecond-scale framework-overhead decomposition that drives this repo.

### RQ tags

- **RQ2**: supports framework redesign around lighter submission/completion paths
- **RQ4**: evidence that batching and queue ownership generalize beyond DSA

### Extracted opening text

> “We present MegaPipe, a new API for efficient, scalable network I/O for message-oriented workloads. The design of MegaPipe centers around the abstraction of a channel – a per-core, bidirectional pipe between the kernel and user space, used to exchange both I/O requests and event notifications.”

> “On top of the channel abstraction, we introduce three key concepts of MegaPipe: partitioning, lightweight socket (lwsocket), and batching.”

> “Such message-oriented workloads include HTTP, RPC, key-value stores with small objects ... A clean-slate redesign offers the opportunity to present an API that is specialized for high performance network I/O.”

## IX: A Protected Dataplane Operating System for High Throughput and Low Latency

- **Venue / Year**: OSDI 2014
- **Paper folder**: [`papers/ix-a-protected-dataplane-operating-system-for-high-throughput-and-low-latency/README.md`](papers/ix-a-protected-dataplane-operating-system-for-high-throughput-and-low-latency/README.md)
- **Source URL**: https://www.usenix.org/system/files/conference/osdi14/osdi14-paper-belay.pdf

### Repo summary

IX is a protected user-space dataplane design that gets high throughput and low latency by giving dedicated threads and queues to dataplane instances, processing bounded batches to completion, and removing coherence-heavy cross-core interaction. For this repo, it is a direct fast-path analogue for poll-mode execution, queue ownership, and bounded work on the hot path.

### Technique

- **Protected user-space dataplane**: move the fast path into user space while retaining isolation and protection.
- **Dedicated threads and queues**: assign hardware threads and NIC queues to each dataplane instance.
- **Bounded batching to completion**: process bounded batches, then run them to completion on the hot path.
- **Coherence avoidance**: minimize cross-core sharing and synchronization so coherence traffic does not dominate.

### Why it matters here

It reinforces the same design instincts already present in the DSA `PollingRunLoop`: keep the hot path single-threaded where possible, own the queue locally, and avoid synchronization that is larger than the work itself. It also sharpens the repo's interest in bounded batches rather than unstructured polling.

### Gap relative to this repo

IX is a NIC dataplane paper, not an accelerator framework paper. It says little about structured async composition, operation-state overhead, or accelerator-specific completion mechanisms.

### RQ tags

- **RQ2**: strong fast-path design evidence for the nanosecond regime
- **RQ4**: strong network-domain analogue for queue ownership and bounded batching

### Extracted opening text

> “The conventional wisdom is that aggressive networking requirements, such as high packet rates for small messages and microsecond-scale tail latency, are best addressed outside the kernel, in a user-level networking stack.”

> “The dataplane architecture builds upon a native, zero-copy API and optimizes for both bandwidth and latency by dedicating hardware threads and networking queues to dataplane instances, processing bounded batches of packets to completion, and by eliminating coherence traffic and multi-core synchronization.”

> “IX demonstrates that, by revisiting networking APIs and taking advantage of modern NICs and multi-core chips, we can design systems that achieve high throughput and low latency and robust protection and resource efficiency.”

## mTCP: a Highly Scalable User-level TCP Stack for Multicore Systems

- **Venue / Year**: NSDI 2014
- **Paper folder**: [`papers/mtcp-a-highly-scalable-user-level-tcp-stack-for-multicore-systems/README.md`](papers/mtcp-a-highly-scalable-user-level-tcp-stack-for-multicore-systems/README.md)
- **Source URL**: https://www.usenix.org/system/files/conference/nsdi14/nsdi14-paper-jeong.pdf

### Repo summary

mTCP argues that short-connection workloads are dominated by software-stack cost and attacks that problem with user-level networking, event aggregation, and batched I/O. The paper is useful here as evidence that high-rate request workloads expose software overhead before raw hardware limits are reached.

### Technique

- **User-level TCP stack**: remove much of kernel TCP processing from the hot path.
- **Shared-memory application interface**: replace many expensive syscalls with shared-memory interactions.
- **Flow-level event aggregation**: aggregate notification-path work across flows.
- **Batched packet I/O**: send and receive packets in batches to improve multicore I/O efficiency.

### Why it matters here

The repo's end-to-end RPC agenda depends on the claim that small operations make software overhead visible. mTCP is a clean prior example of that logic in a TCP stack rather than an accelerator stack.

### Gap relative to this repo

The paper is about a multicore TCP stack, not composable async accelerators or offload-policy crossover analysis. Its conclusions transfer as motivation and analogy more than as a drop-in design.

### RQ tags

- **RQ2**: medium support for reducing aggregation and event-path overhead
- **RQ3**: medium support for why request-heavy RPC decomposition matters
- **RQ4**: strong cross-domain evidence that software costs dominate in short-request regimes

### Extracted opening text

> “Scaling the performance of short TCP connections on multicore systems is fundamentally challenging. ... even state-of-the-art designs spend 70% to 80% of CPU cycles in handling TCP connections in the kernel.”

> “mTCP addresses the inefficiencies from the ground up—from packet I/O and TCP connection management to the application interface.”

> “Its design translates multiple expensive system calls into a single shared memory reference, allows efficient flow-level event aggregation, and performs batched packet I/O for high I/O efficiency.”

## Arrakis: The Operating System is the Control Plane

- **Venue / Year**: OSDI 2014
- **Paper folder**: [`papers/arrakis-the-operating-system-is-the-control-plane/README.md`](papers/arrakis-the-operating-system-is-the-control-plane/README.md)
- **Source URL**: https://www.usenix.org/system/files/conference/osdi14/osdi14-paper-peter_simon.pdf

### Repo summary

Arrakis splits control and data-plane responsibilities so applications can access virtualized I/O devices directly while the kernel retains protection and resource control. For this repo, it is a strong “generality without full mediation” baseline and a reminder that abstraction cost depends on where the control boundary is drawn.

### Technique

- **Control-plane / data-plane split**: keep protection and resource management in the kernel while moving fast-path data operations out of it.
- **Direct access to virtualized I/O devices**: expose hardware more directly to applications without full kernel mediation per operation.
- **Hot-path mediation removal**: retain safety guarantees while taking the kernel out of each fast-path operation.
- **Virtualization for safe direct access**: use virtualization to reconcile direct hardware access with isolation.

### Why it matters here

The repo repeatedly argues that composability is not inherently too expensive. Arrakis is a useful precedent for that style of argument: retain protection and structure, but move the hot path closer to the hardware.

### Gap relative to this repo

Arrakis is not about accelerator submission or async framework overhead on batched hardware. It is more useful as a positioning paper than as a detailed mechanism source.

### RQ tags

- **RQ2**: medium evidence for redesigning the control boundary around the fast path
- **RQ4**: medium evidence that the same tension appears outside the DSA setting

### Extracted opening text

> “We have designed and implemented a new operating system, Arrakis, that splits the traditional role of the kernel in two. Applications have direct access to virtualized I/O devices ... while the kernel is re-engineered to provide network and disk protection without kernel mediation of every operation.”

> “Reducing the overhead of the operating system process abstraction has been a longstanding goal of systems design. ... Many servers spend much of their time executing operating system code: delivering interrupts, demultiplexing and copying network packets.”

> “Arrakis can reduce the latency and throughput gap to raw hardware while retaining protection.”

## Shenango: Achieving High CPU Efficiency for Latency-sensitive Datacenter Workloads

- **Venue / Year**: NSDI 2019
- **Paper folder**: [`papers/shenango-achieving-high-cpu-efficiency-for-latency-sensitive-datacenter-workloads/README.md`](papers/shenango-achieving-high-cpu-efficiency-for-latency-sensitive-datacenter-workloads/README.md)
- **Source URL**: https://www.usenix.org/system/files/nsdi19-ousterhout.pdf

### Repo summary

Shenango is a counterpoint to pure poll-mode dedication: it tries to preserve microsecond-scale latency while greatly improving CPU efficiency through a dedicated IOKernel and very fast core reallocation. In this repo's literature stack, it is important because it shows that low-latency datapaths do not end the runtime-policy question.

### Technique

- **IOKernel**: run a privileged component on a dedicated core to steer packets and manage resource decisions.
- **Fine-grained core reallocation**: reassign cores at very short intervals instead of dedicating them permanently.
- **Demand detection**: detect quickly when a latency-sensitive app needs more cores.
- **Decoupled steering and execution**: keep low-latency I/O while avoiding permanent spin-poll core dedication.

### Why it matters here

The repo already borrows poll-mode ideas from eRPC and DPDK, but Shenango warns that the CPU-efficiency tradeoff remains real. That matters for any future `accel-rpc` design that tries to stay low-latency without burning cores wastefully.

### Gap relative to this repo

Shenango focuses on scheduling and CPU efficiency, not the finer-grained submission/completion overhead decomposition that the DSA work measures directly.

### RQ tags

- **RQ2**: medium support for runtime-level redesign beyond the hot loop itself
- **RQ3**: medium support for RPC-path evaluation under realistic CPU-efficiency constraints
- **RQ4**: medium evidence that runtime policy remains visible at small timescales

### Extracted opening text

> “Because of the high overheads of today’s kernels, the best available solution to achieve microsecond-scale latencies is kernel-bypass networking, which dedicates CPU cores to applications for spin-polling the network card. But this approach wastes CPU.”

> “Shenango achieves comparable latencies but at far greater CPU efficiency. It reallocates cores across applications at very fine granularity—every 5 µs.”

> “The IOKernel runs on a dedicated core, steering packets from the NIC and orchestrating core reallocations.”

## Caladan: Mitigating Interference at Microsecond Timescales

- **Venue / Year**: OSDI 2020
- **Paper folder**: [`papers/caladan-mitigating-interference-at-microsecond-timescales/README.md`](papers/caladan-mitigating-interference-at-microsecond-timescales/README.md)
- **Source URL**: https://www.usenix.org/system/files/osdi20-fried.pdf

### Repo summary

Caladan argues that isolation at microsecond scales is better achieved with fast control signals and core allocation than with static partitioning. For this repo, the paper is useful because it keeps the literature honest: once fast I/O exists, scheduling, interference, and control policy still shape real performance.

### Technique

- **Fast core allocation instead of static partitioning**: treat isolation as a rapid allocation problem, not a permanent slice.
- **Control-signal-driven scheduling**: use runtime signals to react to interference at microsecond timescales.
- **Microsecond-scale interference management**: design scheduling decisions around very short QoS disruptions.
- **Dynamic isolation policy**: make isolation a runtime control loop rather than a fixed provisioning choice.

### Why it matters here

It strengthens the repo's claim that runtime design remains measurable after low-level costs are reduced. That becomes especially important if the repo eventually moves from synthetic DSA paths to mixed-service RPC workloads.

### Gap relative to this repo

Caladan does not address accelerator offload or per-operation framework cost directly. It is a runtime-policy baseline rather than a mechanism paper for the DSA path.

### RQ tags

- **RQ2**: medium evidence that framework redesign cannot ignore control policy
- **RQ3**: medium relevance to microsecond-scale RPC evaluation
- **RQ4**: medium support for the repo's generalization story

### Extracted opening text

> “The conventional wisdom is that CPU resources such as cores, caches, and memory bandwidth must be partitioned to achieve performance isolation between tasks. In this paper, we show that resource partitioning is neither necessary nor sufficient.”

> “Caladan is a new CPU scheduler that can achieve significantly better quality of service through a collection of control signals and policies that rely on fast core allocation instead of resource partitioning.”

> “When colocating memcached with a best-effort, garbage-collected workload, Caladan outperforms Parties ... reducing tail latency from 580 ms to 52 µs.”

## Why these papers were grouped together

These papers jointly ground the repo’s async-overhead and batching-regime thesis. MegaPipe, IX, mTCP, and Arrakis explain why submission/completion structure, batching, queue ownership, and kernel bypass matter. Shenango and Caladan extend that story into the microsecond regime, showing that runtime policy and fast resource management remain visible once lower-level I/O costs are reduced.
