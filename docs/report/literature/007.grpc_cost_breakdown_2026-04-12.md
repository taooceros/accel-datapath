# gRPC performance cost-breakdown shortlist

**Date**: 2026-04-12
**Type**: Cross-paper synthesis plus annotated bibliography
**Purpose**: identify which papers are actually useful for explaining where gRPC-style RPC time goes, then map those papers onto the repo's current tonic profiling questions.

## Review question

The immediate question is not simply “what papers mention gRPC?”, but “which papers help decompose end-to-end RPC cost into actionable buckets?” For this repo, the most important buckets are:

- serialization / deserialization
- memory copies and buffer lifecycle
- RPC stack processing and framing
- queueing / scheduling / runtime policy
- compression and other host-side transforms
- network-wire contribution and tail latency

This matters because the repo's own tonic profiling already suggests a split regime: tiny unary RPCs are control-path heavy, medium and large uncompressed runs are dominated by copy and allocation behavior, and compressed structured runs are dominated by compression transforms. The literature is useful only if it helps explain or qualify those buckets.

## Executive ranking by explanatory value

### Tier 1: strongest papers for cost decomposition

1. **A Cloud-Scale Characterization of Remote Procedure Calls** — SOSP 2023  
   URL: https://dl.acm.org/doi/10.1145/3600006.3613156  
   Paper folder: [`papers/cloud-scale-characterization-of-remote-procedure-calls/README.md`](papers/cloud-scale-characterization-of-remote-procedure-calls/README.md)  
   Why it ranks first: this is the best source found for a true latency-tax breakdown. It separates application processing from the **RPC latency tax**, then further decomposes that tax into request/response queueing, RPC processing and network-stack work, and network-wire time. The paper also reports CPU-cycle shares where compression, networking, and serialization are individually visible.  
   Best bucket coverage: queueing, RPC stack, wire latency, tail latency, CPU-tax framing.  
   Main limitation for this repo: it studies Google's internal RPC stack rather than open-source gRPC or Rust tonic, so the decomposition is highly relevant but not API-identical.

2. **A Hardware Accelerator for Protocol Buffers** — MICRO 2021  
   URL: https://doi.org/10.1145/3466752.3480051  
   Public summary: https://research.google/pubs/a-hardware-accelerator-for-protocol-buffers/  
   Paper folder: [`papers/hardware-accelerator-for-protocol-buffers/README.md`](papers/hardware-accelerator-for-protocol-buffers/README.md)  
   Why it ranks second: this is the strongest source for the serialization/deserialization bucket. It profiles protobuf usage at Google scale, builds **HyperProtoBench**, and shows that serialization-framework behavior is varied enough that accelerator design must be workload-aware rather than built around a single simplistic message model.  
   Best bucket coverage: serialization, deserialization, protobuf field mix, end-to-end protobuf acceleration.  
   Main limitation for this repo: it is not a gRPC end-to-end paper, so it does not directly explain runtime scheduling, HTTP/2 framing, or tonic-specific buffer paths.

3. **Designing a Micro-Benchmark Suite to Evaluate gRPC for TensorFlow: Early Experiences** — BPOE / ASPLOS workshop 2018  
   URL: https://arxiv.org/abs/1804.01138  
   Paper folder: [`papers/designing-a-micro-benchmark-suite-to-evaluate-grpc-for-tensorflow-early-experiences/README.md`](papers/designing-a-micro-benchmark-suite-to-evaluate-grpc-for-tensorflow-early-experiences/README.md)  
   Why it ranks third: this is the cleanest direct gRPC measurement paper found during the search. Its main value is methodological: it explicitly compares **serialized vs non-serialized** payload modes, controls iovec buffer distributions, and varies the underlying transport, which makes it useful for separating communication-engine cost from serialization cost.  
   Best bucket coverage: direct gRPC benchmarking method, serialization-vs-transport separation, payload-shape sensitivity.  
   Main limitation for this repo: the domain is TensorFlow parameter-server traffic, so the workload model is not representative of general unary tonic RPCs.

4. **RPCAcc: A High-Performance and Reconfigurable PCIe-attached RPC Accelerator** — arXiv 2024  
   URL: https://arxiv.org/abs/2411.07632  
   Paper folder: [`papers/rpcacc-a-high-performance-and-reconfigurable-pcie-attached-rpc-accelerator/README.md`](papers/rpcacc-a-high-performance-and-reconfigurable-pcie-attached-rpc-accelerator/README.md)  
   Why it ranks fourth: it is one of the clearest modern papers connecting RPC acceleration to serialization cost under realistic deployment constraints. The key lesson is that simply “accelerating RPC” is not enough; PCIe traversal cost can erase wins unless serializer/deserializer design and memory placement are co-optimized.  
   Best bucket coverage: serialization/deserialization under acceleration, cross-PCIe penalties, collaborative CPU/accelerator partitioning.  
   Main limitation for this repo: the hardware model is PCIe-attached and on-NIC rather than on-die, and the paper is not tied to tonic or Rust async semantics.

## Tier 2: direct-adjacent papers that help explain one bucket well

### RR-Compound: RDMA-Fused gRPC for Low Latency, High Throughput, and Easy Interface

- **Venue / Year**: TPDS 2024 metadata path; public metadata and code were discoverable during the search
- **URL**: https://par.nsf.gov/biblio/10521121-rr-compound-rdma-fused-grpc-low-latency-high-throughput-easy-interface
- **Code**: https://github.com/pwrliang/grpc-rdma
- **Paper folder**: [`papers/rr-compound-rdma-fused-grpc-for-low-latency-high-throughput-and-easy-interface/README.md`](papers/rr-compound-rdma-fused-grpc-for-low-latency-high-throughput-and-easy-interface/README.md)
- **Why it matters**: this is one of the most direct open gRPC-adjacent engineering efforts to preserve the gRPC interface while replacing the transport path with RDMA-aware mechanisms.
- **What it explains well**: interface-preserving transport substitution, gRPC-over-RDMA engineering, and the distinction between API preservation and internal fast-path redesign.
- **Limitation**: the search surfaced metadata rather than a full directly inspectable paper text during this pass, so it is a weaker grounding source than the four papers above.

### Cornflakes: Zero-Copy Serialization for Microsecond-Scale Networking

- **Venue / Year**: SOSP 2023
- **URL**: https://amyousterhout.com/papers/cornflakes_sosp23.pdf
- **Paper folder**: [`papers/cornflakes-zero-copy-serialization-for-microsecond-scale-networking/README.md`](papers/cornflakes-zero-copy-serialization-for-microsecond-scale-networking/README.md)
- **Why it matters**: Cornflakes is not a gRPC paper, but it gives one of the clearest modern arguments that serialization cost is entangled with copy policy and buffer layout, not just with “encoding speed” in the abstract.
- **What it explains well**: hybrid zero-copy/copy tradeoffs, scatter-gather-aware serialization, and why not every field should be treated the same.
- **Limitation**: it is adjacent support for copy/serialization decomposition rather than direct gRPC evidence.

## How the shortlist maps onto the repo's tonic evidence

## 1. Queueing and runtime policy

The repo's tonic profiling shows that tiny RPCs are often dominated by control-path overhead rather than large payload movement. The paper that most directly helps here is **Cloud-Scale RPC Characterization**, because it makes queueing and RPC-stack tax first-class variables instead of collapsing them into a single “network latency” number.

## 2. Serialization and deserialization

The strongest literature support for a serialization-specific bucket is **A Hardware Accelerator for Protocol Buffers**. It should be treated as a first-class citation whenever the repo argues that protobuf work can dominate an RPC path even before transport issues are considered.

## 3. Copies, buffers, and payload layout

The local profiling evidence around `memmove`, `BytesMut`, and allocator growth is better matched by **Cornflakes** and by the methodology in **TF-gRPC-Bench** than by generic transport papers. The literature does not yet provide a perfect tonic-specific copy-path study, but these papers justify analyzing copy and layout as their own bucket instead of burying them inside “serialization”.

## 4. Compression

The most direct compression-aware evidence from the external review came from **Cloud-Scale RPC Characterization**, which explicitly treats compression as part of the RPC-processing CPU tax. The repo's current local findings are stronger than the literature here for tonic-specific behavior.

## 5. API-preserving acceleration

The most relevant precedent is **RPCAcc**, with **RR-Compound** as an additional gRPC-preserving transport-engineering point. Together they support a repo narrative in which users should not have to rewrite application semantics just to benefit from a faster RPC path.

## Annotated bibliography

### A Cloud-Scale Characterization of Remote Procedure Calls

- **Venue**: SOSP 2023
- **Authors**: Korakit Seemakhupt et al.
- **URL**: https://dl.acm.org/doi/10.1145/3600006.3613156
- **Use in this repo**: cite for the claim that tail RPC latency is often dominated by non-application RPC tax, especially queueing and RPC-stack overheads.
- **Why it is strong**: broad production measurement, explicit latency decomposition, explicit CPU-tax framing.
- **Why it is not enough alone**: internal Google RPC stack, not direct tonic/gRPC implementation analysis.

### A Hardware Accelerator for Protocol Buffers

- **Venue**: MICRO 2021
- **Authors**: Sagar Karandikar et al.
- **URL**: https://doi.org/10.1145/3466752.3480051
- **Use in this repo**: cite for the claim that protobuf serialization/deserialization is itself a sufficiently rich and important cost bucket to deserve separate treatment from transport.
- **Why it is strong**: fleet-scale protobuf characterization plus workload-derived benchmark plus end-to-end accelerator prototype.
- **Why it is not enough alone**: no direct queueing/runtime or HTTP/2/gRPC-stack analysis.

### Designing a Micro-Benchmark Suite to Evaluate gRPC for TensorFlow: Early Experiences

- **Venue**: BPOE / ASPLOS workshop 2018
- **Authors**: Rajarshi Biswas, Xiaoyi Lu, Dhabaleswar K. Panda
- **URL**: https://arxiv.org/abs/1804.01138
- **Use in this repo**: cite for methodology when motivating controlled gRPC experiments that separate transport effects from serialization effects.
- **Why it is strong**: direct gRPC focus, explicit serialized/non-serialized split, parameterized payload structure.
- **Why it is not enough alone**: specialized deep-learning communication workload.

### RPCAcc: A High-Performance and Reconfigurable PCIe-attached RPC Accelerator

- **Venue**: arXiv 2024
- **Authors**: Jie Zhang et al.
- **URL**: https://arxiv.org/abs/2411.07632
- **Use in this repo**: cite for the claim that accelerator-assisted RPC paths must co-design serializer/deserializer behavior with interconnect cost, not just move logic to hardware blindly.
- **Why it is strong**: modern RPC-acceleration framing, explicit PCIe-cost discussion, collaborative serializer design.
- **Why it is not enough alone**: non-top-tier venue in the current form and not tonic-specific.

### RR-Compound: RDMA-Fused gRPC for Low Latency, High Throughput, and Easy Interface

- **Venue**: TPDS 2024 metadata path
- **URL**: https://par.nsf.gov/biblio/10521121-rr-compound-rdma-fused-grpc-low-latency-high-throughput-easy-interface
- **Code**: https://github.com/pwrliang/grpc-rdma
- **Use in this repo**: cite as an interface-preserving gRPC-adjacent transport baseline.
- **Why it is useful**: unusually direct to the “keep the API, replace the path” design goal.
- **Why it is weaker in this review**: this pass grounded it mostly through metadata and code-page discovery rather than a directly read full paper text.

## Main gap relative to this repo

The literature is still thin on a single modern paper that jointly satisfies all of these constraints:

- open gRPC-like or tonic-like semantics
- stage-by-stage decomposition across serialization, copy, framing, scheduling, compression, and wire cost
- implementation-preserving rather than API-replacing optimization
- relevance to Rust async runtimes rather than custom network stacks alone

That gap is exactly why the repo's own tonic profiling work remains valuable. The external literature can justify the buckets, but it still does not replace a tonic-specific measurement campaign.

## Recommended citation posture for future repo writing

- cite **Cloud-Scale RPC Characterization** when talking about RPC tax, queueing, and tail behavior
- cite **A Hardware Accelerator for Protocol Buffers** when talking about protobuf as its own cost center
- cite **TF-gRPC-Bench** when defending controlled gRPC microbenchmark methodology
- cite **RPCAcc** when motivating accelerator-aware RPC co-design under deployable hardware constraints
- cite **Cornflakes** and **RR-Compound** as adjacent support, not as substitutes for the stronger primary sources above
