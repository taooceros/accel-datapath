# Paper Module Rebuild Analysis for Tonic Literature Deck

**Date**: 2026-04-13

## Goal

Provide a detailed per-paper analysis artifact that can drive a full rebuild of the literature deck's paper modules.

This report exists because the current deck became stronger as a repo-facing synthesis, but still remained too interpretation-first for self-study. The next deck revision should therefore be driven by a paper-by-paper teaching template rather than by further local polishing of summary slides.

The intended downstream use is:

1. revise the slide plan,
2. rebuild each paper module around actual mechanism and evidence,
3. keep repo interpretation explicit but secondary.

## Cross-paper diagnosis of the current deck

The current deck already gets several high-level things right:

- it names the correct core papers,
- it maps them to the right buckets,
- it explains why they matter to the repo,
- and it gives a plausible self-study structure.

But the review passes converged on one consistent problem:

> the current modules teach why the papers matter to Tonic more clearly than they teach what the papers themselves actually do.

The main missing ingredients are:

1. **Mechanism detail**
   - named techniques are present, but often not explained well enough to build a mental model.

2. **Evaluation setup**
   - readers need to know what is being compared, under what baseline or workload, before the numbers mean anything.

3. **Concrete findings**
   - one headline number is not enough. Most modules need 2 to 4 findings or one compact result table.

4. **Paper-specific visuals**
   - card summaries are not enough for self-study. The rebuilt modules need one real mechanism or methodology visual per major paper.

5. **Clear limits of evidence**
   - the deck should distinguish strong paper-grounded modules from weaker adjacent or metadata-grounded modules.

## Rebuild template for strong paper modules

Every primary paper module should follow this structure:

1. **Question / problem**
   - what the paper is trying to explain, measure, or accelerate

2. **Method / mechanism**
   - what the authors actually build, measure, or vary
   - ideally shown as a pipeline or system diagram

3. **Evaluation setup**
   - where the data comes from
   - what the benchmark or comparison structure is
   - what the baselines are

4. **Findings**
   - 2 to 4 concrete takeaways or quantitative results
   - enough context that the reader knows what comparison each number belongs to

5. **Interpretation for this repo**
   - what the paper changes in how we should characterize Tonic

6. **Limitations**
   - what the paper does not tell us

For weaker or adjacent papers, the same structure should be used in lighter form, with the reduced evidence strength stated explicitly.

## 1. A Cloud-Scale Characterization of Remote Procedure Calls

### Source strength

**Strong** for title, scope, decomposition structure, and several key fleet-scale results.

**Medium** for some slide-friendly summary numbers, because some convenient numerical highlights came from accessible review or talk text rather than a clean directly fetched full paper body.

### What the paper is doing

This is a fleet-wide observational paper about how RPCs behave in a large production cloud environment.

Its main purpose is not to introduce a new RPC mechanism. Its purpose is to challenge stale assumptions about what dominates RPC latency and where cloud RPC traffic actually lives.

The key structural idea is that end-to-end completion time should not be treated as application work plus undifferentiated “network delay.” Instead, the paper separates:

- **application processing**,
- **RPC latency tax**,
- then decomposes that tax into:
  - request/response queueing,
  - RPC processing plus network-stack work,
  - and network-wire time.

It also frames CPU consumption similarly, so stack cost, serialization, networking, and compression can remain visible as separate contributors.

### What the rebuilt module must teach

The rebuilt deck should teach three things clearly:

1. this is a **measurement paper**, not a mechanism paper,
2. its contribution is a **decomposition vocabulary** supported by production evidence,
3. its most important result is not simply “network matters,” but that **tail latency and high-overhead methods are often tax-dominated even when average tax is modest**.

### Mechanism / methodology details to show

- study shape: more than `10,000` methods, more than `1B` traces, roughly every 30 minutes, across nearly two years
- internal Google RPC production setting
- decomposition of completion time into application work vs RPC tax
- decomposition of tax into queueing, stack/network processing, and wire time
- CPU-tax framing rather than only latency framing

### Evaluation setup details to show

- fleet-scale observational study, not benchmark harness
- comparisons across methods, services, clusters, latency components, and cycle tax
- important distinction between average fleet view and high-overhead or tail-heavy methods

### Slide-worthy findings

Strongest slide-worthy findings from the accessible source trail:

1. more than `10,000` RPC methods and more than `1B` traces were analyzed, with monitoring over nearly two years
2. average RPC tax is modest fleet-wide, but high-overhead methods and tail behavior are much more tax-dominated
3. storage-heavy services dominate much of the traffic and the transferred-byte budget
4. the slowest methods account for disproportionate total RPC time relative to their request fraction

### Limitations

- internal Google RPC stack, not gRPC or tonic
- descriptive rather than prescriptive system design
- not a direct guide to byte-path instrumentation points in Rust async stacks

### What the module should emphasize for this repo

- keep runtime/queueing and stack tax visible
- do not collapse tail behavior into one end-to-end number
- treat wire contribution as only one piece of RPC cost

### Recommended deck module shape

**3 slides**:

1. **Decomposition concept**
   - show end-to-end -> application vs RPC tax -> queueing / stack / wire
2. **How the study earns authority**
   - production setting, scale, observation window, what is being compared
3. **Concrete findings and limits**
   - 2 to 4 bullet results plus “what this changes for Tonic” and “what it still does not answer”

## 2. A Hardware Accelerator for Protocol Buffers

### Source strength

**Strong**.

The current accessible source trail supports the paper identity, benchmark name, workflow, main prototype form, and headline numbers well.

### What the paper is doing

This paper argues that protobuf serialization and deserialization are rich enough and costly enough to deserve dedicated hardware support.

Its contribution is not just the hardware block. It is the sequence:

1. characterize protobuf behavior at Google scale,
2. derive a representative benchmark suite (`HyperProtoBench`),
3. design and evaluate a wire-compatible hardware accelerator integrated into a RISC-V SoC,
4. compare against software baselines.

That sequence is exactly why this paper matters to the repo. It shows how to turn fleet-scale profiling into a hardware-relevant benchmark and then into a serious prototype.

### What the rebuilt module must teach

The rebuilt deck should make the reader understand:

1. why protobuf is not a trivial or uniform workload,
2. why `HyperProtoBench` exists,
3. what the accelerator is actually meant to replace or speed up,
4. what the speedup numbers are comparing against.

### Mechanism / methodology details to show

- Google-scale profiling of protobuf usage
- workload diversity and field-mix motivation
- benchmark derivation into `HyperProtoBench`
- RTL accelerator integrated into a RISC-V SoC
- modified protobuf software stack while preserving wire compatibility

The module should also explain why “wire-compatible” matters: the point is to accelerate protobuf processing without changing the application-visible protocol format.

### Evaluation setup details to show

- evaluation on six synthetic benchmarks in `HyperProtoBench`
- comparisons against a BOOM-based baseline SoC and a Xeon-based server
- end-to-end evaluation context rather than raw microbench claims alone

### Slide-worthy findings

1. average `6.2x–11.2x` speedup over the BOOM-based baseline SoC
2. average `3.8x` speedup over a Xeon-based server
3. six synthetic benchmarks derived from Google service-side protobuf behavior
4. the open-sourced RTL claim is itself useful context for why this paper became influential

### Limitations

- near-CPU SoC context, not a NIC-attached drop-in design
- not a gRPC end-to-end paper
- does not answer queueing/runtime/HTTP2 questions

### What the module should emphasize for this repo

- serialization deserves its own bucket
- workload characterization should precede optimization claims
- protobuf acceleration does not eliminate the need to decompose transport, runtime, and buffer movement

### Recommended deck module shape

**3 slides**:

1. **Why protobuf needs separate study**
   - workload diversity, bucket coverage, and the profiling-to-benchmark logic
2. **Workflow and accelerator mechanism**
   - profiling -> `HyperProtoBench` -> RTL accelerator -> wire-compatible integration
3. **Results and interpretation**
   - numbers, baselines, what they prove, and what they do not prove for Tonic

## 3. TF-gRPC-Bench

### Source strength

**Strong**.

This paper has a clearer accessible experiment structure than the current deck reflects.

### What the paper is doing

This is a methodology paper about how to evaluate TensorFlow’s gRPC communication path without repeatedly running full training jobs.

It starts from real TensorFlow traffic, then derives a benchmark suite that isolates the communication path more directly. That makes it a “how to ask the question” paper rather than a “new RPC mechanism” paper.

### What the rebuilt module must teach

The rebuilt deck should make the reader understand:

1. what the three microbenchmark classes are,
2. why serialized vs non-serialized mode matters,
3. what iovec distribution is controlling,
4. what the transport comparison actually demonstrated.

Right now this is one of the biggest gaps between importance and teaching quality.

### Mechanism / methodology details to show

- start from TensorFlow-over-gRPC traffic analysis on popular models
- derive three benchmark classes:
  - point-to-point latency
  - point-to-point bandwidth
  - parameter-server throughput
- vary:
  - serialized vs non-serialized mode
  - iovec distribution / payload shape
  - transport type

The rebuilt module should make explicit that non-serialized mode is used to isolate transport-side effects from serialization overhead.

### Evaluation setup details to show

- cluster comparisons across 40G Ethernet, 10G Ethernet, IPoIB, and RDMA
- separate benchmark classes rather than a single “performance” number
- workload-structured microbenchmarks derived from TensorFlow communication patterns

### Slide-worthy findings

The external pass surfaced several concrete results worth showing:

1. for `64 KB` serialized payloads, RDMA cuts point-to-point latency by about `40%` versus 40G Ethernet and IPoIB
2. in non-serialized mode on one cluster, RDMA cuts skewed-payload latency by about `59%` versus Ethernet and `56%` versus IPoIB
3. on another cluster, RDMA cuts latency by about `78%` versus 10G Ethernet and `69%` versus IPoIB
4. parameter-server throughput improves by roughly `4.1x` versus 40G Ethernet, `3.43x` versus IPoIB, and `5.9x` versus 10G Ethernet in the reported settings

### Limitations

- TensorFlow-specific traffic model
- 2018-era stack and transport context
- not a direct tonic paper

### What the module should emphasize for this repo

- benchmark derivation should be grounded in real workload structure
- control payload formatting separately from transport choice
- matched experiment design is part of the contribution, not just a setup detail

### Recommended deck module shape

**3 slides**:

1. **Why full-application measurement is the wrong starting point**
   - TensorFlow traffic -> benchmark derivation
2. **Actual experiment matrix**
   - benchmark classes, serialized/non-serialized, iovec shape, transport
3. **Reported findings and how to reuse the method**
   - result table plus explicit measurement-discipline lessons for Tonic

## 4. RPCAcc

### Source strength

**Strong** for the overall system story and several quantitative results.

### What the paper is doing

This paper asks whether RPC acceleration can remain useful under realistic **PCIe-attached** deployment constraints instead of assuming an ideal on-die or special-interconnect setting.

Its most important teaching value is that hardware acceleration is not one idea. It is a constrained co-design problem around movement cost.

### What the rebuilt module must teach

The rebuilt deck should make the reader understand:

1. what problem PCIe introduces,
2. what the three named techniques actually do,
3. how the host, PCIe path, accelerator, and data placement interact,
4. which result belongs to which experiment.

### Mechanism / methodology details to show

- PCIe-attached on-NIC accelerator context
- target-aware deserializer
- memory-affinity CPU-accelerator collaborative serializer
- automatic field-update technique
- programmable compute units plus RoCE-based transport path

The mechanism visual should be an actual datapath diagram, not a list of features.

### Evaluation setup details to show

- HyperProtoBench use
- end-to-end cloud workload use
- image-compression latency case
- comparisons against a baseline RPC accelerator and other reference designs
- FPGA prototype context, especially U280

### Slide-worthy findings

Strongest accessible findings:

1. Google fleet motivation: RPC processing occupies about `7.1%` of CPU cycles
2. HyperProtoBench result: about `2.3x` lower RPC-layer processing time than a comparable accelerator baseline
3. end-to-end cloud workload throughput improves by about `2.6x`
4. image-compression workload cuts average latency by about `2.6x`, p99 by about `1.9x`, and reduces serialization time by about `4.3x` geometric mean

### Limitations

- 2024 arXiv preprint
- FPGA prototype, not deployed production hardware
- not tonic-specific and not Rust-async-specific

### What the module should emphasize for this repo

- accelerator results only make sense with traversal-aware design
- serialization, placement, and interconnect should be taught together
- do not use a single top-line throughput number as the whole story

### Recommended deck module shape

**3 slides**:

1. **Deployment problem and bottleneck model**
   - why PCIe breaks naive RPC offload
2. **System mechanism**
   - host / serializer / PCIe / accelerator / deserializer / field-update datapath
3. **Results and interpretation**
   - result table keyed by benchmark or workload, then repo lesson and limitations

## 5. Cornflakes

### Source strength

**Mixed**.

The idea and general mechanism are well grounded. The strongest conveniently accessible numbers came more clearly from the HotOS precursor path than from directly recovered SOSP evaluation text.

### What the paper is doing

Cornflakes asks whether commodity NIC scatter-gather can reduce or avoid serialization-related copy and flattening costs for microsecond-scale networking.

Its core point is not that zero-copy is always better. Its point is that the best design is **hybrid**: use scatter-gather when it helps, and ordinary copies when the bookkeeping cost would dominate.

### What the rebuilt module must teach

The rebuilt deck should make the reader understand:

1. why pure zero-copy is not always optimal,
2. what the hybrid decision logic is trying to exploit,
3. why this paper belongs in the copy/layout bucket rather than as a transport substitute.

### Mechanism / methodology details to show

- schema-driven serialization library
- networking runtime integration
- hybrid decision between copy-based flattening and scatter-gather-based transmission
- the idea that data movement and layout policy are inseparable from serialization cost

### Evaluation setup details to show

- microsecond-scale networking context
- custom KV-store path and Redis path in the accessible source trail
- be explicit that the paper is adjacent support rather than direct gRPC evidence

### Slide-worthy findings

Grounded findings available from the accessible source trail:

1. `15.4%` higher throughput on a Twitter cache trace in a custom KV store
2. `8.8%` higher throughput relative to Redis serialization inside Redis
3. the precursor path reports about `9.15 Gbps` highest throughput while staying under `15 μs` tail latency
4. Mellanox CX-5 scatter-gather support for many discontiguous chunks is part of the mechanism intuition

### Limitations

- not a gRPC paper
- strongest numeric details in the accessible trail partially come from the precursor path
- should not be used as a substitute for a tonic-specific decomposition claim

### What the module should emphasize for this repo

- copy policy and serialization layout must stay visible
- zero-copy is a regime-dependent design choice, not a universal answer

### Recommended deck module shape

**2 slides**:

1. **Hybrid copy vs scatter-gather mechanism**
   - explain when each path is preferred
2. **Evidence and implication**
   - evaluation context, numbers, and why this sharpens the copy/layout bucket

## 6. RR-Compound

### Source strength

**Weak by the standard needed for a teaching module**.

The paper identity, repo, and compatibility framing are confirmed. The accessible evidence for mechanism and evaluation remains much thinner than for the other papers.

### What the paper is doing

At the current evidence level, RR-Compound is best understood as a compatibility-first gRPC-over-RDMA design point.

The deck should not pretend we have a full mechanism or evaluation story if we do not.

### What the rebuilt module must teach

The module should teach exactly what is known and exactly what is not:

1. the design goal is API-preserving acceleration,
2. the current pass confirms the compatibility-first framing,
3. the current pass does **not** recover enough paper text to teach the evaluation deeply.

### Mechanism / methodology details to show

- fully compatible gRPC drop-in framing from the artifact repo
- RDMA-enabled internal fast path
- runtime-tunable transport knobs available in the repo

### Evaluation setup details to show

- explicitly state that the current repo grounding is metadata and artifact-repo level, not full directly read paper-level evaluation

### Slide-worthy findings

At this stage, only qualitative or artifact-configuration details are safe:

- compatibility-first gRPC-over-RDMA framing
- RDMA disabled by default in the artifact
- one polling thread by default
- `500 μs` busy-poll timeout
- `4096 KB` ring buffer per connection

These are not the same thing as paper evaluation results. The rebuilt deck should say so.

### Limitations

- weakest source in the current set
- not enough accessible paper text recovered for a strong teaching module

### What the module should emphasize for this repo

- include only as design-space context
- keep qualitative unless stronger paper ingestion is done later

### Recommended deck module shape

**1 slide only**:

1. **Compatibility-first design-space note**
   - problem, what is locally supported, what is missing, and why this remains adjacent support rather than a core teaching module

## Recommended plan changes

The deck plan should be updated to reflect:

1. three-slide modules for the four strongest papers,
2. two-slide treatment for Cornflakes,
3. one explicit low-confidence qualitative slide for RR-Compound,
4. more compact repo-synthesis slides after the literature teaching section,
5. per-paper mechanism visuals and result tables as required components rather than optional guidance.

## Bottom line

The next deck revision should stop treating the literature section as a polished summary and instead treat it as a set of mini-teaching units.

If the rebuild follows this report, the resulting deck should finally let a reader learn:

- what each paper is trying to solve,
- how it actually studies or accelerates the RPC path,
- what the most important evidence is,
- and what the paper changes in how Tonic should be characterized.
