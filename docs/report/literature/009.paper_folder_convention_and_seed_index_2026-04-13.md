# Paper folder convention and seeded index

**Date**: 2026-04-13
**Type**: Convention plus seed index
**Purpose**: define the paper-oriented literature layout for KB-searchable markdown under `docs/report/literature/`, then list the initial Tonic literature backfill set.

## Why this exists

The active Tonic literature flow already has strong cross-paper artifacts:

- `007.grpc_cost_breakdown_2026-04-12.md` for the cost-bucket shortlist
- `008.paper_module_rebuild_analysis.md` for the paper-by-paper teaching structure

Those reports are good synthesis layers, but they are not stable homes for paper-specific explanation pages. This convention adds that missing layer without moving or renaming the existing review artifacts.

## Convention

Paper-oriented literature material lives under:

`docs/report/literature/papers/<paper-slug>/README.md`

### Rules

1. keep paper-specific explanation in the paper folder,
2. keep topic-oriented thesis notes in `docs/related_work/`,
3. keep cross-paper ranking, synthesis, and planning in numbered report files under `docs/report/literature/`,
4. use one stable folder per paper,
5. use `README.md` as the paper folder entry point so the folder remains readable and searchable.

### Slug rule

Use lower-case kebab-case derived from the canonical paper title. Preserve distinctive acronym tokens when they are part of the title, for example `rpcacc` and `rr-compound`.

The slug should change only if the canonical paper identity changes, not when the repo later adds stronger notes or more citations.

### Paper README structure

Each paper folder `README.md` should use this shape:

1. paper identity
2. why it matters here
3. problem
4. method / mechanism
5. evaluation setup
6. key findings
7. limits and confidence
8. repo takeaways
9. source links within repo

### Optional paper artifacts

When a paper is locally acquired, keep the acquisition artifacts inside the same paper folder:

- `paper.pdf` for the local source PDF
- `paper.txt` for extracted plain text when extraction is feasible with the available tooling

If extraction quality is limited or the available CLI tooling stalls on a specific PDF, keep the artifact but state the limitation explicitly in the paper folder rather than pretending the text is clean or complete.
