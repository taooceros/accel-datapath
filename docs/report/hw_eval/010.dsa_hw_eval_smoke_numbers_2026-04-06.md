# `hw-eval` DSA raw-number capture for interview slide (2026-04-06)

## Summary

A fresh `hw-eval` run on real DSA hardware produced a concise raw-hardware lower
bound suitable for the Google interview deck.

## Command run

```bash
launch ./target/release/hw-eval --accel dsa --sizes 64 --iterations 1000 --max-concurrency 32 --json
```

Run directory: `hw-eval/`

## Environment metadata

- accelerator: `dsa`
- device: `/dev/dsa/wq0.0`
- dedicated work queue: `true`
- pinned core: `3`
- iterations: `1000`
- cold cache: `false`

## Key numbers

### Single-op latency at 64 B

- `noop` median latency: `649 ns`
- `memmove` median latency: `955 ns`
- `copy_crc` median latency: `1053 ns`

### Throughput at 64 B

- plain sliding-window `memmove` peaks at `9.56 Mops/s` at concurrency `32`
- plain sliding-window `copy_crc` peaks at `5.89 Mops/s` at concurrency `16`
- pipelined batched DSA `memmove` peaks at `48.39 Mops/s` with `batch_size=128`, concurrency `4`

## Recommended slide number

For a concise interview slide, the clearest number is:

> raw DSA memmove reached **48.4 Mops/s** on 64 B transfers with pipelined batched submission (`batch=128`, `c=4`)

This fits the talk's batching-driven message better than the plain sliding-window
number because it directly shows the raw hardware regime after submission
amortization.

## Interpretation boundary

This is a raw-hardware lower-bound measurement from `hw-eval`, not an end-to-end
RPC result. It is useful in the slide as evidence that once batching is used,
the remaining question is how much of that headroom survives the software path.
