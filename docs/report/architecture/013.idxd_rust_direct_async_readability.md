# idxd-rust direct async readability map

**Milestone:** M009 — Hardware Rust readability cleanup  
**Slice:** S02 — Clarify `idxd-rust` direct async runtime flow  
**Status:** S02 guard plus T03 operation module split; this report documents the target responsibilities, the current named registration/submission helper sequence, and the internal operation-owned lifecycle module.  
**Reader:** A maintainer or future agent preparing to move or rename direct async runtime code in `idxd-rust`.  
**Post-read action:** Change direct async source organization only along the named seams, then prove that the operation-owned descriptor/completion/source/destination lifetime and submit-now / complete-later ENQCMD trace still hold with host-free checks.

## Purpose

This report advances **R018** for S02 by turning the S01 direct async inventory and the M007 direct completion-record contract into a smaller responsibility map for the current `idxd-rust` direct async runtime. It is intentionally written before runtime code motion so later tasks have a stable target and a mechanical guard.

S02 is readability work. It may name phases, split internal helpers, and move internal direct-runtime responsibilities into better-labeled modules. It must not change public async APIs, direct failure accessors, request recovery semantics, destination-length safety, fake backend behavior, verifier artifacts, or prepared-host claim criteria.

The central invariant is: accepted direct async operations keep operation-owned descriptor/completion/source/destination lifetime until terminal completion or runtime cleanup. Recovery of owned request buffers is allowed only before hardware acceptance or where the current typed failure path already proves it is safe.

## Non-change boundaries for S02

S02 direct async readability work must preserve these boundaries unless a later task deliberately versions the contract and updates tests/report guards together:

- No public async allocation convenience constructor in the canonical async API.
- No public borrowed copy-back helper in the canonical async API.
- No public payload logging, source payload byte logging, destination payload byte logging, payload dump fields, or payload-content examples.
- No blocking `memmove_uninit` future resolution for the direct completion-driven async path.
- No Tonic prerequisite for proving the direct async substrate.
- No preallocated completion-record registry or slot pool as the current v1 design.
- No unbounded async-context spin loop for ENQCMD rejection or completion polling.
- No weakening of the operation-owned descriptor/completion/source/destination lifetime invariant.
- No prepared-host hardware success claim from ordinary-host static checks or fake backend tests.

MOVDIR64/MOVDIR64B serialized submission, worker-owned batching runtimes, registry pooling, public borrowed helpers, and public destination allocation helpers remain future work outside the current direct Tokio v1 path.

## Guard contract

`idxd-rust/scripts/check_m009_s02_direct_async_readability.sh` is the mechanical guard for this report. It must remain host-free and deterministic. It checks that:

1. the report exists and names R018, S02, ordinary-host limits, and prepared-host limits;
2. the report names every current direct async source/test owner listed in the source input section;
3. every responsibility phase remains present: public owner/session wiring, request validation/recovery, registration and initial ENQCMD submission, operation-owned lifetime, monitor polling, completion/retry/recovery, and fake/backend seams;
4. the operation-owned descriptor/completion/source/destination lifetime phrase remains visible;
5. future resolution remains completion-snapshot-driven and not blocking-wrapper-driven;
6. no-payload diagnostic rules remain visible; and
7. stale claims are rejected with labeled `verdict=fail` output.

The Cargo test `idxd-rust/tests/direct_async_readability_contract.rs` launches this script through Cargo so ordinary test runs can catch stale responsibility claims before runtime code motion continues.
