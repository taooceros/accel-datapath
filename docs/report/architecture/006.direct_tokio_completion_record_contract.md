# Direct Tokio completion-record contract

## Architecture name

The M007 direct async architecture is **direct completion-driven async memmove**, also described as a **submit-now / complete-later ENQCMD model**.

A Tokio caller submits an owned memmove request, the implementation builds operation-owned hardware state, ENQCMD accepts the descriptor, a long-lived Tokio monitor observes the operation's completion record, and the awaiting future resolves only after the monitor classifies the observed completion. This is the normative S02 target. The current worker-thread wrapper remains useful as prior behavior and comparison material, but it is not the direct architecture.

## Reader and post-read action

The reader is the engineer implementing M007/S02 inside `idxd-rust`. After reading this contract, they should be able to split synchronous submit/poll/validate behavior into direct submit-now / complete-later operation state and write host-free proof tests for registration, completion observation, retry, cancellation, shutdown, and failure preservation.

## Requirement coverage

### Owned by this design

- **R002 — Direct Tokio IDXD memmove integration.** The direct model keeps the public Tokio-facing request shape while replacing hidden synchronous completion with operation registration, ENQCMD submission, monitor-observed completion, and future signaling.
- **R003 — Per-request completion record registration.** The first implementation dynamically creates/registers one completion record for each submitted request. A preallocated registry or pool is explicitly not the v1 design.
- **R008 — Failure information preservation.** Request, submission, monitor, lifecycle, retry, and hardware completion context must be preserved where practical and must never include source or destination payload bytes.

### Downstream constraints this design imposes

- **R004 — Completion monitor resolves futures.** Futures must resolve because `CompletionMonitor` observes `DsaCompletionRecord` state, not because synchronous `memmove_uninit` returned.
- **R005 — ENQCMD direct submission.** Direct Tokio v1 uses ENQCMD-oriented submission suitable for many Tokio runtime threads. MOVDIR64/MOVDIR64B serialized submission remains future worker/runtime work.
- **R009 — Direct baseline before Tonic.** The direct proof must not require Tonic. Tonic integration may build on the direct substrate after the substrate is proven and benchmarked.

## Preserved M006 API constraints

The caller-facing async request/result shape remains the M006 bytes contract unless a later explicit decision changes it:

- The canonical request constructor is `AsyncMemmoveRequest::new(source: Bytes, destination: BytesMut)`.
- The source argument stays first and the destination remains explicit.
- The result returns destination ownership and validation metadata.
- Destination allocation stays at the call site; there is no allocation convenience constructor in the canonical API.
- There is no public borrowed copy-back helper in the canonical async API.
- `BytesMut` readable length advances only after successful completion observation and validation.

Pre-submit validation remains responsible for malformed request shapes such as zero-length source or destination spare capacity that is smaller than the requested transfer. These cases should fail before descriptor submission or completion-record registration and should return typed request diagnostics with recoverable owned buffers where possible.

## Operation ownership model

S02 should introduce internal operation state comparable to the placeholder name `PendingMemmoveOp`. The name is descriptive, not a public API requirement.

A `PendingMemmoveOp` owns or keeps alive for the full hardware lifetime:

- an aligned DSA descriptor;
- an aligned DSA completion record;
- the source `Bytes`;
- the destination `BytesMut`;
- the original destination readable length;
- requested byte count;
- remaining source and destination offsets;
- retry count and maximum retry configuration;
- device path or validation metadata needed to build reports;
- completion sender, waker, or equivalent future-signaling state;
- the current lifecycle classification needed for cleanup and diagnostics.

The operation is pending from the moment ENQCMD accepts the descriptor until a terminal success or failure is observed and classified. During that interval, dropping the awaiting future does not drop hardware-referenced state.

## Unsafe invariants

The direct implementation crosses raw-pointer, FFI, and device-DMA boundaries. It must document these invariants at each unsafe boundary:

- The descriptor is valid, initialized, and 64-byte aligned at every ENQCMD submission.
- The completion record is valid, writable by hardware, zeroed/reset before submission, and 32-byte aligned.
- The completion record remains allocated and writable until terminal hardware completion is observed.
- The descriptor remains allocated and unchanged for each submitted operation while hardware may read it.
- Source memory remains allocated and immutable for the bytes still reachable by hardware.
- Destination spare capacity remains allocated, writable, and not aliased as initialized data while hardware may write it.
- Destination readable length is not advanced before validated success.
- Error, cancellation, and shutdown paths must not expose uninitialized destination bytes.
- Page-fault continuation descriptors use offsets derived from the observed completion snapshot and never exceed the original request bounds.

The current synchronous state machine already demonstrates the pieces to preserve: request validation, descriptor fill, completion reset, submission, snapshot classification, page-fault touch/retry, and post-copy verification. S02 should split those semantics into reusable direct-async pieces rather than duplicating behavior loosely.

## ENQCMD submission and backpressure

Direct Tokio v1 is ENQCMD-oriented. The implementation should use a primitive that exposes ENQCMD accept/reject state instead of hiding rejection behind an unbounded spin loop.

Required behavior:

1. Build and reset operation-owned descriptor/completion state.
2. Attempt ENQCMD submission.
3. If ENQCMD accepts, register or retain the accepted operation for monitor progress and return a pending future to the caller.
4. If ENQCMD rejects because the shared work queue is busy, retry with a bounded or adaptive policy that yields/backoffs in Tokio context.
5. If the retry budget is exhausted, fail with a typed submission/backpressure failure and recover request buffers where hardware has not accepted the descriptor.

The direct path must not spin indefinitely on an executor thread. It must not silently fall back to MOVDIR64 or MOVDIR64B for direct Tokio v1. Dedicated-queue or MOVDIR64 behavior may remain in synchronous or future software-serialized worker/runtime designs, but it is outside this direct baseline.

## CompletionMonitor progress model

`CompletionMonitor` is a descriptive internal placeholder for the long-lived Tokio progress task. It should be pure Tokio first if feasible and should avoid blocking helper loops that busy-wait until completion.

The monitor owns or has exclusive progress authority over the pending set. Its loop should:

- drain newly accepted/registered operations;
- scan pending completion records using volatile status reads or helper snapshots;
- build a `CompletionSnapshot` when status changes;
- classify completion through existing validation semantics;
- notify futures outside long-held locks;
- apply explicit `yield_now`, sleep, or adaptive backoff when no progress occurs;
- continue making progress even if an awaiting receiver was dropped;
- surface monitor death or registration closure as an explicit async failure class.

The monitor must not hold locks across `.await` unless the lock is specifically designed for that usage and contention behavior is understood. Prefer a single monitor owner plus message passing for registrations over shared mutable operation state spread across submitters.

## Future signaling

A submitted future resolves only when the monitor observes terminal completion state in the operation's completion record and classifies that state.

Success path:

1. Monitor observes success status.
2. Monitor validates the copied prefix against the request contract.
3. Monitor advances the destination `BytesMut` readable length from the original length by the requested byte count.
4. Monitor sends the destination and validation report to the waiting future.

Failure path:

1. Monitor observes timeout, malformed completion, retry exhaustion, non-success completion status, byte mismatch, monitor failure, lifecycle failure, or submission failure.
2. It preserves typed context in the async error surface.
3. It does not advance destination readable length on hardware or validation failure.
4. It never logs source or destination payload bytes.

Direct futures must not resolve because a blocking synchronous wrapper returned. In particular, `DsaSession::memmove_uninit` is not the future-resolution mechanism for the direct completion-driven async memmove path.

## Page-fault retry continuation

Page-fault retry behavior from the synchronous path remains part of the direct async contract.

When the monitor observes a page-fault completion that validation classifies as retryable, it should:

1. snapshot status, result, bytes completed, and fault address;
2. touch the faulted page through the existing fault-touch primitive;
3. update remaining source/destination offsets and remaining byte count from `MemmoveRetry`;
4. increment retry count;
5. reset the same operation's completion record;
6. refill a continuation descriptor for the remaining range;
7. resubmit through ENQCMD using the same bounded/yielding backpressure policy;
8. keep the same logical `PendingMemmoveOp` pending until terminal success or failure.

Page-fault retry exhaustion remains a typed failure. Malformed page-fault completions remain typed malformed-completion failures. Retry must not expose partial destination bytes as initialized user-visible output.

## Cancellation and dropped futures

Cancellation after hardware acceptance cannot cancel DMA. Dropping the awaiting future or response receiver only means there may be nobody left to receive the result.

Required behavior:

- If cancellation happens before ENQCMD acceptance, the implementation may recover the owned request and classify the cancellation as pre-submit or lifecycle behavior.
- If cancellation happens after ENQCMD acceptance, the monitor must keep descriptor, completion record, source, and destination alive until terminal completion is observed.
- After terminal observation, the monitor may drop the completed result if the receiver is gone, but it should record/drop with a dropped-receiver lifecycle classification when exposed through diagnostics or tests.
- Cancellation must not be reported as hardware cancellation unless the hardware really provides and completes such a cancellation mechanism. M007 direct v1 does not assume such a mechanism.

## Shutdown behavior

Shutdown has two distinct phases: refusing new work and resolving accepted work.

Required behavior:

- New submissions after owner shutdown starts fail with an explicit lifecycle/shutdown classification.
- Operations not yet accepted by ENQCMD should be returned or failed with recoverable request context where practical.
- Operations already accepted by hardware should be drained to terminal completion when practical because their buffers and completion records cannot be freed safely before completion.
- If shutdown cannot drain accepted pending operations, the failure must be explicitly classified as abandoned/shutdown-with-pending rather than silently dropping state.
- Monitor shutdown, registration channel closure, or lost pending-operation state must be visible to tests and diagnostics.

## Failure taxonomy and no-payload diagnostics

The direct async path should keep expected failures in `Result`-style typed error surfaces. Existing `MemmoveError` variants should remain authoritative for request validation, queue-open, completion timeout, malformed completion, page-fault retry exhaustion, hardware completion status, and byte mismatch where they fit.

S02 may extend async-specific failure kinds for direct behavior. At minimum, diagnostics should distinguish:

- request validation failure;
- queue open or device setup failure;
- ENQCMD busy/rejection/backpressure exhaustion;
- descriptor submission failure;
- registration channel closed;
- monitor stopped or monitor task failed;
- owner shutdown before acceptance;
- shutdown with accepted pending operations;
- dropped receiver after hardware acceptance;
- completion timeout;
- malformed completion;
- page-fault retry exhaustion;
- non-success hardware completion status;
- post-copy verification byte mismatch.

Allowed diagnostic fields include phase, lifecycle class, requested byte count, destination capacity/length, retry count, status byte, result byte, bytes completed, fault address, device path, and comparison target. The implementation must never log source payload bytes or destination payload bytes. The implementation must forbid payload logging in report artifacts, verifier output, test failure messages, and benchmark evidence.

## Explicit stale-claim rejections

This contract rejects the following stale or conflicting claims for M007 direct Tokio v1:

- Resolving direct async futures from synchronous `memmove_uninit` completion.
- Making Tonic a prerequisite for proving the direct IDXD substrate.
- Treating a preallocated completion-record registry or slot pool as the first-version design.
- Allowing MOVDIR64 or MOVDIR64B fallback in direct Tokio v1.
- Reintroducing public borrowed copy-back helpers into the canonical async API.
- Reintroducing allocation convenience APIs into the canonical async API.
- Using unbounded async-context spin loops for ENQCMD rejection or completion polling.
- Allowing source or destination payload byte logging.

These rejections are intentionally phrased so mechanical guards can fail if future text reintroduces the stale claims as active design.

## S02 implementation checklist

S02 should not claim done until host-free tests or guards prove the following:

1. Request validation rejects zero-length and destination-too-small requests before submission/registration.
2. A submitted operation creates dynamic per-request completion record state.
3. `PendingMemmoveOp` owns descriptor, completion record, source `Bytes`, destination `BytesMut`, original destination length, requested length, retry state, and future signal state.
4. ENQCMD acceptance registers/retains the operation for monitor progress.
5. ENQCMD rejection uses bounded/adaptive yield/backoff and exposes typed submission failure on exhaustion.
6. The future does not resolve until `CompletionMonitor` observes a completion record transition.
7. Success advances `BytesMut` readable length only after successful completion observation and validation.
8. Hardware failure does not advance destination readable length and preserves completion snapshot/status context.
9. Page-fault retry touches the faulted page, updates offsets, resets completion, refills descriptor, resubmits, and preserves retry count.
10. Page-fault retry exhaustion returns a typed failure.
11. Dropping the awaiting future after hardware acceptance still drains and cleans up operation-owned state.
12. Shutdown refuses new submissions and drains or explicitly classifies accepted pending operations.
13. Monitor death or registration closure is surfaced as a failure, not a hung future.
14. Tokio multi-producer submission does not depend on the old single worker-thread serialization test contract.
15. Diagnostics include submission phase, monitor phase, retry count, completion snapshot/status, lifecycle/shutdown classification, and no payload bytes.

Expected host-free proof obligations include fake completion-record transitions, fake ENQCMD accept/reject behavior, fake retry completion snapshots, dropped-receiver cleanup, shutdown classification, and destination-length assertions. Live hardware proof belongs after these host-free contracts pass.

## Explicitly deferred work

Deferred beyond S01/S02 direct v1:

- Tonic/RPC integration.
- Preallocated completion-record registry or slot pool.
- Worker-owned batching runtime.
- MOVDIR64/MOVDIR64B serialized submission path.
- Sophisticated scheduling/fairness policy.
- Criterion benchmark integration.
- Borrowed async zero-copy public helper.
- Allocation convenience constructors.
- Public copy-back helpers.

Workers are not rejected forever. A future worker/runtime may be the right abstraction when it owns batching, fairness, registry pooling, and serialized submission policy. It is simply not the direct completion-driven async memmove baseline for M007.

## Reader-test result

A cold reader should leave with one concrete implementation shape: preserve the public bytes request, validate before submission, build a dynamic per-request `PendingMemmoveOp`, submit through ENQCMD with bounded/yielding backpressure, let `CompletionMonitor` observe completion records and signal futures, preserve page-fault retry, keep hardware-referenced state alive across cancellation/shutdown, and report metadata-rich failures without payload bytes.
