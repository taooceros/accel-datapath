Status: S06/T01 errata for M007 validation-scope interpretation. This report is a validation aid, not new runtime, benchmark, or hardware evidence.
Date: 2026-04-28

## Reader and action

Reader: the future validator, reviewer, or planner deciding whether M007 requirement coverage is complete enough to close the milestone without silently expanding the milestone into deferred worker-runtime work.

Post-read action: classify each touched requirement as one of four outcomes: `PASS active-scope coverage`, `WARN deferred`, `PRESERVED GUARD`, or `LABEL DRIFT`, then apply that classification when reading M007 reports and GSD requirement metadata.

## Status and scope

This report narrows validation semantics for the M007 direct Tokio baseline. It does not add new implementation scope or new claim evidence.

## Remediation verdict

The M007 validation mismatch is a scope-classification problem, not a missing implementation slice. Active M007 coverage is limited to the direct Tokio IDXD baseline and its fail-closed proof surfaces. Deferred worker-runtime, RPC, registry, scheduling, and borrowed zero-copy work must remain future work. R015 must remain a no-payload evidence/logging guard, not active feature coverage.

The corrected scope is:

- `PASS active-scope coverage` for R002, R003, R004, R005, R006, R007, R008, and R009.
- `WARN deferred` for R010, R011, R012, R013, and R014.
- `PRESERVED GUARD` for R015.
- `LABEL DRIFT` for older S03/S04 prose that used R014/R015 as benchmark and verifier coverage labels before the requirement registry separated direct-baseline active scope from deferred and guard-only requirements.

## Validation rule

M007 validation should apply the following rule:

1. Fail closed if R002, R003, R004, R005, R006, R007, R008, or R009 lacks current evidence.
2. Emit `WARN deferred` for R010, R011, R012, R013, or R014 when they appear as future scope.
3. Preserve R015 as `PRESERVED GUARD` and reject any evidence surface that exposes copied caller data rather than metadata.
4. Treat historical R014/R015 benchmark/verifier wording as `LABEL DRIFT`, not as active M007 scope expansion.
5. Keep claim gates explicit: software diagnostics stay `claim_eligible=false`; ordinary-host hardware checks may be `expected_failure`; prepared-host claims require `verdict=pass claim_eligible=true` with verifier-accepted direct evidence.

This rule intentionally does not introduce prepared-host throughput or latency claims, Tonic/RPC evidence, worker batching, MOVDIR64/MOVDIR64B serialization, registry pooling, scheduling fairness, or borrowed zero-copy wrapper claims.
