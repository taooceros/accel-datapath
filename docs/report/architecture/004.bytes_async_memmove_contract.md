# Bytes-based async memmove API contract

Status: review-ready contract for M006/S02
Date: 2026-04-27

## Reader and post-read action

Reader: an internal executor migrating the async memmove API and a reviewer checking that the migration preserves diagnostic behavior.

After reading this document, the reader should be able to implement and review the S02 async API migration without inferring source/destination ordering, destination ownership, validation timing, failure classification, or submission-path requirements from prior conversation.

## Expected executor skills

- `codemogger` for confirming the live async API, validation helpers, and host-free tests before editing code.
- `unsafe-checker` for evaluating `BytesMut` initialized-length, spare-capacity, aliasing, cancellation, and completion-safety invariants.
- `write-docs` for keeping the contract reader-oriented and directly actionable.

## Contract summary

M006 migrates the public async memmove API to explicit owned `bytes` buffers:

```rust
AsyncMemmoveRequest::new(source: Bytes, destination: BytesMut)

AsyncMemmoveResult {
    destination: BytesMut,
    report: MemmoveValidationReport,
}
```

`AsyncMemmoveRequest::new(source: Bytes, destination: BytesMut)` is the only canonical public constructor. The source length is the requested transfer size. The caller owns and supplies the destination buffer. The async operation returns the full destination in `AsyncMemmoveResult`, together with the `MemmoveValidationReport` produced by the validation path.

The first M006 implementation should submit hardware work inline via `enqcmd`. Inline submission is acceptable for the first version and allows multiple submitting threads under the `enqcmd` model. M006 must not require a software aggregation thread, software batching, or `movdir64` support.

## Current-source delta from the pre-migration implementation

The local pre-migration async code still exposes a `Vec<u8>` request, allocation helpers named `copy_exact` and `copy_into`, slice accessors named `source_bytes()` and `destination_bytes()`, and a public borrowed-destination `memmove_into` helper that copies the successful prefix back into caller memory. Those names describe the old surface and must not survive as public post-migration APIs.

The existing diagnostics are valuable and must survive the migration: explicit owner shutdown is distinct from worker/channel compatibility failures, and wrapped `MemmoveError` failures remain visible as validation or hardware failures rather than being collapsed into generic async errors.

## Public API shape

### Request

The public request type owns exactly two buffers:

- `source: Bytes`
- `destination: BytesMut`

The public constructor is:

```rust
AsyncMemmoveRequest::new(source: Bytes, destination: BytesMut)
```

This constructor validates the request before the operation can be submitted. It must not allocate a destination, infer a destination length, or hide the source/destination ordering behind another convenience method. The request remains spawn-friendly because both buffers are owned and can cross async task boundaries without borrowing caller stack frames.

Do not introduce a generic public `Buf` / `BufMut` request type in M006. Generic buffer abstractions would widen the contract before the unsafe and completion invariants are reviewed. Keep the first contract concrete: `Bytes` for source and `BytesMut` for destination.

### Result

The public result type is explicit:

```rust
AsyncMemmoveResult {
    destination: BytesMut,
    report: MemmoveValidationReport,
}
```

The result returns the full caller-supplied destination, not just a copied prefix. `report` is the successful validation report from the memmove path. There is no `result.bytes` shortcut and no public post-migration `destination_bytes()` view method.

### Ownership and cost profile

Each operation owns one `Bytes` source and one `BytesMut` destination. The library performs request validation and inline hardware submission; it does not allocate a destination on behalf of the caller, does not provide a CPU copy-back helper, does not silently coalesce operations, and does not require software aggregation. This keeps per-operation ownership and copy costs visible to the caller.

## Validation semantics

Validation is performed before enqueue or inline hardware submission. A malformed request must fail without touching the work queue.

Normative rules:

- A zero-length source is invalid and maps through the existing `MemmoveError::InvalidLength` validation path.
- `destination.len()` must be at least `source.len()` when using initialized destination bytes. A shorter destination maps through `MemmoveError::DestinationTooSmall`.
- The requested transfer size is exactly `source.len()`.
- The destination may be larger than the source. On success, only the `source.len()` prefix is guaranteed to have been overwritten by DSA. The oversized tail remains part of the returned `BytesMut` and must be preserved.
- Validation errors stay wrapped as `AsyncMemmoveError::Memmove(...)` or the migration's equivalent wrapped `MemmoveError` path. Do not relabel validation failures as lifecycle or worker/channel failures.

## Destination initialization and spare-capacity rules

`BytesMut` has an initialized length and may also have spare capacity. Only bytes inside the initialized length are readable. Spare capacity is not readable until it has been initialized and the visible length has been advanced.

For M006's first public contract, the safe path should submit only into the initialized destination region covered by `destination.len()`. If S02 needs to write into DSA-visible spare capacity, that path must be hidden behind a narrow unsafe-backed helper rather than exposed as a broad public API.

Any future spare-capacity DSA helper must document and enforce these invariants:

- Pointer validity: the source and destination pointers remain valid for the full DSA operation and any retry descriptors.
- Aliasing: the destination mutable region is not aliased by another readable or writable reference while hardware may write it.
- Initialization: bytes written by DSA are considered initialized only after successful completion is observed.
- Length advancement: `BytesMut` length is advanced only after successful completion for the exact initialized range.
- Cancellation safety: dropping or aborting the awaiting future cannot expose uninitialized bytes and cannot free memory still reachable by hardware.
- Completion safety: error, timeout, retry-exhaustion, and malformed-completion paths must not advance length or return a destination that claims uninitialized bytes are readable.

Until those invariants are implemented and reviewed, the public API remains `AsyncMemmoveRequest::new(source: Bytes, destination: BytesMut)` over an initialized destination length.

## First-version hardware submission

M006 accepts inline `enqcmd` submission as the first implementation strategy. The executor may submit the descriptor from the calling async context or a blocking boundary chosen by the implementation, but the contract does not require a dedicated software aggregator.

The `enqcmd` model allows multiple submitting threads. Therefore, M006 should not preserve the old single-worker serialization merely to protect the submission path. If compatibility code temporarily keeps worker/channel failure classes, those classes must remain separately named; they should not be confused with the target inline submission design.

The inline v1 path must keep submission failures observable. If `enqcmd` submission fails, classify it as an inline submission failure with its own diagnostic name rather than as owner shutdown, worker panic, or validation failure.

## Rejected stale surfaces

The following names are prohibited as public post-migration surfaces:

- `copy_exact`
- `copy_into`
- `with_destination_len`
- source-only constructors that allocate destination storage internally
- public `memmove_into`
- `result.bytes`
- `source_bytes()`
- `destination_bytes()`

These names either allocate destination storage inside the library, hide destination ownership, expose copy-back semantics, or preserve the old Vec-based accessor model. Tests should reject them at compile time or through source-level contract checks.

## Async failure taxonomy

The migration must preserve existing failure observability and add inline submission naming where applicable.

Required separately named classes:

1. Owner lifecycle shutdown: `owner_shutdown` or an equivalent lifecycle kind for use-after-owner-shutdown and post-shutdown submissions.
2. Inline submission failures: a distinct inline `enqcmd` submission failure class for first-version hardware submission faults.
3. Worker/channel compatibility failures where still present: `worker_init_closed`, `request_channel_closed`, `response_channel_closed`, `worker_panicked`, or equivalent names if transitional compatibility code remains.
4. Wrapped `MemmoveError` validation and hardware failures: `invalid_length`, `destination_too_small`, `queue_open`, `completion_timeout`, `malformed_completion`, `page_fault_retry_exhausted`, `completion_status`, and `byte_mismatch` remain visible as validation or hardware memmove failures.

Do not collapse these into a single async failure. A caller should be able to distinguish lifecycle shutdown, inline submission failure, compatibility worker/channel failure, and wrapped memmove validation/hardware failure from the error kind alone.

## S02 test matrix

S02 should update `idxd-rust/tests/async_memmove_contract.rs` to prove the new bytes API contract with host-free workers or compile-time checks where hardware is unavailable.

Required tests:

- `AsyncMemmoveRequest::new(source: Bytes, destination: BytesMut)` is the canonical construction path.
- A successful async operation returns `AsyncMemmoveResult { destination, report }` with the full `BytesMut` destination.
- Zero-length source fails before enqueue as wrapped `MemmoveError::InvalidLength`.
- Destination too small fails before enqueue as wrapped `MemmoveError::DestinationTooSmall`.
- Oversized destination tail is preserved after the copied prefix.
- Validation-before-enqueue is proved by a fake worker call count of zero for malformed requests.
- Hardware `MemmoveError` propagation preserves the underlying kind and phase.
- Cancellation after submission does not expose uninitialized bytes or poison later submissions.
- FIFO/ordering behavior is specified for any transitional worker path; if inline submission replaces the worker path, tests should instead prove the chosen ordering contract and must not assume hidden software aggregation.
- Owner shutdown remains a lifecycle failure distinct from worker/channel and memmove errors.
- Inline submission failure classification is separately named.
- Compatibility worker/channel failure classification remains separately named where transitional worker code still exists.
- Stale public surfaces are rejected: `copy_exact`, `copy_into`, `with_destination_len`, source-only constructors, public `memmove_into`, `result.bytes`, `source_bytes()`, and `destination_bytes()`.

S02 should update `idxd-rust/tests/tokio_handle_contract.rs` to remove assumptions that are only true because of the old single-worker surface, while preserving Tokio composition checks that remain true for owned requests. If the implementation uses inline `enqcmd`, the tests must allow multiple submitting tasks and must not require software serialization as a correctness property.

## S03 downstream migration notes

Downstream callers must construct a `Bytes` source and provide an initialized `BytesMut` destination. They must read copied bytes from the returned `destination` prefix and retain `report` for validation metadata. Downstream code must not call public allocation helpers or borrowed copy-back helpers because those helpers should no longer exist.

If a downstream path previously depended on a source-only request, allocate or reuse the destination at the call site. If it previously depended on `memmove_into`, rewrite it to submit the owned destination and then consume the returned `BytesMut` explicitly.

## Future borrowed zero-copy placeholder

A future borrowed or spare-capacity zero-copy API may be useful, but it is not part of the M006 public surface. That future API must be designed separately with unsafe invariants reviewed up front. In particular, it must specify how borrowed memory remains pinned or otherwise valid for hardware, how cancellation is handled, how completion initializes bytes, and how error paths prevent reads of uninitialized memory.

## Future software aggregation, batching, and movdir64 support

Software aggregation, batching, and `movdir64` support are future work. They may become useful for throughput tuning or for alternate submission paths, but they are not required for M006. The M006 contract explicitly rejects treating software aggregation, batching, or `movdir64` support as prerequisites for the bytes API migration.

If a future milestone adds aggregation, it must be observable as an implementation strategy rather than silently changing the API contract. It should document queueing, coalescing, backpressure, fairness, failure classification, and how individual `MemmoveValidationReport` values map back to submitted requests.
