# M011 S05 generic IDXD elegance and duplication audit

## Purpose and R024 claim boundary

Reader: a future IDXD maintainer or milestone validator landing cold on M011.
Post-read action: decide whether the generic IDXD architecture is safe to hand off, and know which follow-up work is deliberately outside M011.

This report closes **R024** for **M011/S05** by auditing whether the delivered generic IDXD seam is clean, lean, elegant, and free of avoidable duplicate lifecycle code. It is an architecture and documentation guard surface, not another refactor and not a new hardware run.

Claim boundary:

- This report can claim that the implemented `IdxdSession<Accel>` boundary, crate-private `run_blocking_operation` lifecycle, DSA/IAX operation adapters, and `WqPortal::submit_desc64` WQ-mode boundary are handoff-worthy for the representative M011 scope.
- This report can claim that S03 and S04 evidence were consumed as tracked inputs.
- This report cannot claim new hardware success, new benchmark numbers, full accelerator coverage, production scheduling, or RPC integration.

## Handoff verdict

**Handoff verdict: pass, with named first-version compromises.**

M011 is handoff-worthy because the core architecture has one generic session owner, one shared blocking submit/complete lifecycle, operation-local descriptor/completion state, and one portal-level WQ-mode branch. The design is narrow rather than generic for its own sake: `IdxdSession<Dsa>` and `IdxdSession<Iax>` expose only representative operations that are actually implemented, while `Iaa` remains the compatibility spelling for the IAX/IAA family.

The verdict is not a claim that every surrounding proof artifact is perfectly factored. The proof and benchmark CLIs share visible wrapper/helper patterns, and the IAX crc64 path carries a deliberately broad diagnostic surface for a first representative operation. Those flaws are non-blocking because they do not duplicate the core DSA/IAX submit/complete lifecycle and do not define the public API.

## Core architecture audit

Core verdict: clean, lean, and elegant for the M011 representative scope.

- `IdxdSession<Accel>` is the single generic session seam. It owns one `WqPortal`, stores an `IdxdSessionConfig<Accel>`, reports queue metadata, and keeps the accelerator family selected by static marker type rather than runtime dispatch.
- `Dsa`, `Iax`, and `Iaa` keep the public family names explicit. `Dsa` and `Iax` are sealed marker families; `Iaa` is an alias to the implemented IAX/IAA representative path instead of a separate half-supported implementation.
- `IdxdSession<Dsa>::memmove` consumes an already-open portal and delegates to the DSA operation helper. It rebuilds a `DsaConfig` per call as a compatibility wrapper, then uses `run_direct_memmove` and post-copy validation instead of inlining submission or completion polling.
- `IdxdSession<Iax>::crc64` consumes the same generic session shape and delegates to `run_iax_crc64`. The IAX path uses borrowed source bytes, operation-owned descriptor/completion storage, and typed CRC metadata.
- `run_blocking_operation` is crate-private static dispatch. The `BlockingOperation` trait and `BlockingOperationDecision` enum are not public API; they express the internal reset/fill, submit, observe, classify, retry, and return loop once.
- The DSA adapter (`DirectMemmoveOperation`) and IAX adapter (`IaxCrc64State`) implement `BlockingOperation`. Each adapter owns descriptor/completion state and operation-specific classification; neither reimplements the lifecycle loop.
- `WqPortal::submit_desc64` is the only generic 64-byte descriptor WQ-mode branch for the blocking path. Typed DSA and IAX portal helpers forward to it, so operation modules do not branch on dedicated vs shared WQ mode.

This is the right amount of abstraction for M011. The architecture avoids a public operation hierarchy, avoids `dyn` dispatch, avoids a runtime accelerator dispatcher, and keeps unsafe descriptor submission contracts close to the portal and operation-owned state that can justify them.

## Duplication audit

Core lifecycle duplication verdict: no blocking DSA/IAX submit/complete lifecycle duplication found.

The DSA path and the IAX/IAA path share the same internal lifecycle shape through `run_blocking_operation`. DSA supplies `DirectMemmoveOperation`; IAX supplies `IaxCrc64State`; both let the lifecycle own only control flow while operation-specific code owns descriptor fill, completion interpretation, retry side effects, and typed result construction. `WqPortal::submit_desc64` owns the WQ-mode choice below both adapters.

Proof-layer duplication verdict: acceptable, visible, and bounded.

- `live_idxd_op` consumes the API by opening `IdxdSession::<Dsa>` and `IdxdSession::<Iax>` and calling `session.memmove` / `session.crc64`. It does not define the API, open raw portals, call `DsaSession`, or submit descriptors directly.
- `idxd_representative_bench` also consumes the API by opening `IdxdSession::<Dsa>` and `IdxdSession::<Iax>` and timing calls through `session.memmove` / `session.crc64`. It does not define the API and does not bypass the generic session boundary.
- `verify_idxd_representative_ops.sh` and `verify_idxd_representative_bench.sh` duplicate wrapper patterns around launcher preflight, artifact/stdout/stderr capture, JSON validation, no-payload field rejection, and final `verdict=` lines.

That proof-layer duplication is acceptable in M011 because it keeps S03 hardware proof and S04 measured-number proof independently auditable. It should be extracted only when a **third** proof or verifier grows the same launcher/artifact/no-payload helper shape. At that point, the extraction target should be a small shared verifier utility, not a broader benchmark framework and not a new runtime API.

## Simplicity and scope audit

M011 stayed lean by not expanding from a representative seam into a framework. The following are explicit non-goals and non-deliverables:

- no full DSA surface,
- no full IAX surface,
- no full IAA surface,
- no AECS-heavy flows,
- no pooling/registry,
- no scheduler,
- no worker batching,
- no MOVDIR64/MOVDIR64B strategy framework,
- no benchmark matrix/framework,
- no RPC/Tonic integration,
- no broad operation hierarchy,
- no public operation trait, and
- no runtime accelerator dispatcher.

This boundary is part of the elegance verdict. Adding any of those features inside M011 would have made the seam heavier without improving the representative proof that this milestone needed.

## Diagnostics and no-payload audit

The diagnostics boundary is preserved. S03/S04 reports and CLI artifacts may expose paths, operation names, family names, WQ mode, requested byte counts, status values, retry counts, timing aggregates, CRC scalar values, artifact paths, stdout/stderr paths, and launcher status. They must not expose raw payload material or sensitive operator values.

Concrete forbidden examples for S05 guards and future reviewers:

- raw buffer bytes,
- source bytes,
- destination bytes,
- payload dumps,
- descriptors,
- tokens, and
- secrets.

The source supports this boundary. DSA post-copy verification reports mismatch metadata without dumping buffers. IAX crc64 diagnostics preserve requested length, status, error code, invalid flags, fault address, retry count, and CRC scalar metadata without source payload dumps. The S03/S04 verifier scripts reject payload-shaped JSON fields before reports can consume artifacts.

## Known compromises

These are known first-version compromises, not hidden blockers:

- `IdxdSession<Dsa>::memmove` rebuilds a `DsaConfig` per call to reuse the mature validation path. The wrapper cost is acceptable for the representative proof but should be revisited if the generic DSA path becomes a hot public API.
- The IAX crc64 proof uses a deterministic input whose T10DIF CRC field can verify to the zero-residue scalar reported by hardware. That is sufficient for a representative hardware completion proof, but it is not broad IAX/IAA analytics coverage.
- The IAX diagnostic surface is intentionally large for a first path: timeout, malformed completion, page-fault retry exhaustion, terminal status, error code, invalid flags, fault address, and retry metadata are all visible. This is better than hiding failure modes, but future operations should watch for repeated boilerplate.
- `live_idxd_op`, `idxd_representative_bench`, `verify_idxd_representative_ops.sh`, and `verify_idxd_representative_bench.sh` duplicate proof-layer CLI/verifier helper shapes. This is non-core duplication and should be extracted only after a third peer makes the duplication stable enough to abstract.
- S04 measured numbers are intentionally small and release-profile. They prove that representative rows can produce positive metrics through the generic seam; they do not characterize performance.

## Deferred work handoff

Deferred work should be filed as future milestone scope, not treated as unfinished M011 work:

1. Expand DSA operations only when there is a concrete consumer and hardware proof for each operation.
2. Expand IAX/IAA beyond crc64 only when AECS-heavy flows and their diagnostics can be designed deliberately.
3. Extract proof/verifier helpers if a third representative proof repeats the same launcher, artifact, and no-payload validation code.
4. Consider amortizing or session-scoping the DSA config wrapper only if measurements show the per-call rebuild matters.
5. Design pooling/registry, scheduler, worker batching, MOVDIR64/MOVDIR64B strategy, benchmark matrix/framework, and RPC/Tonic integration as separate milestones with their own proof bars.
