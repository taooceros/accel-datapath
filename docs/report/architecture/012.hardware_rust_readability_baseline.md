## Purpose

This report is the durable readability baseline for M009/S01. It freezes the shape of the responsibility map and verification contracts before any code-moving or readability refactor begins.

S01 changes documentation and verification baselines only. It does not change runtime behavior, public APIs, DSA/IAX submission semantics, artifact schemas, or prepared-host claim criteria. Downstream slices may improve readability only when their changes preserve the contracts named here or update a deliberately versioned replacement contract with matching tests and verifiers.

The baseline is contract-first: if a future refactor makes a module easier to read but weakens error classification, request recovery, stdout/artifact equality, JSON field stability, ordinary-host expected-failure behavior, or the no-payload diagnostic rule, that refactor is outside the M009 cleanup contract.

### Ordinary-host and prepared-host direct async claims

Ordinary-host checks may claim that the direct async code preserves public API shape, request validation, host-free fake backend behavior, lifecycle classification, bounded retry/backpressure metadata, completion snapshot propagation, no-payload diagnostics, and documented contract wording. Ordinary-host checks must not claim prepared-host hardware success, real ENQCMD throughput, real DSA/IAX completion success, or performance behavior.

Prepared-host success remains limited to launch-wrapped hardware verifier scripts and real work-queue runs. Host-free tests can keep S02 honest before refactoring, but any statement that direct async submission completed on DSA/IAX hardware requires prepared-host evidence.

### Tokio benchmark public artifact freeze

S03 must treat the benchmark artifact as preserve-by-default. The top-level schema fields `schema_version`, `backend`, `suite`, `claim_eligible`, `verdict`, `ok`, `failure_class`, `error_kind`, `direct_failure_kind`, `validation_phase`, `validation_error_kind`, direct retry metadata, completion metadata, and `results` must not be renamed or removed without an explicit versioned replacement contract.

Each result row preserves `mode`, `target`, `comparison_target`, request knobs, operation counts, elapsed/latency/rate metrics, row verdict, row failure classifications, direct/validation metadata, completion metadata, and row `claim_eligible`. Passing rows require completed operations and no failure metadata. Failing rows require non-empty failure classification and error kind.

Stdout/artifact equality is part of the public contract: when `--artifact` is supplied, the file content must match stdout exactly after trimming verifier-captured newlines. Verifier consumers should parse `verify_tokio_memmove_bench.sh` final `phase=done` lines and artifact fields instead of inferring validity from filenames or process exit alone.

### Software diagnostic and prepared-host benchmark claims

Software diagnostic mode is useful ordinary-host evidence that the Tokio benchmark CLI, async row generation, schema rendering, artifact writing, and verifier validation still work. It is never a hardware-performance or DSA/IAX success claim, so software top-level and row `claim_eligible` must remain `false`.

Prepared-host hardware claims require launch-wrapped verifier evidence, a successful hardware artifact, `claim_eligible=true`, the expected `direct_async` rows for the selected suite, and a `direct_sync` comparison row for canonical or latency suites. Missing launcher prerequisites and classified queue-open/runtime failures remain `expected_failure` outcomes, not broken contracts, while malformed artifact/schema/parity results are contract blockers.

### Tokio benchmark negative and redaction baseline

The host-free negative baseline names invalid numeric CLI inputs, invalid enum values, empty device paths, missing artifact values, directory artifact paths, missing or contradictory verifier overrides, malformed JSON, missing required schema fields, stdout/artifact mismatches, software claim-eligibility contradictions, missing hardware sync comparison rows, and forbidden payload dump fields.

No-payload for benchmark artifacts and verifier diagnostics means they may include metadata such as device paths, requested byte counts, iterations, concurrency, duration, backend, suite, mode, target, operation counts, rates, failure class, error kind, validation phase, direct failure kind, completion status, completion result, bytes completed, fault address, stdout/stderr paths, and launcher status. They must not include copied source bytes, destination bytes, raw payload dumps, or fields intended to serialize payload content.
