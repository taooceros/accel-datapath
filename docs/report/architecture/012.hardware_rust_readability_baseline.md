# Hardware Rust readability baseline

**Milestone:** M009 — Hardware Rust readability cleanup  
**Slice:** S01 — Inventory post-M008 seams and contract baselines  
**Status:** Initial T01 scaffold; later S01 tasks must fill the inventories and preserve this contract shape  
**Reader:** A maintainer or future agent preparing an M009 readability refactor in the hardware Rust stack.  
**Post-read action:** Identify which `idxd-rust`, `idxd-sys`, and `hw-eval` responsibilities may move in S02-S06, which public/API/artifact contracts protect those moves, and which ordinary-host checks must still pass before a refactor proceeds.

## Purpose

This report is the durable readability baseline for M009/S01. It freezes the shape of the responsibility map and verification contracts before any code-moving or readability refactor begins.

S01 changes documentation and verification baselines only. It does not change runtime behavior, public APIs, DSA/IAX submission semantics, artifact schemas, or prepared-host claim criteria. Downstream slices may improve readability only when their changes preserve the contracts named here or update a deliberately versioned replacement contract with matching tests and verifiers.

The baseline is contract-first: if a future refactor makes a module easier to read but weakens error classification, request recovery, stdout/artifact equality, JSON field stability, ordinary-host expected-failure behavior, or the no-payload diagnostic rule, that refactor is outside the M009 cleanup contract.

## Source inputs

This scaffold is grounded in the post-M008 hardware cleanup reports, the `idxd-rust` operational README, the repository report-writing conventions, and the active requirement set for M009. Later S01 tasks should fill the detailed rows from tracked source and tracked reports only.

Required inputs for the completed S01 baseline:

- M008 migration contract and cleanup-convention reports.
- `idxd-rust` README and verifier scripts.
- `idxd-rust` library, proof-binary, and contract-test surfaces.
- `idxd-sys` raw ABI/MMIO boundary and layout tests.
- `hw-eval` benchmark harness, report schema, plotting consumers, and README behavior.
- R018 readability-maintainability requirement context.

Do not use `.gsd/` artifacts, ignored `target/` output, copied payload buffers, or prepared-host-only evidence as fixtures for this host-free baseline.

## Responsibility map

This section will become the maintainer-readable map of what each hardware Rust area owns after M008 and before M009 refactors begin.

| Area | Current responsibility | Candidate readability pressure | Contract that must stay stable | S02-S06 owner |
|---|---|---|---|---|
| `idxd-rust` public validation/session surface | Safe sync and async DSA memmove APIs, validation reports, typed failure classification, request recovery, and proof binaries. | Separate public API contracts from direct-runtime internals and proof-binary plumbing without changing caller-visible behavior. | Public constructors/accessors, error classifications, request recovery, proof artifact fields, stdout/artifact equality, and no-payload diagnostics. | S02, S03, S04 |
| `idxd-rust` direct async runtime | Direct ENQCMD-oriented async submission, completion monitoring, retry/backpressure metadata, and operation ownership. | Make runtime ownership and completion-state transitions easier to follow without redesigning submission semantics. | Accepted-operation ownership, completion snapshot metadata, direct failure kinds, bounded retry/backpressure behavior, and destination-length safety. | S03, S04 |
| `idxd-sys` raw boundary | Bindgen-backed DSA/IAX descriptors, completion records, portal mapping, raw submission helpers, topology helpers, and OS error surfaces. | Clarify raw-boundary comments and module organization without wrapping low-level hardware signals in higher-level abstractions. | Raw ABI layout, unsafe preconditions, OS `io::Error` sources, status bytes, and transparent descriptor/completion behavior. | S05 |
| `hw-eval` benchmark harness | CLI-driven DSA/IAX/software benchmark execution, JSON report emission, plotting inputs, and operational diagnostics. | Separate normalized config/reporting concerns from hot-loop benchmark code while preserving artifact shape. | CLI flags, report fields, benchmark names, graphing assumptions, non-fatal pin behavior, prepared-host launch expectations, and no-payload diagnostics. | S06 |
| Downstream proof consumers | Repo-local verifier scripts, contract tests, plotting scripts, and downstream async-handle proof paths that consume hardware Rust APIs or artifacts. | Keep compatibility checks visible before moving symbols or reshaping modules. | Required fields, final-line diagnostics, expected-failure classification, claim eligibility, and forbidden payload-content checks. | S02-S06 |

## Downstream slice contract baselines

Each downstream M009 slice must preserve the baseline below before declaring a readability refactor complete.

| Slice | Baseline obligation | Refactor is allowed only if |
|---|---|---|
| S02 | Public `idxd-rust` API and contract-test baseline. | The public sync/async construction, validation report, accessor, and proof-binary contracts remain intentionally stable or are updated with explicit compatibility tests. |
| S03 | `idxd-rust` direct async readability baseline. | Completion ownership, request recovery, backpressure/retry metadata, and direct failure classification stay observable through host-free tests and verifier artifacts. |
| S04 | `idxd-rust` proof/verifier/artifact readability baseline. | Sync, async, and benchmark proof artifacts keep required fields, stdout/artifact equality, expected-failure classification, and no-payload redaction. |
| S05 | `idxd-sys` raw-boundary readability baseline. | Raw descriptor/completion layout, unsafe submission preconditions, status bytes, and OS error propagation stay transparent and testable without hardware. |
| S06 | `hw-eval` readability baseline. | CLI behavior, benchmark report schema, plotting consumers, software-only ordinary-host checks, and prepared-host claim boundaries remain stable. |

The S02-S06 references above are deliberately mechanical. The guard script rejects this report if any downstream slice reference disappears, because later agents need a single checklist before touching code.

## Non-change boundaries

M009 readability cleanup must not accidentally become a behavior rewrite. Unless a later slice explicitly versions the affected contract and updates every producer, verifier, and consumer together, these boundaries are non-changeable:

- No DSA/IAX hardware semantics rewrite, batching-policy rewrite, MOVDIR64/MOVDIR64B policy change, or fallback-to-software behavior change.
- No broad `idxd-sys` domain-error layer or builder layer that hides raw ABI/MMIO facts.
- No public async request allocation convenience API that weakens explicit source/destination ownership or recovery semantics.
- No proof-binary or benchmark JSON field rename/removal without an explicit replacement contract.
- No graphing/report schema drift in `hw-eval` without updating plotting consumers and tracked samples together.
- No use of ignored `.gsd/` or generated `target/` artifacts as verification fixtures.
- No copied source payload bytes, destination payload bytes, buffer dumps, or payload-content examples in diagnostics, reports, logs, manifests, or test fixtures.

S01 itself changes documentation and mechanical verification baselines only. Runtime behavior is intentionally unchanged.

## Verification matrix

The completed S01 baseline should map every readability move to a host-free proof first, then name prepared-host proof only for real hardware claims.

| Surface | Ordinary-host baseline | Prepared-host limit |
|---|---|---|
| `idxd-rust` public contracts | Contract tests and verifier artifact validation can prove field stability, classification stability, expected failures, and redaction without DSA/IAX hardware. | Hardware success and performance claims still require launch-wrapped prepared-host runs. |
| `idxd-rust` direct async runtime | Host-free tests can prove lifecycle, request recovery, backpressure, retry metadata, completion snapshot propagation, and no-payload formatting. | Real ENQCMD completion success requires a prepared device and launcher. |
| `idxd-sys` raw boundary | Layout tests and compile checks can preserve ABI shapes, raw helper behavior, and OS error propagation. | Real portal open/submission success requires prepared work queues and privileges. |
| `hw-eval` benchmark reports | Software-only or static report checks can preserve schema, plotting compatibility, and diagnostic classification. | DSA/IAX throughput, latency, NUMA, and accelerator-performance conclusions require prepared-host evidence. |
| Documentation guard | `bash idxd-rust/scripts/check_m009_s01_readability_baseline.sh` fails fast on missing sections, missing S02-S06 references, missing R018 coverage, and missing no-payload language. | Not applicable; the documentation guard is intentionally host-free. |

## R018 coverage

R018 is advanced by making the hardware Rust readability cleanup falsifiable before code motion begins. This scaffold gives maintainers a single baseline that names responsibilities, non-change boundaries, downstream slice obligations, host-free checks, and prepared-host limits.

Later S01 tasks must complete R018 coverage by filling concrete hotspots and contract references for `idxd-rust`, `idxd-sys`, and `hw-eval`. Downstream S02-S06 tasks then preserve or explicitly update those references while making the code easier to read.

## Ordinary-host and prepared-host limits

Ordinary-host verification can prove documentation completeness, compile-time API compatibility, host-free contract tests, verifier behavior, malformed-artifact rejection, expected-failure classification, schema stability, and no-payload redaction. It cannot prove that DSA/IAX hardware completed work successfully.

Prepared-host verification is required for any claim that a hardware-backed memmove, direct async submission, IAX/DSA benchmark path, or performance result succeeded. Prepared-host evidence must still preserve the same artifact contracts and no-payload constraints used on ordinary hosts.

No-payload means diagnostics may include metadata such as paths, requested byte counts, retry counts, phases, status bytes, completion results, fault addresses, benchmark names, concurrency, accelerator type, OS errors, and claim eligibility. Diagnostics must not include copied source bytes, destination bytes, raw payload dumps, or fields intended to serialize payload content.

## Reader-test result

A cold maintainer can use this scaffold to see that S01 is documentation-only, that S02-S06 each have named contract baselines, that `idxd-rust`, `idxd-sys`, and `hw-eval` have distinct responsibilities, and that ordinary-host checks protect readability work without pretending to prove prepared-host hardware success.
