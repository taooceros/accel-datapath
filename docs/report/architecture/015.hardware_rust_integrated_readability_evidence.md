**Milestone:** M009 — Hardware Rust readability cleanup

## Purpose

This report advances **R018** by reconciling the S01 baseline sketch with the S02-S05 changes that actually shipped. It is not a refactor plan. It is the final responsibility/readability map for the current `idxd-rust`, `hw-eval`, and `idxd-sys` hardware Rust path.

The scope is deliberately narrow: M009 improved readability and navigation while preserving behavior. Public import surfaces, CLI/report schemas, verifier classifications, host-free contracts, raw unsafe semantics, and no-payload guarantees are preserved. **Prepared-host hardware success is not claimed** by this report or by ordinary-host guard checks. In guard-script wording: prepared-host hardware success is not claimed.

## Claim boundaries

M009 can now claim that the hardware Rust path is easier to navigate by responsibility:

- direct async ownership is split between public session wiring, top-level direct runtime phases, operation-owned lifecycle, monitor polling, and scripted test support;
- `tokio_memmove_bench` ownership is split between private binary modules for CLI/config, artifacts, failure metadata, software diagnostics, hardware claim gating, mode loops, and backend dispatch;
- `hw-eval` ownership is split between config, report/statistics, binary-local methodology modules, orchestration, and primitive helper modules;
- `idxd-sys` ownership is split behind a lean public facade into private raw-boundary modules for descriptors, portal submission, completion helpers, timing, topology, and cache utilities.

M009 does **not** claim:

- prepared-host hardware success from this report;
- DSA/IAX throughput, latency, or performance behavior from ordinary-host tests;
- a new public API, schema version, or benchmark artifact format;
- a higher-level abstraction that hides raw unsafe semantics;
- permission to include payload bytes, copied source bytes, destination bytes, raw payload dumps, or payload-content examples in diagnostics, reports, tests, or artifacts.

No-payload diagnostics may still include metadata such as device paths, request lengths, iterations, concurrency, benchmark names, operation counts, latency/rate fields, retry counts, status bytes, completion result bytes, fault addresses, OS error kinds, CPU/core ids, NUMA nodes, and launcher/preflight status.

## Intentional non-changes and descoped changes

These boundaries are intentional. Future edits should treat them as contract text, not omissions:

- The `tokio_memmove_bench` split remains private to the benchmark binary. No new library API is introduced for CLI, artifacts, runners, mode loops, software diagnostics, hardware claim gating, or failure metadata.
- `idxd-sys` keeps root-level re-exports and the `idxd` compatibility alias. The private-module split is an implementation readability improvement, not a public import-surface change.
- **Prepared-host hardware success is not claimed** by S06 documentation, ordinary-host guard scripts, fake-binary verifier tests, software diagnostic artifacts, or host-free Cargo tests.
- Public APIs, public import surfaces, CLI flags, JSON report schemas, benchmark artifact fields, verifier final-line classifications, and plotting-facing row names do not churn under M009.
- The no-payload rule remains active across docs, tests, shell guards, CLI diagnostics, verifier diagnostics, JSON/text reports, and artifacts.
- `idxd-sys` does not gain a broad builder, domain-error layer, or vague safe wrapper that hides ABI layout, packed-field access, volatile reads, raw assembly, mmap/munmap lifetime, or raw accepted/rejected ENQCMD semantics.
- `hw-eval` methodology modules own benchmark policy, while `dsa.rs`, `iax.rs`, `submit.rs`, and `sw.rs` remain primitive helper surfaces. That split is deliberate so hot-loop behavior and raw helper semantics stay easy to audit separately.
- Direct async code does not add a blocking fallback, public borrowed copy-back helper, public destination allocation convenience API, preallocated registry claim, or Tonic prerequisite under this readability milestone.
- Software diagnostic success and ordinary-host JSON success remain useful contract evidence only. They do not become DSA/IAX hardware performance evidence.

## Requirement mapping

**R018** is advanced by making the shipped hardware Rust structure navigable from a single document and tying each owner to concrete host-free proof:

- S02 provides the direct async owner map and guard.
- S03 provides the Tokio benchmark/artifact/verifier owner map and guard.
- S04 provides the `hw-eval` config/report/methodology/helper split and CLI/report contract tests.
- S05 provides the `idxd-sys` raw facade/private-module split and raw-boundary contract tests.
- S06 integrates those outputs and defines the final guard target for stale claims, missing owner modules, missing evidence references, claim overreach, and no-payload drift.

The evidence is concrete because every owner family maps to an executable check. The evidence is bounded because none of those checks claims prepared-host hardware success.

## Reader-test result

A cold maintainer should now be able to start at the public hardware Rust behavior and choose the right owner without opening every file:

- public async calls and request recovery start in `async_session.rs`;
- direct ENQCMD submission, pending operation lifetime, monitor polling, and fake backend proof live under `async_direct` and `direct_memmove`;
- Tokio benchmark CLI/artifact/software/hardware/mode/failure/runner responsibilities live under the private `tokio_memmove_bench` module tree;
- `hw-eval` benchmark policy lives in `methodology`, while config/report/orchestration and primitive helpers stay separate;
- raw descriptor, portal, completion, timing, topology, and cache semantics live in `idxd-sys` private modules behind the compatible root facade.

The maintainer can then run the verification matrix above and state only the corresponding bounded claim. If they need real DSA/IAX success or performance evidence, they must use the prepared-host launcher/verifier path rather than this host-free integrated evidence report.
