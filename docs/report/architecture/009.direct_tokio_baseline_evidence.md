## Status and scope

This report is the reviewer-facing evidence index for the M007 direct Tokio IDXD baseline. It explains how the standalone Tokio memmove benchmark, verifier, and collection workflow separate four evidence classes:

1. host-free tests and binary checks,
2. software direct-async diagnostics,
3. ordinary-host unsupported-hardware classifications, and
4. prepared-host hardware claim evidence.

The report is intentionally conservative. It does not claim prepared-host performance evidence from the current final validation run because the hardware verifier ended with `verdict=expected_failure failure_phase=preflight launcher_status=missing_launcher`, not `verdict=pass claim_eligible=true`. It also does not treat software diagnostics as hardware evidence, even when those diagnostics pass.

## Direct async versus direct sync interpretation

The S03 verifier is the authority for benchmark artifact validity. For prepared hardware evidence, it must validate both the direct Tokio async baseline and the synchronous comparison surface in the same accepted artifact set:

| Label | Meaning | Claim rule |
|---|---|---|
| `direct_async` | Direct Tokio async IDXD memmove benchmark rows for single latency, concurrent submissions, and fixed-duration throughput. | Necessary but not sufficient for a hardware claim. |
| `direct_sync` | Direct synchronous `DsaSession::memmove` comparison row paired with the async single-latency row. | Required for prepared-host comparison claims. |
| `software_direct_async_diagnostic` | Software backend diagnostic rows that exercise the CLI, runtime, JSON, and verifier plumbing. | Always `claim_eligible=false`; never hardware acceleration evidence. |
| `expected_failure` | A classified unsupported-host or typed runtime failure from the verifier. | Truthful diagnostic evidence only; not a pass, not a speedup, and not a prepared-host claim. |
| `verdict=pass claim_eligible=true` | Verifier accepted a hardware artifact with the required direct async rows and direct sync comparison. | The only prepared-host claim path for S04. |

If hardware output is missing, empty, malformed, contradictory, lacks the `direct_sync` comparison, lacks `direct_async` rows, or exits nonzero, the report must not turn it into claim evidence. If the verifier exits zero with `verdict=expected_failure`, the report may cite it only as an unsupported-host or runtime-failure classification.

## Ordinary-host rules and non-claims

Acceptable ordinary-host evidence has strict limits:

- The collection workflow may pass when the hardware verifier exits `0` with a classified `verdict=expected_failure`.
- The unsupported-host classification must name a phase such as `failure_phase=preflight` and a concrete status such as `launcher_status=missing_launcher`, `missing_work_queue`, `missing_devenv`, or `missing_capability`.
- Such a row proves that the verifier classified the host truthfully; it does not prove hardware performance.
- Software diagnostics must remain `claim_eligible=false`, even when `verdict=pass`.
- Prepared-host claims require `verdict=pass claim_eligible=true` plus both `direct_async` and `direct_sync` labels in verifier-accepted hardware evidence.

This report does not claim throughput, latency, speedup, fairness, queue-depth behavior, Tonic behavior, MOVDIR64 behavior, worker-owned batching, or payload contents.
