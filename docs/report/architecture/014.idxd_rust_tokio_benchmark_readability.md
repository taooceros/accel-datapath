## Purpose

This report advances **R018** for S03 by turning the S01 preserve-by-default baseline into a concrete responsibility map for `tokio_memmove_bench`. It also consumes the S02 direct async module split: the benchmark may call `AsyncDsaSession`, `AsyncDsaHandle`, `AsyncMemmoveRequest`, and the existing direct failure accessors from the reorganized direct async runtime, but S03 must not alter those runtime contracts.

S03 is readability work. It may name phases, split internal benchmark helpers, and move artifact/verifier plumbing into better-labeled owners. It must not change the public async runtime, direct failure metadata, artifact schema version, serialized field names, software diagnostic non-claim eligibility, prepared-host hardware claim criteria, expected-failure classifications, stdout/artifact equality, or no-payload reporting.

## CLI/config owner

The CLI/config owner validates inputs before benchmark execution and before any hardware touch. It owns defaults for device path, backend, suite, transfer bytes, iterations, concurrency, duration, output format, and optional artifact path.

The stable parse contract is:

- `--help` succeeds without hardware access and prints `tokio_memmove_bench`, `--backend <hardware|software>`, and `--suite <canonical|latency|concurrency|throughput>`.
- Invalid numeric values for `--bytes`, `--iterations`, `--concurrency`, and `--duration-ms` fail before execution.
- Invalid enum values for `--backend`, `--suite`, and `--format` fail before execution.
- Empty device paths, missing `--artifact` values, directory artifact paths, and readonly or unwritable artifact paths fail before execution.
- `MemmoveRequest` validation for `--bytes` remains part of the config boundary.

S03 may move parser code, but it must not convert invalid config into a runtime or hardware failure.

## Suite/mode dispatch owner

The suite/mode owner maps benchmark suites to result rows:

- `canonical` emits `single_latency`, `concurrent_submissions`, and `fixed_duration_throughput`.
- `latency` emits only `single_latency`.
- `concurrency` emits only `concurrent_submissions`.
- `throughput` emits only `fixed_duration_throughput`.

The owner also preserves target naming: software rows use `software_direct_async_diagnostic`; hardware async rows use `direct_async`; hardware sync comparison rows use `direct_sync`. The `comparison_target` relation stays `direct_async` ↔ `direct_sync` for latency comparison rows.

## Software diagnostic backend

The software diagnostic owner provides ordinary-host evidence that the Tokio benchmark CLI, direct async request path, row generation, artifact schema, renderer, writer, and verifier validation still compose. It consumes `AsyncDsaSession`, `AsyncDsaHandle`, and `AsyncMemmoveRequest` from the direct async runtime through the existing software diagnostic backend, but it does not change direct async behavior or public accessors.

Software diagnostic artifacts and rows must keep:

- `backend="software"`.
- `target="software_direct_async_diagnostic"`.
- `runtime_flavor="current_thread"`.
- `worker_threads=1`.
- `claim_eligible=false` at top level and in every row.
- positive operation counts, latency metrics, and rate metrics for passing rows.
- null failure metadata on passing rows.

Software diagnostic success is useful host-free proof, not prepared-host hardware proof. It must never be described as DSA/IAX hardware success or performance evidence.

## Hardware execution and claim gating owner

The hardware owner opens the prepared-host direct async session, runs `direct_async` modes, runs the `direct_sync` comparison row where required, and converts queue-open, validation, direct, lifecycle, worker, join, and sync failures into artifact fields.

Prepared-host hardware claims require all of these conditions together:

- top-level `ok=true`;
- top-level `verdict=pass`;
- top-level `claim_eligible=true`;
- all suite-required `direct_async` rows pass and are claim-eligible;
- canonical and latency suites include the required `direct_sync` comparison row;
- verifier final `phase=done` reports `verdict=pass` and `claim_eligible=true`.

Expected hardware failures must remain non-claim-eligible and machine-readable. Queue-open failures, classified direct async failures, validation failures, sync comparison failures, and prepared-host preflight failures preserve `failure_class`, `error_kind`, `direct_failure_kind`, `validation_phase`, `validation_error_kind`, direct retry metadata, and completion metadata where applicable.

## Verifier semantics owner

The verifier owner is the shell-level inspection surface. It validates preflight prerequisites, executes the benchmark under timeout, validates the emitted artifact, and produces final machine-readable lines.

The final `phase=done` line remains the inspection surface for successful runs and expected failures. The `phase=artifact_validation` line remains the inspection surface for hard schema, JSON, redaction, and stdout/artifact failures.

Expected failures remain expected failures when prerequisites or real hardware conditions are absent or classified:

- missing work queue;
- missing `devenv`;
- missing launcher;
- missing launcher capability;
- classified hardware runtime failure with machine-readable `failure_class` and `error_kind`.

Hard failures remain hard failures when the benchmark or artifact violates the contract:

- invalid verifier backend or suite override;
- contradictory verifier overrides;
- timeout;
- malformed JSON;
- missing required schema fields;
- stdout/artifact mismatch;
- software artifact or software row with `claim_eligible=true`;
- hardware success missing the required `direct_sync` comparison row;
- forbidden payload dump fields.

## No-payload constraints

Benchmark artifacts, verifier diagnostics, report text, tests, and guard output may include metadata such as device paths, requested bytes, iterations, concurrency, duration, backend, suite, mode, target, operation counts, rates, failure class, error kind, validation phase, direct failure kind, retry budget, retry count, completion status, completion result, completion bytes completed, completion fault address, stdout/stderr paths, launcher status, and timeout values.

They must not include copied source bytes, destination bytes, raw payload dumps, payload bytes, source payload fields, destination payload fields, or payload-content fields. This no-payload rule applies to successful artifacts, expected-failure artifacts, verifier hard-failure diagnostics, report examples, and future test fixtures.
