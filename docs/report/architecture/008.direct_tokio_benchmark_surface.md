# Direct Tokio benchmark proof surface

## Reader and action

This handoff is for the engineer preparing S04 evidence. After reading it, they should be able to decide whether a `tokio_memmove_bench` JSON artifact is eligible as hardware acceleration evidence and know which verifier outcome is ordinary on an unprepared host.

## Benchmark schema labels

The benchmark emits schema version `1` with one top-level artifact and a `results` array. The canonical suite contains three direct async modes:

- `single_latency`
- `concurrent_submissions`
- `fixed_duration_throughput`

Hardware artifacts use `backend=hardware`. Claim-eligible hardware success requires `claim_eligible=true`, direct async rows with `target=direct_async`, and a paired synchronous comparison row with `target=direct_sync` for the single-latency comparison. The async single-latency row names `comparison_target=direct_sync`; the sync comparison row names `comparison_target=direct_async`.

Software diagnostic artifacts use `backend=software` and `target=software_direct_async_diagnostic`. They must remain `claim_eligible=false` at both the artifact and row level. They are useful for host-free contract checks but are not hardware acceleration evidence.

## Verifier verdicts

`verify_tokio_memmove_bench.sh` validates stdout and artifact equality before trusting any result. Malformed JSON, missing fields, stdout/artifact divergence, contradictory claim eligibility, and hardware success without the direct sync comparison are hard verifier failures.

A prepared hardware host should end with `phase=done verdict=pass`. The final phase line includes backend, suite, claim eligibility, completed and failed operation counts, target labels, artifact path, stdout path, and stderr path.

An ordinary host without a visible work queue, launcher, launcher capability, or `devenv` should end with `phase=done verdict=expected_failure failure_phase=preflight`. That is not S04 evidence, but it is a truthful verifier outcome.

A hardware runtime artifact that reports a typed queue-open, async direct, validation, or sync comparison failure should end with `verdict=expected_failure failure_phase=runtime` while preserving the artifact/stdout/stderr paths. That means the benchmark/verifier surface worked and the hardware run did not produce claim-eligible evidence.

## S04 evidence recommendation

Use the verifier in release profile for claim-oriented measurements:

```bash
IDXD_RUST_VERIFY_PROFILE=release \
IDXD_RUST_VERIFY_BYTES=4096 \
IDXD_RUST_VERIFY_ITERATIONS=1000 \
IDXD_RUST_VERIFY_CONCURRENCY=16 \
IDXD_RUST_VERIFY_DURATION_MS=5000 \
bash idxd-rust/scripts/verify_tokio_memmove_bench.sh
```

Keep the default verifier knobs for quick operator checks and CI-style contract validation. The defaults are intentionally short and should not be presented as final performance evidence.

## Out of scope

This benchmark is deliberately standalone and Tokio-only. It does not prove Tonic integration, codec behavior, worker-fixture batching, MOVDIR64 alternatives, or multi-operation aggregation. Those remain separate integration or optimization questions for later slices.
