# Tonic Interception Points for `accel-rpc`

**Date**: 2026-03-30
**Scope**: `accel-rpc/tonic` and surrounding `accel-rpc` integration crates

## Executive Summary

For this repository, the word **intercept** maps to four different mechanisms inside tonic:

1. **gRPC interceptors** for request metadata and early rejection,
2. **Tower layers/services** for whole-request and whole-response wrapping,
3. **transport builders/connectors** for connection setup and protocol tuning,
4. **codec/body boundaries** for actual payload bytes, framing, compression, and buffer movement.

The main conclusion is:

- **If the goal is payload-path work, use codec/body hooks first.**
- **If the goal is cross-cutting request/response wrapping, use Tower layers.**
- **If the goal is ergonomic adoption, wire those hooks in through `tonic-build`.**
- **Do not treat tonic interceptors as payload hooks.** They are intentionally metadata-only.

That means the best fit for this repo remains:

- `accel-codec/` at codec/body boundaries,
- `accel-middleware/` at Tower layer boundaries,
- `dsa-ffi/` under copy/checksum-oriented payload transforms,
- `iax-ffi/` under compression/decompression boundaries,
- `tonic-build` only as the mechanism that makes those real hooks easier to adopt.

This report complements `docs/report/architecture/002.tonic_component_analysis.md` by narrowing from general offloadability to **concrete interception points**.

## 1. End-to-End Map: Where Interception Can Happen

### 1.1 Generated client path

The generated client surface in `accel-rpc/tonic/tonic-build/src/client.rs` is the first interception layer.

- `new(...)` wraps the inner transport in `tonic::client::Grpc::new(...)` (`client.rs:75-78`).
- `with_origin(...)` does the same with an explicit origin (`client.rs:80-83`).
- `with_interceptor(...)` wraps the inner service in `InterceptedService::new(inner, interceptor)` before constructing the client (`client.rs:85-96`).
- Each generated RPC method inserts `GrpcMethod` into request extensions before calling the runtime client (`client.rs:235-247`).
- Generated `connect(...)` goes through `tonic::transport::Endpoint::new(dst)?.connect().await?` (`client.rs:142-151`).

So the client chain is:

```text
generated client
 -> optional with_interceptor(...)
 -> tonic::client::Grpc<T>
 -> Endpoint / Channel / Service
 -> EncodeBody::new_client(...)
 -> hyper/h2 transport
```

### 1.2 Generated server path

The generated server surface in `accel-rpc/tonic/tonic-build/src/server.rs` provides the mirrored hooks.

- `Server::new(...)` and `from_arc(...)` build the generated wrapper (`server.rs:124-137`).
- `with_interceptor(...)` returns `InterceptedService<Self, F>` (`server.rs:139-144`).
- The generated `Service<http::Request<B>>` dispatches by URI path (`server.rs:151-177`).
- For each RPC style, generated code constructs `tonic::server::Grpc::new(codec)` and then calls `.unary(...)`, `.server_streaming(...)`, `.client_streaming(...)`, or `.streaming(...)` (`server.rs:485-490`, `552-557`, `610-615`, `678-683`).

So the server chain is:

```text
Server::builder()
 -> optional Server::layer(...)
 -> generated FooServer<T>
 -> optional with_interceptor(...)
 -> tonic::server::Grpc<TCodec>
 -> Streaming::new_request(...) / EncodeBody::new_server(...)
 -> hyper/h2 transport
```

### 1.3 Generated vs handwritten boundary

The important structural boundary is:

- **generated code** decides how application traits are wrapped and forwarded,
- **handwritten tonic runtime** performs request preparation, framing, compression, streaming decode, and transport I/O.

For interception design, this means:

- if you want an ergonomic API, change or wrap **codegen**;
- if you want to touch payload bytes, change or wrap the **runtime**.

## 2. Interception Taxonomy

## 2.1 Codec/body boundary — best hook for payload work

The strongest payload interception point lives in the client/server runtime and codec code.

### Client-side anchors

In `accel-rpc/tonic/tonic/src/client/grpc.rs`, the runtime client turns an outbound request stream into an encoded HTTP body:

- `Grpc::streaming(...)` maps the request stream into `EncodeBody::new_client(...)` (`client/grpc.rs:297-305`),
- wraps it in tonic `Body` (`client/grpc.rs:306`),
- prepares headers and URI (`client/grpc.rs:308`, `372-391`),
- then calls the inner service (`client/grpc.rs:310-314`).

On the response path it creates `Streaming::new_response(...)` (`client/grpc.rs:353-361`).

### Server-side anchors

In `accel-rpc/tonic/tonic/src/server/grpc.rs`:

- unary requests become `Streaming::new_request(...)` in `map_request_unary(...)` (`server/grpc.rs:376-381`),
- streaming requests become `Streaming::new_request(...)` in `map_request_streaming(...)` (`server/grpc.rs:407-413`),
- outbound responses become `EncodeBody::new_server(...)` in `map_response(...)` (`server/grpc.rs:447-453`).

### Byte-level anchors

`accel-rpc/tonic/tonic/src/codec/encode.rs` shows the exact outbound sequence:

- reserve the 5-byte gRPC header (`encode.rs:145-150`),
- encode the message (`encode.rs:155-157`, `172-174`),
- optionally compress (`encode.rs:152-170`),
- write the frame header (`encode.rs:177-205`).

`accel-rpc/tonic/tonic/src/codec/decode.rs` shows the inbound mirror:

- read compression flag and frame length (`decode.rs:152-205`),
- accumulate message bytes,
- optionally decompress (`decode.rs:214-245`),
- then decode the protobuf message from the completed buffer.

### What this hook can do

- observe encoded payload sizes,
- replace or wrap compression/decompression,
- control buffer reuse and pooling,
- insert copy/checksum operations close to the actual bytes,
- add tracing at message/frame boundaries.

### What it cannot do well

- replace HTTP/2 stream scheduling,
- change Tower readiness/backpressure semantics,
- improve Tokio task orchestration directly.

### Fit for `accel-rpc`

This is the **highest-value hook** for `accel-codec`, `dsa-ffi`, and `iax-ffi`.

## 2.2 Tower layer boundary — best hook for transparent request/response wrapping

The server builder exposes a full Tower layer stack.

In `accel-rpc/tonic/tonic/src/transport/server/mod.rs`:

- `Server::trace_fn(...)` attaches tracing spans from inbound headers (`server/mod.rs:494-503`),
- `Server::layer(...)` installs a Tower `Layer` around all services (`server/mod.rs:563-650`),
- the docs example explicitly shows `ServiceBuilder` stacking `InterceptorLayer`s (`server/mod.rs:582-617`).

On the client side, `Channel` is a `Service<http::Request<Body>>` (`transport/channel/mod.rs:208-221`), so it can also participate in service wrapping.

### What this hook can do

- wrap full request/response objects,
- attach policy, metrics, tracing, and extensions,
- wrap bodies without changing generated application code,
- centralize cross-cutting middleware in `accel-middleware/`.

### What it cannot do as cleanly

- reason about protobuf messages without parsing payload framing itself,
- expose byte-level compression and frame assembly as directly as the codec layer.

### Fit for `accel-rpc`

This is the **second-best hook**, especially when the repo wants transparent middleware-like transforms across many services.

## 2.3 gRPC interceptors — metadata-only hook

`accel-rpc/tonic/tonic/src/service/interceptor.rs` defines tonic's interceptor model.

Key properties are explicit in the code and comments:

- an interceptor can add/remove/check metadata and extensions, or cancel with `Status` (`interceptor.rs:18-23`),
- tonic itself says interceptors are less flexible than Tower middleware and not the preferred logging mechanism (`interceptor.rs:30-36`),
- `InterceptedService::call(...)` intentionally strips the body before calling the interceptor and reconstructs the HTTP request afterward (`interceptor.rs:129-149`).

This design is the critical limit: **the interceptor never sees the message body**.

Generated client and server APIs expose this directly:

- client `with_interceptor(...)` (`tonic-build/src/client.rs:85-96`),
- server `with_interceptor(...)` (`tonic-build/src/server.rs:139-144`).

### What this hook can do

- auth and admission checks,
- attach request metadata,
- attach extensions that later layers consume,
- reject early before payload handling.

### What it cannot do

- inspect or rewrite protobuf messages,
- observe responses,
- implement compression/copy/checksum transforms,
- serve as the main payload-path instrumentation hook.

### Fit for `accel-rpc`

Useful only as a **control-plane hint path**. For example, an interceptor can attach an extension or metadata flag saying “eligible for offload,” but the actual offload must happen later in middleware or codec code.

## 2.4 Codegen boundary — ergonomic hook, not byte hook

`tonic-build` is not the runtime data path, but it is where tonic decides what client/server APIs expose.

Important generated client hooks:

- `new(...)`, `with_origin(...)`, `with_interceptor(...)` (`tonic-build/src/client.rs:75-96`),
- per-method `GrpcMethod` insertion (`tonic-build/src/client.rs:235-247`).

Important generated server hooks:

- `new(...)`, `from_arc(...)`, `with_interceptor(...)` (`tonic-build/src/server.rs:124-144`),
- URI dispatch to method-specific service adapters (`tonic-build/src/server.rs:165-177`),
- handoff to `tonic::server::Grpc` (`tonic-build/src/server.rs:485-490`, `552-557`, `610-615`, `678-683`).

### What this hook can do

- standardize how clients/servers are built,
- make accelerator-aware layers/codecs the default constructor path,
- automatically attach method metadata or extensions,
- reduce app-level boilerplate.

### What it cannot do by itself

- touch payload bytes,
- replace encode/decode/compression work without delegating to runtime hooks.

### Fit for `accel-rpc`

High leverage, but only as a **wiring layer**. It should make the real codec/layer hooks easy to use, not become the payload implementation itself.

## 2.5 Transport builders/connectors — connection and protocol tuning hook

`accel-rpc/tonic/tonic/src/transport/channel/endpoint.rs` exposes the transport-setup boundary.

- `Endpoint::connect(...)` constructs a `Channel` from the endpoint (`endpoint.rs:472-480`),
- `connect_lazy(...)` defers connection establishment (`endpoint.rs:482-493`),
- `connect_with_connector(...)` and `connect_with_connector_lazy(...)` let callers supply a custom connector (`endpoint.rs:495-542`).

`transport/channel/mod.rs` shows where the transport is buffered and executed:

- `Connection::connect(...)` establishes the lower service,
- `Buffer::pair(...)` inserts a buffered service/worker split (`channel/mod.rs:182-188`),
- `Channel` then acts as `Service<http::Request<Body>>` (`channel/mod.rs:208-221`).

On the server side, `Server::builder()` exposes protocol and socket configuration such as timeouts, windows, keepalive, and frame size (`transport/server/mod.rs:156-409`).

### What this hook can do

- swap connectors,
- experiment with custom transports,
- tune connection-level behavior,
- improve locality and integration with custom executors.

### What it cannot do well

- act on gRPC message payloads at a method/message granularity,
- cleanly implement compression/copy/checksum offload.

### Fit for `accel-rpc`

Important for attribution and future transport experiments, but **not the first place** to implement accelerator-oriented payload processing.

## 2.6 Trace and extension hooks — observe more than modify

Two smaller hooks matter for observability and policy propagation.

- `Server::trace_fn(...)` builds a `tracing::Span` from inbound headers (`transport/server/mod.rs:494-503`).
- Generated clients add `GrpcMethod` into request extensions before the runtime call (`tonic-build/src/client.rs:246-247`).

These hooks are valuable because they can feed profiling, method classification, and routing decisions without disturbing payload handling.

### Fit for `accel-rpc`

Use them for attribution and policy selection, not for the actual data transform.

## 3. Practical Limits of Each Hook

| Hook | Can observe/modify | Cannot do well | Best use in this repo |
|---|---|---|---|
| gRPC interceptor | metadata, extensions, early reject | payload/body, responses | auth, offload eligibility hints |
| Tower layer | full HTTP request/response, body wrappers | protobuf-aware transforms without extra parsing | `accel-middleware` wrappers |
| Endpoint / transport | connectors, connection setup, protocol tuning | per-message payload transforms | custom transport experiments |
| Generated code | API shape, default wiring, method metadata | byte processing by itself | ergonomic adoption of custom hooks |
| Codec/body | framing, compression, buffers, payload bytes | scheduler / HTTP/2 state machines | `accel-codec`, DSA/IAX insertion |

## 4. Recommended Interception Strategy for `accel-rpc`

## 4.1 First: intercept at codec/body boundaries

Primary implementation targets:

- wrap `EncodeBody::new_client(...)` and `EncodeBody::new_server(...)`,
- wrap or replace `compress(...)` / `decompress(...)`,
- add pooled-buffer management around `BytesMut` growth and reuse,
- add tracing spans around encode, compress, frame write, decode, and decompress.

This is where the data path is concrete enough for DSA/IAX to matter.

## 4.2 Second: use Tower layers for transparent cross-cutting behavior

Put service-wide transforms and instrumentation in `accel-middleware/` when they should apply uniformly across services without modifying generated application code.

Examples:

- body-size accounting,
- request classification,
- extension injection for later codec decisions,
- whole-request tracing and correlation.

## 4.3 Third: use codegen to make the real hooks easy to adopt

The cleanest long-term developer experience is:

- generated clients default to the repo's custom channel/layer/codec stack,
- generated servers default to the repo's layer and codec wrappers,
- method-level extensions are added automatically.

That keeps app code simple while leaving the real work in runtime hooks.

## 4.4 Fourth: use interceptors only for policy hints

Interceptors are still useful, but narrowly:

- attach metadata or extensions,
- reject unsupported requests early,
- mark requests as offload-eligible before the request reaches middleware/codec code.

They should **not** be sold as the payload interception mechanism.

## 5. Concrete Report-Level Conclusions

1. **Tonic has multiple interception layers, but only one payload-centric layer.**
2. **The codec/body path is the right place to intercept actual bytes.**
3. **Tower layers are the right place to wrap whole services.**
4. **Interceptors are intentionally weaker than middleware.**
5. **`tonic-build` is the ergonomic boundary, not the data boundary.**

## 6. Suggested Next Steps

1. Add tracing around `EncodeBody::{new_client,new_server}` and `Streaming::{new_request,new_response}`.
2. Prototype a custom codec or body wrapper in `accel-codec/` that records buffer growth, copy volume, and compression time.
3. Prototype a Tower layer in `accel-middleware/` that tags requests with method and payload-size hints.
4. Use `tonic-build` to expose repo-specific constructors that install those wrappers by default.

If those measurements show that copy and compression dominate for the target payloads, the repo has a clean path to DSA/IAX-backed interception without trying to replace tonic's control plane.
