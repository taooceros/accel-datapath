# Tonic Component Analysis: Architecture, Profiling, and Offloadability

**Date**: 2026-03-30
**Scope**: `accel-rpc/tonic` and the surrounding `accel-rpc` workspace

## Executive Summary

In this repository, tonic is the software control plane for a gRPC-over-HTTP/2 stack, while the intended accelerator work sits on the data path edges: serialization output buffers, compression/decompression, copy/integrity, and possibly buffer lifecycle management.

The most important conclusion is:

1. **Best offload candidates** are compression/decompression and copy+CRC.
2. **Possible but lower-value candidates** are buffer movement, prefetch-friendly pooling, and some framing-adjacent copies.
3. **Poor offload candidates** are Tokio scheduling, Tower middleware orchestration, HTTP/2 state machines, HPACK, protobuf serialization/deserialization, and general request routing.

That matches both tonic's architecture and the repo's intended crate split:

- `accel-codec/` for codec/buffer work,
- `accel-middleware/` for Tower-layer transforms,
- `dsa-ffi/` for copy/CRC,
- `iax-ffi/` for compression,
- `tonic-profile/` for end-to-end profiling,
- `async-bench/` for runtime overhead characterization.

## 1. Repo-Specific Architecture

### 1.1 Workspace structure

`accel-rpc/README.md` defines the intended architecture:

- `tonic/` — forked tonic subtree
- `accel-codec/` — custom tonic codec with pooled buffers
- `accel-middleware/` — Tower CRC/compression middleware
- `dsa-ffi/` — DSA bridge
- `iax-ffi/` — IAX bridge
- `async-bench/` — async overhead baseline tool
- `tonic-profile/` — tonic profiling harness

Today, the accelerator-facing crates are mostly placeholders, but the tonic fork already exposes the right insertion points.

### 1.2 tonic stack layering

From `accel-rpc/tonic/README.md` and the forked source, the stack is:

```text
tonic-build          codegen from .proto
  -> tonic           gRPC API, codec, streaming, metadata, status
    -> transport     channel/server glue, tower layering, connection setup
      -> hyper       HTTP client/server integration
        -> h2        HTTP/2 framing, flow control, stream state
          -> tokio   runtime, reactor, sockets, task scheduling
```

The send path is roughly:

```text
app object
 -> prost encode
 -> tonic codec encode
 -> optional compression
 -> gRPC 5-byte frame header
 -> tower layers
 -> hyper/h2 HTTP/2 framing
 -> tokio socket I/O
```

The receive path reverses this:

```text
tokio socket I/O
 -> hyper/h2 frame decode
 -> tonic streaming decode
 -> optional decompression
 -> prost decode
 -> app object
```

## 2. Concrete Component Boundaries in the Fork

### 2.1 Codec boundary: the main data-path insertion point

`accel-rpc/tonic/tonic/src/codec/mod.rs` defines the central abstraction:

- `Codec`
- `Encoder`
- `Decoder`
- `BufferSettings`

This is the cleanest public extension point because it controls message-to-bytes and bytes-to-message transformation without requiring transport rewrites.

Important implications:

- custom buffer policies fit here,
- custom instrumentation fits here,
- accelerator-aware encode/decode wrappers fit here,
- zero-copy or pooled-buffer experiments should start here.

### 2.2 Outbound encode pipeline

`accel-rpc/tonic/tonic/src/codec/encode.rs` shows the actual outbound work sequence:

1. reserve `HEADER_SIZE`
2. encode message into `BytesMut`
3. optionally compress
4. write the 5-byte gRPC header
5. batch/yield encoded bytes to the HTTP body

Key observations:

- The hot path allocates and grows `BytesMut` buffers.
- Compression is an explicit step with a tight boundary.
- The gRPC header work is trivial.
- Streaming batches messages up to `yield_threshold`, which affects cache and allocation behavior.

### 2.3 Inbound decode pipeline

`accel-rpc/tonic/tonic/src/codec/decode.rs` shows the receive path:

1. accumulate `Body` data into `BytesMut`
2. read the compression flag + length
3. optionally decompress into a separate buffer
4. decode the message from the completed message buffer

Key observations:

- Decompression is isolated and measurable.
- Length parsing and framing are cheap.
- Buffer accumulation and copy behavior matter for large or streaming workloads.
- Decode happens only after a full gRPC message is assembled.

### 2.4 Compression boundary

`accel-rpc/tonic/tonic/src/codec/compression.rs` is the most obvious accelerator hotspot.

The file exposes:

- `CompressionEncoding`
- `compress(...)`
- `decompress(...)`

Current implementation uses software codecs (`flate2`, `zstd`) and `std::io::copy` into `BytesMut` writers. For this repo, that is the natural swap point for IAX-based deflate experiments.

### 2.5 Client transport boundary

`accel-rpc/tonic/tonic/src/transport/channel/service/connection.rs` builds the client stack:

- connection setup
- tower layers for origin, user-agent, timeout, concurrency, rate limit
- hyper HTTP/2 handshake
- spawned connection task

This area is important for profiling, but not a good accelerator target. It is mostly orchestration, readiness handling, and protocol glue.

### 2.6 Server transport boundary

`accel-rpc/tonic/tonic/src/transport/server/mod.rs` builds the server-side stack:

- server builder configuration
- connection limits / load shedding / timeouts
- hyper-util server connection setup
- Tower service composition

Again, important for attribution, but usually not offloadable. Its value is in helping isolate whether the CPU budget is being spent above or below the codec boundary.

## 3. Component-by-Component Offloadability

### 3.1 Tokio runtime and task scheduling

**Role**: polling futures, waking tasks, socket readiness, task execution.

**Examples of cost**:

- excessive poll/wake traffic
- cross-thread wakeups
- runtime contention
- spawn/join overhead

**Offloadability**: **very poor**.

Reason:

- this is control flow, not a bulk data transform,
- it is latency-sensitive and stateful,
- accelerators do not help with future scheduling.

What to optimize instead:

- reduce task hops,
- avoid unnecessary channels,
- pin runtime placement and NUMA locality,
- overlap accelerator completion with async polling rather than trying to offload runtime logic.

Repo hook:

- `accel-rpc/async-bench/` should measure spawn, wake, poll, and completion-notification overhead.

### 3.2 hyper/h2 transport

**Role**: HTTP/2 frames, flow control, stream multiplexing, headers/trailers.

**Offloadability**: **poor**.

Reason:

- state-machine heavy,
- branchy and control-dominant,
- tied closely to connection semantics,
- difficult to isolate into accelerator-friendly descriptors.

Possible exception:

- if transport-level buffer copies become visible in profiles, use DSA for the copies, not for HTTP/2 semantics.

### 3.3 Tower middleware

**Role**: request/response pipeline transformations and policy layers.

**Offloadability**: **mixed**.

- generic policy layers: **not offloadable**
- payload transforms implemented as middleware: **sometimes offloadable**

Good candidates inside middleware:

- checksum generation/verification
- compression/decompression if implemented above codec
- large payload copy/duplication

Poor candidates:

- auth logic
- timeout/retry/rate-limit decisions
- routing and request classification

### 3.4 Protobuf serialization/deserialization

**Role**: `prost` encodes Rust structs to protobuf bytes and decodes them back.

**Offloadability**: **poor**.

Reason:

- field- and schema-dependent branching,
- varint handling,
- nested object traversal,
- irregular control flow.

This stage is usually the irreducible CPU core of the application data path.

What still matters:

- buffer allocation policy around prost,
- minimizing copies after prost writes bytes,
- keeping serialization overlapped with accelerator work on previous messages.

### 3.5 Codec buffers and message assembly

**Role**: `BytesMut` growth, staging buffers, stream batch thresholds, gRPC message assembly.

**Offloadability**: **medium-low**, but important.

The assembly logic itself is not worth offloading, but the surrounding memory movement may be.

Good targets:

- DSA-backed copy from serialization buffer to transport-ready buffer
- pooled/pre-faulted buffers
- avoiding double-buffering before/after compression
- using DSA `copy_crc` when a copy is unavoidable and integrity is useful

### 3.6 Compression/decompression

**Role**: payload reduction before send, expansion after receive.

**Offloadability**: **high**.

This is the strongest candidate because it is:

- computationally heavy,
- bulk-data oriented,
- naturally delimited in tonic,
- already represented by explicit `compress` / `decompress` functions.

Repo mapping:

- IAX fits send compression and receive decompression.
- `accel-middleware/` or codec-level wrappers can host the experiment.

### 3.7 Copy and integrity

**Role**: moving encoded/compressed data among buffers and optionally computing checksums.

**Offloadability**: **medium to high** when copies are large enough or frequent enough.

Best fit:

- DSA `data_move`
- DSA `copy_crc`

Usefulness depends on whether the software path is actually copy-heavy. If the profile shows only small payloads and negligible copy cost, DSA may lose on submission overhead.

### 3.8 TLS and socket I/O

**Role**: encryption and kernel/network transfer.

**Offloadability**: **low in this environment**.

Reason:

- TLS is already hardware-assisted on CPU via AES-NI,
- network I/O is bounded by the NIC/kernel path available here,
- this repo is centered on DSA/IAX, not NIC offload.

## 4. Profiling Strategy: What to Measure First

Profiling must separate four questions:

1. where CPU time goes,
2. where wall-clock latency goes,
3. which costs are bulk-data transforms versus orchestration,
4. whether accelerator submission/completion overhead is smaller than the software cost being replaced.

The correct workflow is staged.

### Stage A: End-to-end baseline

Build a simple unary-RPC baseline in `tonic-profile/` with message size sweeps.

Measure at minimum:

- messages/sec
- p50/p95/p99 latency
- client CPU%
- server CPU%
- bytes/sec
- compression ratio if compression is enabled

Sweep dimensions:

- payload sizes: 64 B, 256 B, 1 KiB, 4 KiB, 16 KiB, 64 KiB, 256 KiB, 1 MiB
- unary vs streaming
- compression off vs on
- single connection vs many concurrent streams
- same-socket/NUMA-local vs cross-socket placement where relevant

Without this, offload conclusions are usually wrong.

### Stage B: Runtime attribution

Use Tokio-focused tooling to learn whether the bottleneck is orchestration.

Tools:

- `tokio-console` / `console-subscriber`
- `async-bench` microbenchmarks

Questions to answer:

- Are tasks spending time runnable but unscheduled?
- Are there too many wakeups per RPC?
- Is there cross-thread bouncing?
- Would accelerator completions introduce more task churn than they save?

If runtime overhead dominates, offloading compression will not fix the real bottleneck.

### Stage C: CPU hotspot sampling

Use sampling profilers to identify real hot code.

Tools:

- `perf record`
- flamegraphs / `cargo flamegraph`
- Rust-native `pprof` if integrated into `tonic-profile`

Questions:

- How much inclusive time lands in `prost`?
- How much lands in `tonic::codec::compression::{compress,decompress}`?
- How much lands in copy-heavy buffer manipulation?
- How much is spent below tonic in hyper/h2/tokio?

This step determines whether the workload is data-transform heavy enough to justify DSA/IAX.

### Stage D: Span-based phase timing

Add `tracing` spans around major stages.

Recommended span boundaries:

- `serialize`
- `compress`
- `copy_crc`
- `grpc_frame`
- `http2_send`
- `socket_write`
- `socket_read`
- `http2_decode`
- `grpc_deframe`
- `decompress`
- `deserialize`

This gives per-stage latency distributions rather than only CPU samples.

### Stage E: Offload crossover analysis

For each candidate stage, compare:

```text
software stage cost
vs
submit cost + queueing + DMA/setup + completion polling/wakeup + copy side effects
```

If accelerator total cost is not lower at a given size/concurrency regime, it is not offloadable in practice even if it is theoretically offloadable.

This repo's earlier DSA reports already show why analytical estimates alone are dangerous; phase-level attribution must be measured, not guessed.

## 5. Detailed Profiling Playbook Per Component

## 5.1 Tokio runtime

### Objective

Determine whether async orchestration cost dominates enough to block offload gains.

### Method

- expand `async-bench/` to measure:
  - `tokio::spawn` + join
  - oneshot completion notification
  - channel send/recv
  - manual poll/wake loops
  - `spawn_blocking` baseline
- run `tokio-console` on `tonic-profile`
- pin tasks and worker threads to inspect NUMA effects

### Metrics

- polls per RPC
- wakeups per RPC
- task migration count
- scheduler delay
- completion-to-wakeup latency

### Offload decision rule

- If completion notification and wake costs are a large fraction of the stage you want to offload, that stage likely should stay synchronous/inlined or use batching.

## 5.2 hyper/h2 transport

### Objective

Measure transport protocol cost independently from codec and compression.

### Method

- use tiny messages with compression disabled
- compare unary and streaming
- sample with `perf`
- inspect functions in `h2`, `hyper`, and `hyper-util`

### Metrics

- CPU share in frame encode/decode
- stream control overhead
- header/trailer overhead
- lock/contention behavior if present

### Offload decision rule

- Only offload adjacent memory movement if transport code reveals significant copy cost. Do not try to offload HTTP/2 logic itself.

## 5.3 Tower middleware

### Objective

Classify middleware as policy-heavy or payload-transform-heavy.

### Method

- benchmark with middleware disabled, then enabled one layer at a time
- add `tracing` spans per layer
- if implementing checksum/compression middleware, benchmark middleware-only transforms against codec-integrated transforms

### Metrics

- per-layer added latency
- per-layer CPU%
- allocation count deltas
- bytes touched per layer

### Offload decision rule

- Layers that mostly inspect metadata are not offload targets.
- Layers that transform bulk bytes are candidates if the transform cost exceeds accelerator overhead.

## 5.4 prost serialization/deserialization

### Objective

Quantify the irreducible CPU floor.

### Method

- benchmark serialization-only and deserialization-only in-process
- use representative message schemas: small flat, nested, repeated-field heavy, large byte payload
- collect flamegraphs with compression disabled

### Metrics

- ns/message
- CPU cycles/byte
- branch misses
- allocations/message

### Offload decision rule

- Treat as CPU-resident baseline.
- Use these results to know the maximum possible speedup from offloading everything else.

## 5.5 Codec buffers and message assembly

### Objective

Understand allocation growth, copy count, and buffer churn.

### Method

- instrument `encode.rs` and `decode.rs` around:
  - buffer reserve/grow events
  - `yield_threshold` batching behavior
  - decompression buffer use
- compare default `BytesMut` growth with a pooled-buffer prototype in `accel-codec/`

### Metrics

- allocations per RPC
- bytes copied per RPC
- peak live buffer footprint
- buffer growth count
- batch size distribution

### Offload decision rule

- If extra copies are frequent and payloads are large enough, DSA-assisted copies may help.
- If costs are mostly allocation and bookkeeping, buffer pooling is likely better than accelerator offload.

## 5.6 Compression/decompression

### Objective

Decide whether IAX compression offload is worthwhile and where it should attach.

### Method

- benchmark software compression in `tonic::codec::compression`
- measure send and receive separately
- sweep payload size and compression ratio
- compare software deflate vs accelerator deflate once `iax-ffi/` is wired
- test batched versus per-message submission

### Metrics

- compress ns/byte
- decompress ns/byte
- achieved compression ratio
- queue depth / in-flight ops
- completion latency
- CPU freed on client/server

### Offload decision rule

- High CPU share + favorable compression ratio + accelerator cost below software cost = good candidate.
- Small messages with poor compression ratio are likely below crossover and should stay on CPU.

## 5.7 Copy + CRC

### Objective

Determine whether DSA `data_move` or `copy_crc` can eliminate enough copy/checksum work.

### Method

- instrument copy points before send and after receive
- compare:
  - memcpy only
  - memcpy + software CRC
  - DSA `data_move`
  - DSA `copy_crc`
- test with and without extra transport-buffer staging

### Metrics

- bytes copied per RPC
- copy ns/byte
- CRC ns/byte
- submission/completion overhead
- cache miss deltas

### Offload decision rule

- Best when copy is unavoidable and CRC is useful anyway.
- Weak when payloads are tiny or zero-copy already dominates.

## 5.8 TLS and network I/O

### Objective

Avoid misclassifying network bottlenecks as codec bottlenecks.

### Method

- compare localhost/in-memory style baselines to real socket runs
- measure with compression off/on
- inspect kernel and NIC counters as needed

### Metrics

- socket write/read latency
- throughput ceiling
- retransmits / stalls if any

### Offload decision rule

- If the workload is network-bound, DSA/IAX data-path offload will not move the top-line result much.

## 6. Recommended Experiment Order for This Repo

1. **Fill out `tonic-profile/`** with unary RPC baseline and payload sweep.
2. **Expand `async-bench/`** to measure scheduler and completion-notification overhead.
3. **Instrument tonic fork with `tracing` spans** at serialize/compress/copy/frame/send/recv/decompress/deserialize boundaries.
4. **Collect `perf`/flamegraph baselines** with compression off and on.
5. **Prototype software-only pooled codec** in `accel-codec/` to remove allocation noise.
6. **Prototype software middleware checksum/compression** in `accel-middleware/` to compare placement choices.
7. **Wire IAX compression** into the codec compression boundary.
8. **Wire DSA copy or copy+CRC** only if the measured copy budget is significant.

This order minimizes the risk of building accelerator plumbing before proving the stage is worth offloading.

## 7. Practical Decision Matrix

| Component | Measure first | Offloadability | Likely target |
|---|---|---:|---|
| Tokio runtime | polls, wakeups, scheduler delay | Low | optimize software only |
| hyper/h2 | CPU share in framing/state | Low | maybe adjacent copy only |
| Tower policy layers | per-layer latency | Low | software only |
| Tower payload transforms | bytes touched, CPU share | Medium | DSA/IAX if bulk transform |
| prost encode/decode | cycles/msg, branch misses | Low | CPU baseline only |
| Codec buffers | allocations, copies, growth | Medium-Low | pooling first, DSA second |
| Compression | ns/byte, ratio, CPU share | High | IAX |
| Copy + CRC | bytes moved, ns/byte | Medium-High | DSA |
| TLS / network I/O | throughput ceiling | Low | not DSA/IAX target |

## 8. Bottom Line

Tonic's architecture is favorable to offload work precisely because it separates protocol orchestration from payload transformation.

For this repository, the right interpretation is:

- **Keep tonic/tower/hyper/h2/tokio as the control plane.**
- **Treat codec/compression/copy boundaries as the accelerator data plane.**
- **Prove offloadability with measured crossover analysis, not intuition.**

If the profiling results show that `prost` and runtime orchestration dominate, accelerator work will have limited upside. If compression and copy dominate for the target message sizes and concurrency levels, IAX and DSA are well aligned with tonic's existing component boundaries.
