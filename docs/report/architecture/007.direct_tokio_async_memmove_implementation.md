# Direct Tokio async memmove implementation note

## Summary

M007/S02 implements the public `idxd-rust` async memmove path as a direct completion-driven runtime. `AsyncDsaSession::open`, `open_with_retries`, and `open_default` construct a `DirectAsyncMemmoveRuntime` over a real IDXD work-queue portal, and cloned `AsyncDsaHandle`s submit owned `AsyncMemmoveRequest` values through that runtime. The hidden legacy worker fixture remains available only for host-independent compatibility tests; it is not the public default path and is not a fallback when direct setup or submission fails.

## Runtime shape

The direct async runtime keeps each accepted operation alive as operation-owned state:

- aligned DSA descriptor and dynamic per-request completion record;
- source `Bytes` and destination `BytesMut`;
- original destination readable length and requested byte count;
- retry budget/count and completion snapshot metadata;
- a waiter signal used to resolve the awaiting future after completion observation.

Submission uses ENQCMD-oriented accept/reject behavior. Rejection is bounded with Tokio-friendly yield/backoff and reports typed `AsyncDirectFailure` metadata such as `backpressure_exceeded`, retry budget/count, requested byte count, and any observed completion snapshot. A long-lived Tokio monitor scans completion records, classifies snapshots through the existing validation state machine, performs page-fault retry continuation when requested, and advances `BytesMut` length only after terminal successful completion plus post-copy validation.

Dropping an awaiting task after accepted submission does not drop descriptor, completion, source, or destination state early. The monitor owns the accepted operation until terminal completion and discards the result only if the receiver has gone away.

## Operator proof commands

Host-free S02 proof from the repository root:

```bash
cargo test -p idxd-rust --test async_memmove_contract
cargo test -p idxd-rust --test tokio_handle_contract
cargo test -p idxd-rust --test async_verifier_contract
cargo test -p idxd-rust --test direct_async_contract_guard
cargo test -p idxd-rust --lib
bash idxd-rust/scripts/verify_direct_async_completion_contract.sh
```

Live smoke proof on a prepared host:

```bash
bash idxd-rust/scripts/verify_async_memmove.sh --bytes 64
```

A prepared DSA shared-WQ host should end with `verdict=pass`. An unprepared host is still acceptable when the verifier emits a classified `verdict=expected_failure` with explicit fields such as `launcher_status=missing_work_queue`, `missing_launcher`, `missing_capability`, or a typed validation/direct failure. Missing artifacts, malformed JSON, stdout/artifact divergence, and unclassified failures remain verifier failures.

## Observability surfaces

Future agents should inspect:

- `AsyncMemmoveError::kind`, `lifecycle_failure_kind`, `worker_failure_kind`, `direct_failure_kind`, and `direct_failure` accessors;
- `await_memmove` JSON fields: `error_kind`, `direct_failure_kind`, `retry_budget`, `retry_count`, `completion_result`, `completion_bytes_completed`, `completion_fault_addr`, `validation_phase`, and `validation_error_kind`;
- `verify_async_memmove.sh` phase logs, especially `phase=artifact_validation` and final `phase=done` lines;
- host-free tests in `idxd-rust/tests/async_memmove_contract.rs` and cloned-handle composition tests in `idxd-rust/tests/tokio_handle_contract.rs`.

Diagnostics intentionally include sizes, phases, retry counts, completion status/result, bytes completed, fault address, and device path. They must not log or serialize source or destination payload bytes.

## S03 handoff

S03 can benchmark the public direct async path through the existing caller contract:

```rust
AsyncDsaHandle::memmove(AsyncMemmoveRequest::new(source, destination)?).await
```

Benchmark work should treat the direct runtime as the baseline substrate and should not reintroduce public allocation convenience constructors, borrowed copy-back helpers, MOVDIR64 fallback, FIFO worker serialization assumptions, or payload-byte diagnostics. Throughput and Criterion-style measurement are intentionally left to S03.
