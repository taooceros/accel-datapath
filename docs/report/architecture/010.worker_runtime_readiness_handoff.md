# Worker-runtime readiness handoff

Status: S05/T01 handoff for future worker-runtime, MOVDIR64, registry/pool, and Tonic/RPC planning. This is not a chronological recap of S04; it is the boundary contract a future planner or executor should apply before treating M007 as evidence for worker-runtime work.
Date: 2026-04-28

## Reader and action

Reader: the engineer planning or executing the next worker-runtime milestone after the direct Tokio IDXD baseline.

Post-read action: decide which M007 evidence can be used for planning, which claims are still blocked, and which prepared-host checks must pass before implementing or advertising worker batching, MOVDIR64/MOVDIR64B serialization, registry/pool ownership, scheduling policy, or Tonic/RPC integration.

## Readiness classification

| Readiness level | Current M007 status | What it permits | What it forbids |
|---|---|---|---|
| planning readiness | Ready. M007 defines the direct completion-driven async substrate, benchmark/verifier contract, software diagnostic path, and ordinary-host `expected_failure` classification. | Use M007 to plan worker-runtime interfaces, failure taxonomy, proof commands, and future milestone gates. | Do not present ordinary-host `expected_failure` or software diagnostics as prepared-host hardware proof. |
| execution readiness | Blocked until prepared-host evidence exists. | Start execution only after a prepared-host run proves the direct baseline can produce accepted hardware evidence. | Do not start from an assumption that this ordinary host already proved hardware throughput, latency, fairness, or queue behavior. |
| claim readiness | Blocked until the verifier reports `verdict=pass claim_eligible=true` with accepted `direct_async` rows and a paired `direct_sync` comparison. | Make hardware-performance or integration-readiness claims only from verifier-accepted prepared-host artifacts. | Do not claim Tonic, RPC, production worker batching, MOVDIR64 serialization, registry pooling, or scheduling policy from M007 alone. |

Rule of thumb: M007 is a planning-ready baseline. It is not execution-ready or claim-ready for worker-runtime work until prepared-host verifier evidence exists.

## Execution gates for the next worker-runtime milestone

A future worker-runtime milestone should not cross from planning into implementation or claims until these gates are explicit in its plan:

1. Baseline gate: rerun the standalone direct proof path and preserve R006 as the input baseline.
2. Prepared-host gate: collect a hardware verifier result on a prepared-host setup; ordinary-host `expected_failure` is planning evidence only.
3. Claim gate: require `verdict=pass claim_eligible=true`, accepted `direct_async` evidence, and paired `direct_sync` evidence before making hardware-performance claims.
4. Boundary gate: list every new claim separately from the direct baseline: worker batching, MOVDIR64/MOVDIR64B, registry, pool, scheduling policy, Tonic, and RPC.
5. Failure gate: preserve `failure_phase`, `launcher_status`, typed runtime failures, and nonzero/malformed artifact rejection semantics.
6. Redaction gate: keep diagnostics limited to status, phases, labels, counts, paths, sizes, and typed errors; never include caller buffer contents or copied data.

## Failure and redaction rules

- `expected_failure` is an accepted classification only when it is explicit, typed, and bounded by fields such as `failure_phase=preflight` and `launcher_status=missing_launcher`.
- A classified ordinary-host failure proves diagnosability, not hardware success.
- `software_direct_async_diagnostic` proves software plumbing and report/verifier shape only; it remains `claim_eligible=false`.
- Missing, empty, malformed, contradictory, or nonzero verifier evidence must remain a failure, not a weak pass.
- Reports, logs, tests, and future worker-runtime artifacts may include phase names, status codes, labels, requested sizes, counts, paths, retry metadata, and typed errors.
- Reports, logs, tests, and future worker-runtime artifacts must not include caller buffer contents, copied data, or before/after data dumps.
