## Canonical API proof

The canonical public construction text for this migration is:

```rust
AsyncMemmoveRequest::new(Bytes, BytesMut)
```

The source buffer is first and is owned as `Bytes`. The destination buffer is second and is caller-supplied as `BytesMut`. The result surface returns explicit destination ownership alongside validation metadata:

```rust
AsyncMemmoveResult {
    destination: BytesMut,
    report: MemmoveValidationReport,
}
```

The validation package must prove the active source-first constructor and result ownership through code, docs, tests, and downstream consumer checks. It must not imply a source-only constructor, a public allocation convenience constructor, a compatibility alias, or a hidden copy-back helper.

## Stale-reference and package guard proof

`idxd-rust/scripts/verify_package_inventory.sh` is the focused stale-reference guard. It checks these categories:

- `package-owner`: rejects stale package-owner references such as old `dsa-ffi`, `iax-ffi`, or binding-name variants outside historical/guard contexts.
- `old-async-api`: rejects removed active async surfaces such as `copy_exact`, `copy_into`, `with_destination_len`, one-argument `AsyncMemmoveRequest::new(...)`, `source_bytes()`, `destination_bytes()`, `result.bytes`, and public `memmove_into` call shapes outside allowed stale-reference tests or guard text.
- `source-only-request-shape`: rejects active source-only request wording or `Vec<u8>` request-shape remnants outside historical/guard contexts.
- `dst-len-side-channel`: rejects destination-length side-channel language when it represents the old async request shape, while allowing validation/report fields that use source and destination lengths to explain errors.
- `handwritten-dsa-abi`: rejects handwritten DSA ABI structs outside the canonical binding layer.

`idxd-rust/scripts/verify_async_memmove_contract.sh` checks the normative report `docs/report/architecture/004.bytes_async_memmove_contract.md` for the bytes constructor, result shape, stale-surface rejection, destination initialization and unsafe-boundary guidance, inline `enqcmd` v1 policy, failure taxonomy, and deferred future-work boundaries.

## Downstream integration proof

The downstream proof seam is `tonic-profile`'s standalone `downstream_async_handle` binary and its wrapper script. It proves a package outside `idxd-rust` can use the public async owner/handle surface from ordinary Tokio code.

The downstream artifact contract includes:

- `proof_seam=downstream_async_handle`
- `consumer_package=tonic-profile`
- `binding_package=idxd-rust`
- `composition=tokio_join`
- `operation_count=2`
- typed lifecycle, worker, validation phase, and validation error fields

This is intentionally separate from `idxd-rust`'s crate-local `await_memmove` proof. It is also separate from `tonic-profile/src/custom_codec.rs`, which remains a synchronous codec seam and must not be treated as the async integration seam.

## Ordinary-host classification rules

The report separates API-migration proof from prepared-host hardware proof.

Acceptable ordinary-host outcomes:

- The verifier exits with status `0`.
- The final verifier line includes `verdict=expected_failure`.
- The failure is explicitly classified with `failure_phase=preflight` or `failure_phase=runtime`.
- Preflight blockers include concrete launcher/device status such as `missing_work_queue`, `missing_devenv`, `missing_launcher`, or `missing_capability`.
- Runtime expected failures include typed lifecycle, worker, or validation fields rather than an unstructured crash.
- The final line preserves raw artifact/stdout/stderr paths where applicable.

Unacceptable outcomes:

- Treating `expected_failure` as prepared-host success.
- Treating direct proof-binary output as final evidence without the wrapper.
- Treating a missing, empty, malformed, or internally contradictory artifact as acceptable.
- Treating timeout, unclassified launcher failure, or non-zero verifier exit as ordinary-host proof.

## Downstream assumptions

Downstream consumers may assume:

- The public async request is built from an owned `Bytes` source and an owned caller-supplied `BytesMut` destination.
- The async result returns explicit destination ownership and a validation report.
- v1 submission is inline per logical request or an equivalent explicit worker-owned boundary documented by active code; callers must not depend on hidden software aggregation.
- Operational proof should use wrapper scripts so ordinary-host blockers are classified truthfully.

Downstream consumers must not assume:

- A hidden allocation convenience constructor exists.
- A public `memmove_into` copy-back helper exists.
- Software batching, aggregation, or flush timers exist.
- MOVDIR64 support exists.
- `tonic-profile/src/custom_codec.rs` is the async proof seam.

## Non-claims

This report does not claim live DSA success until a T02 verifier row records `verdict=pass` from the wrapper on a prepared host.

This report does not claim payload contents. It may describe validation of copied destination prefixes, requested byte counts, schema fields, package identity, proof seam identity, and operation counts, but it must not dump source bytes or destination bytes.

This report does not claim throughput, latency, batching behavior, fairness, queue-depth behavior, or MOVDIR64 behavior.

This report does not claim the future borrowed async zero-copy API is designed or safe.

## Future work explicitly deferred

- Software aggregation policy, including queueing, coalescing, backpressure, fairness, and failure mapping.
- Batching policy and any flush timer behavior.
- MOVDIR64 submission support.
- Borrowed async zero-copy or spare-capacity safe wrapper design, including pointer validity, aliasing, cancellation safety, completion safety, and initialized-length rules.
- Restoring real downstream codec/body-stage accounting where prior shimmed instrumentation is insufficient for seam-cost evidence.
