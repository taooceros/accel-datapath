# Lean bon/snafu refactor evidence

**Milestone:** M010 — Lean `bon`/`snafu` refactor  
**Slice:** S05 — Publish module-scoped convention and proof  
**Status:** Final module-scoped evidence report for S01-S04.  
**Reader:** A maintainer or future agent landing cold on the hardware Rust path after M010.  
**Post-read action:** Decide whether a future constructor or error path should keep, replace, or avoid `bon` and `snafu` without weakening diagnostics, source chains, recovery metadata, or no-payload boundaries.

## Purpose

M010 did not remove `bon` or `snafu` wholesale. It made their use conditional on readability and diagnostic value. This report is the reader-facing evidence for **R019**: hardware Rust abstraction helpers should stay lean, explicit, and module-scoped.

Use this report when deciding whether to add, keep, or remove either crate in the hardware Rust path. The rule is deliberately practical:

- keep `bon` when a method-level builder gives a clearer named construction boundary than positional arguments;
- replace struct-level builder derives when they pollute domain structs or hide normalization;
- keep `snafu` when it preserves source chains and structured diagnostic metadata more clearly than hand-written boilerplate;
- avoid either crate when direct Rust construction or `map_err` is shorter, clearer, and equally diagnostic.

The preservation boundary from **R008** remains active: diagnostics may carry device paths, phases, retry counts, completion status/result bytes, requested lengths, operation names, accelerator names, and source errors, but they must not log or serialize payload bytes, copied source bytes, destination bytes, raw payload dumps, or payload-content examples.

## Claim boundaries

M010 can claim:

- the previously noisy struct-level `bon::Builder` usage in `idxd-rust/src/validation.rs` was replaced with an explicit method-level constructor builder while preserving compatibility;
- sync and async session builders in `idxd-rust/src/lib.rs` and `idxd-rust/src/async_session.rs` were audited and retained only because they name diagnostic-preserving prebuilt-config entrypoints;
- `hw-eval/src/config.rs` now routes production CLI construction through direct `from_args` to `from_parts` normalization while retaining a method-level builder for sparse named construction;
- `hw-eval/src/main.rs` now keeps SNAFU selectors only where they carry structured top-level diagnostics, while localized warning and serialization paths use direct enum construction;
- no-payload diagnostic boundaries remain part of the documented contract.

M010 does **not** claim:

- that all future `bon` or `snafu` usage is approved;
- that all existing Rust error paths should migrate to `snafu`;
- that all constructors should gain builders;
- prepared-host DSA/IAX hardware success from this documentation task;
- any change to public artifact schemas, payload-redaction policy, or hardware behavior.

## Module outcomes

### `idxd-rust/src/validation.rs`

**What changed:** `DsaConfig` no longer derives a struct-level `bon::Builder`. The public builder-shaped compatibility surface remains available through an explicit method-level `#[bon::builder]` constructor that calls the same normalization path as direct constructors. Repeated `MemmoveError` accessor metadata was also centralized behind a private context mapping so helper methods do not duplicate match logic.

**What was retained:** `DsaConfig::builder().build()` remains compatible for callers and tests. `MemmoveError` remains a SNAFU-backed enum for queue-open, request validation, completion, retry, and post-copy validation failures.

**Why this is lean:** The builder belongs at the construction boundary, not on the data struct. That keeps the struct readable while still supporting named default construction. Centralizing `MemmoveError` metadata removes repeated logic without changing the public diagnostic surface.

**Diagnostics and no-payload boundary preserved:** Validation errors still expose kind strings, device path, phase, requested bytes, retry counts, final status, and completion snapshot metadata where applicable. They do not expose source or destination payload bytes.

**Rule for future edits:** If a config type needs normalization or validation, prefer an explicit constructor method with `#[bon::builder]`. Do not reintroduce struct-level `Builder` derive unless the struct is a trivial data holder and the derive is demonstrably clearer than a constructor boundary.

### `idxd-rust/src/lib.rs`

**What changed:** The sync session entrypoint was audited rather than rewritten. Comments now document why the generated `DsaSession::builder().dsa_config(config).open()` path is intentionally retained.

**What was retained:** `DsaSession::open`, `open_with_retries`, `open_config`, and the generated builder entrypoint remain. The builder path is a named prebuilt-config opening path, not a substitute for validation logic.

**Why this is lean:** The generated builder remains useful because it names an already-normalized config handoff. Removing it would churn a tested public convenience surface without simplifying runtime behavior.

**Diagnostics and no-payload boundary preserved:** Queue-open failures still preserve work-queue device path and phase through the underlying validation/session error path. No payload bytes are logged by this entrypoint.

**Rule for future edits:** A generated method builder may stay when it is a thin named adapter over an explicit config object and preserves a public construction contract. It should not become a place for hidden runtime branching or duplicate validation.

### `idxd-rust/src/async_session.rs`

**What changed:** The async session entrypoint and async error surfaces were audited and documented. No functional rewrite was needed because the existing split between explicit request ownership, direct-runtime failures, lifecycle failures, worker-fixture failures, and memmove failures already provides readable diagnostic boundaries.

**What was retained:** `AsyncDsaSession::builder().dsa_config(config).open()` stays as the async prebuilt-config entrypoint. `AsyncMemmoveRequestError`, `AsyncMemmoveError`, and their typed accessors remain SNAFU-backed. The owned request and rejected-buffer recovery APIs remain unchanged.

**Why this is lean:** The async API carries ownership and recovery semantics that are easy to weaken accidentally. SNAFU keeps source-chain display concise while preserving typed categories for lifecycle, worker, direct-runtime, and memmove failures. Manual replacement would add boilerplate without improving readability.

**Diagnostics and no-payload boundary preserved:** Async request construction failures preserve requested byte count, destination length/capacity/writable capacity, and the underlying `MemmoveError`; recoverable failures may return owned buffers to the caller. Display text remains payload-redacted and does not print source or destination bytes.

**Rule for future edits:** Keep SNAFU on async hardware error surfaces when it protects source chains, recovery ownership, direct failure kinds, retry metadata, and completion snapshots. Do not simplify by erasing recoverable requests or replacing typed failure categories with strings.

### `idxd-rust/src/async_direct.rs`

**What changed:** The direct async failure type was reviewed as part of the session-entrypoint slice. No source rewrite was needed.

**What was retained:** `AsyncDirectFailure` remains SNAFU-backed because it compactly preserves direct-runtime failure kind, requested byte count, retry budget/count, and optional completion snapshot metadata.

**Why this is lean:** Direct-runtime failures are not just display strings; they are contract data consumed by tests and future diagnostics. SNAFU keeps the display/source behavior compact while allowing metadata accessors to remain explicit.

**Diagnostics and no-payload boundary preserved:** Direct async failures may expose requested length, retry counts, completion status/result bytes, bytes completed, and fault address. They must not include payload bytes.

**Rule for future edits:** Treat direct-runtime failure metadata as a contract. Replace SNAFU only if the replacement keeps the same source-chain and metadata behavior with less code and no diagnostic loss.

### `hw-eval/src/config.rs`

**What changed:** Production CLI construction now flows through `BenchmarkConfig::from_args` into `BenchmarkConfig::from_parts` directly. `from_parts` remains the normalization boundary for accelerator-specific default devices and parsed sizes.

**What was retained:** `BenchmarkConfig` remains a plain runtime struct. Its method-level `bon` builder around `from_parts` is retained for sparse named construction in tests and diagnostics. `BenchmarkConfigError` remains a small SNAFU enum.

**Why this is lean:** Production code no longer routes through builder ceremony. The builder remains valuable for tests because it avoids nine positional arguments and makes sparse default construction obvious. SNAFU remains valuable for invalid `--sizes` because it preserves concise Display text and a `ParseIntError` source chain.

**Diagnostics and no-payload boundary preserved:** CLI config errors preserve the raw `--sizes` string, offending token, zero/empty-token classification, and parse source. Config parsing happens before hardware queue-open and before benchmark payload generation; no payload bytes are involved.

**Rule for future edits:** Keep `from_args` as the sole Clap-to-runtime adapter and `from_parts` as the normalization boundary. Retain the method-level builder only if it continues to serve sparse named construction; do not move production flow back through builder ceremony.

### `hw-eval/src/main.rs`

**What changed:** Top-level orchestration became leaner by extracting `open_work_queue` and `build_report`. `run()` no longer carries WQ selector payload details inline, and software-only and hardware paths now share one schema-preserving report assembly helper.

**What was retained:** `ConfigSnafu` and `OpenWqSnafu` remain because they preserve structured top-level diagnostics. `PinWarning::Affinity` and `HwEvalError::SerializeReport` use direct enum construction where selector plumbing would be unnecessary ceremony.

**Why this is lean:** The retained SNAFU selectors encode meaningful context: config source-chain preservation, operation name, accelerator, device, launcher/CAP_SYS_RAWIO hint, and `std::io::Error` source metadata. Direct construction is clearer for localized warning-only pin failures and JSON serialization failures.

**Diagnostics and no-payload boundary preserved:** Missing work-queue errors still identify `open_wq`, accelerator, device path, and the launcher/CAP_SYS_RAWIO hint. Invalid pin affinity remains a non-fatal warning. `--sw-only --json` still exits before hardware access and emits JSON metadata without payload bytes.

**Rule for future edits:** Keep SNAFU selectors when they make structured diagnostic contracts shorter and clearer than direct construction. Use direct enum construction for local one-step errors where no source-chain or selector context would be lost.

### `idxd-sys`

**What changed:** Nothing in M010. This crate remains in the final verification boundary because the hardware Rust path depends on its raw contracts, but it was not a `bon`/`snafu` refactor target.

**What was retained:** `idxd-sys` stays a lean raw/std boundary with `std::io::Result`-style OS errors, generated UAPI/layout contracts, explicit unsafe/raw semantics, and no `bon` or `snafu` layer.

**Why this is lean:** The raw ABI, portal, completion, timing, topology, and cache helpers should not gain builders or domain-error wrappers that hide low-level behavior. Keeping `idxd-sys` out of the abstraction cleanup is an intentional convention, not an omission.

**Diagnostics and no-payload boundary preserved:** Raw boundary tests still preserve missing-WQ error kinds and descriptor/completion layout visibility. Payload bytes are not diagnostic material in this boundary.

**Rule for future edits:** Do not force `bon` or `snafu` into `idxd-sys` merely for consistency with higher-level crates. Add abstractions only when they expose, rather than hide, raw contract facts.

## Guardable convention

Future agents can mechanically check this report and the source markers against these phrases and module names:

| Decision point | Keep | Replace or avoid |
|---|---|---|
| Constructor with normalization | Method-level `#[bon::builder]` on the constructor boundary, as in `DsaConfig::builder` or `BenchmarkConfig::from_parts`. | Struct-level `Builder` derive that decorates domain fields and hides validation flow. |
| Prebuilt config entrypoint | Generated method builder that names an already-normalized config handoff, as in sync/async session opening. | Builder path that performs hidden runtime branching or duplicates `open_config`/`from_args` validation. |
| Parse/config error | Small SNAFU enum with source-chain preservation and user-facing Display text, as in `BenchmarkConfigError`. | Manual strings that drop `ParseIntError` source or selector boilerplate that makes direct construction harder to read. |
| Top-level orchestration error | SNAFU selector carrying operation, accelerator, device, hint, and source, as in `OpenWqSnafu`. | Inline selector use throughout `run()` when a focused helper can keep the orchestration readable. |
| Local warning or serialization error | Direct enum construction when the context is one-step and localized. | SNAFU selector plumbing that adds ceremony without improving source chains. |
| Raw ABI/sys boundary | Plain raw/std contracts in `idxd-sys`. | Broad builder or domain-error framework that obscures unsafe layout, mmap, or ENQCMD semantics. |

## Reader-test result

A cold maintainer should be able to apply the M010 convention as follows:

1. If the change is in `idxd-rust/src/validation.rs`, use an explicit constructor boundary and preserve validation metadata; do not add struct-level builder noise.
2. If the change is in `idxd-rust/src/async_session.rs`, protect request ownership and recoverability before simplifying errors.
3. If the change is in `hw-eval/src/config.rs`, keep CLI parsing in `from_args`, normalization in `from_parts`, and parse source chains intact.
4. If the change is in `hw-eval/src/main.rs`, keep structured diagnostics for config and WQ open failures, but prefer direct construction for localized errors.
5. If the change touches raw `idxd-sys` boundaries, do not add `bon` or `snafu` merely for consistency.

The document therefore supports the post-read action: decide when to keep, replace, or avoid `bon` and `snafu` using concrete shipped modules rather than abstract style rules alone.
