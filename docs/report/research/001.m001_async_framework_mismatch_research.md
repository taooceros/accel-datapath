# M001 — Research

**Date:** 2026-04-24

## Summary

M001 should start by proving the measurement harness and integration boundary before attempting any deep accelerator wiring. The strongest existing foundation is already in `accel-rpc/tonic-profile/src/main.rs`: it can run a real unary Tonic path, switch instrumentation on and off, vary buffer policy, toggle gzip compression, and emit stage snapshots for `encode`, `decode`, `compress`, `decompress`, `buffer_reserve`, `body_accum`, and `frame_header`. That makes the first proof obligation much narrower than “build an entire accelerator framework”: prove that this harness can represent a same-repo ordinary path and evolve into a realistic IDXD comparison path at the codec/body boundary identified in the architecture reports.

The codebase and prior reports point to a natural slice order. First, finish the software-only characterization lane so the repo can distinguish fixed async/control costs from payload-shape, copy, and compression costs under realistic Tonic workloads. Second, make the first integration boundary concrete in the harness at the payload path, not at interceptors or generic middleware metadata hooks. Third, only after the software baseline is trustworthy should M001 add a narrow DSA/IAX-backed comparison path and collect end-to-end traces against the same workloads. This ordering minimizes the risk of building hardware plumbing before the repo can tell whether copy/buffer/compression stages are large enough to beat submission/completion overhead.

## Recommendation

Use `accel-rpc/tonic-profile` as the milestone proving ground and define the first integration boundary as a codec/body-path comparison point inside the Tonic request lifecycle. Reuse the existing instrumentation and buffer-policy controls, extend the harness with a FleetBench-inspired protobuf-shape mode, then add a narrow accelerated path only at payload-transform boundaries already called out in the architecture notes: copy/buffer movement and compression-like transforms. Avoid spending M001 on transport, scheduler, or interceptor abstractions because the repo’s own analysis already shows those are mostly control-plane surfaces rather than accelerator-friendly data-path hooks.

For planning, treat M001 as four slices: (1) harden the software baseline and workload realism in `tonic-profile`; (2) add lower-distortion evidence collection and split-endpoint traces; (3) prototype a same-repo accelerated path through one chosen codec/body boundary using the existing FFI crates as the seed; (4) run the curated workload set through ordinary and accelerated paths and write the narrow claim analysis. This produces a working measurable prototype while keeping the claim explicitly scoped to IDXD + Tonic.

## Implementation Landscape

### Key Files

- `accel-rpc/tonic-profile/src/main.rs` — current measurement harness; already supports unary selftest/client/server modes, instrumentation on/off, buffer-policy controls, compression toggles, JSON output, and stage snapshots. This is the primary place to extend workload realism, split-endpoint measurement, and the ordinary-vs-accelerated comparison entry point.
- `accel-rpc/tonic-profile/Cargo.toml` — harness dependencies; currently depends on the local tonic fork with `gzip` enabled plus `hdrhistogram`, `serde`, and `tokio-stream`. Any new proto-shape mode or accelerator-backed comparison path will surface here first.
- `accel-rpc/async-bench/benches/async_overhead.rs` — currently only benchmarks `tokio::spawn` + join. This is the obvious place to expand runtime/control-path microbenchmarks so M001 can compare async overhead against accelerator submit/completion overhead instead of guessing.
- `accel-rpc/dsa-ffi/src/lib.rs` — placeholder FFI bridge for DSA copy/CRC-style operations. It documents the intended contract but does not yet provide the real Rust-facing path M001 needs.
- `accel-rpc/iax-ffi/src/lib.rs` — placeholder IAX bridge for compression/decompression. Useful as a milestone seed, but still only a contract stub.
- `accel-rpc/accel-codec/src/lib.rs` — placeholder custom codec crate intended for pooled/prefaulted buffers and accelerator-friendly buffer management. This is a better long-term home for codec-boundary experiments than modifying unrelated transport code.
- `accel-rpc/accel-middleware/src/lib.rs` — placeholder Tower middleware crate for CRC/compression transforms. Useful for transparent service-wide wrapping, but the current repo research says payload-path proof should start at codec/body hooks, not middleware metadata layers.
- `docs/report/architecture/002.tonic_component_analysis.md` — architecture map showing that compression/decompression and copy+CRC are the strongest offload candidates, while Tokio/h2/Tower orchestration are poor offload targets.
- `docs/report/architecture/003.tonic_interception_points.md` — identifies codec/body boundaries as the main payload hook and explicitly warns that tonic interceptors are metadata-only.
- `docs/report/benchmarking/012.tonic_characterization_refinement_results.md` — current strongest evidence for regime changes, instrumentation distortion, and buffer-policy sensitivity; this should directly shape slice ordering.
- `docs/report/benchmarking/013.fleetbench_rpc_characterization_intake.md` — recommends the missing workload-realism lane: protobuf-shape modes, request/response asymmetry, closed-loop outstanding RPCs, and paired CPU/decomposition runs.
- `hw-eval/src/main.rs` — raw DSA/IAX lower-bound benchmark harness. It is not the Tonic integration point, but it is the repo-local hardware-floor reference for later offload crossover interpretation.

### Build Order

1. **Prove the software baseline can answer the right question first.** Extend `accel-rpc/tonic-profile/src/main.rs` with the missing realistic workload dimensions from report `013`: protobuf-shaped request/response modes, asymmetry, and clearer load-shaping metadata. This retires the risk that the current bytes-only harness overstates copy/buffer costs and understates schema-traversal costs.
2. **Reduce evidence distortion before stronger conclusions.** Improve stage accounting in `tonic-profile` so instrumentation can be split by endpoint or accumulated with lower overhead than the current hot-path `Instant::now` + shared atomics. This retires the already-documented risk from report `012` that instrumentation changes throughput too much outside the tiny regime.
3. **Expand async/control-path baselines in `async-bench`.** Build the missing microbenchmarks for notification, polling, and completion costs so the later accelerated path can be compared against real software control overhead rather than only end-to-end throughput.
4. **Define one narrow accelerated boundary and wire it through the harness.** Use `dsa-ffi` or `iax-ffi` only for the first measured codec/body-path comparison, not for a broad abstraction. The first boundary should reuse the architecture reports’ recommendation: payload transform/copy path, not interceptor, transport, or scheduler logic.
5. **Run the curated workload set through ordinary and accelerated paths with the same harness and artifact format.** Only after steps 1–4 should M001 generate the comparative traces and measurements that support, narrow, or falsify the mismatch claim.
6. **Write the narrow analysis artifact from measured evidence.** The writeup should explicitly connect Tonic stage buckets, async control costs, and hardware-floor comparisons, rather than claiming a generalized accelerator framework result.

### Verification Approach

- `cd accel-rpc && cargo build --release -p tonic-profile` — confirm the measurement harness builds after each major extension.
- `cd accel-rpc && cargo run --release -p tonic-profile -- --mode selftest --rpc unary --payload-size 256 --payload-kind structured --compression off --concurrency 1 --runtime single --instrumentation on` — verify the current ordinary path still runs end to end and emits stage JSON.
- `cd accel-rpc && cargo run --release -p tonic-profile -- --mode selftest ... --instrumentation off` on matched regimes — pair throughput-baseline runs with attribution runs to confirm instrumentation distortion stays explicit.
- `cd accel-rpc && cargo bench -p async-bench` — verify runtime/control-path microbenchmarks and capture control-plane floor data for later comparison.
- `cd hw-eval && cargo build --release` and selected repo-local hardware-floor runs in the intended environment — use only as lower-bound reference, not as milestone acceptance by themselves.
- For milestone acceptance, require the same `tonic-profile` workflow to run curated workloads through both ordinary and accelerated paths and leave behind JSON/traces plus a comparison table and written interpretation.

## Constraints

- The workspace explicitly excludes `accel-rpc/tonic` from normal member edits in `accel-rpc/Cargo.toml`; the fork is present locally, but milestone work should prefer harness-side and crate-side integration unless a fork change is truly required.
- `tonic-profile` currently supports only unary mode and a simple echo-style payload contract. That keeps scope tight, but it also means workload realism must be added intentionally rather than assumed.
- Reports `002` and `003` constrain the first accelerator boundary to codec/body-path work. Interceptors and most transport/runtime code are intentionally poor fits for M001’s first offload attempt.
- Report `012` shows current internal timers are diagnostic-only in larger or high-concurrency regimes. Any planning that depends on detailed stage attribution must include lower-distortion measurement work first.
- The repo root runbook and milestone context require all claims to stay narrow to IDXD + Tonic. M001 should not drift into a general accelerator-runtime framework project.

## Common Pitfalls

- **Treating tonic interceptors as payload hooks** — `docs/report/architecture/003.tonic_interception_points.md` is explicit that interceptors are metadata-only. Start at codec/body hooks instead.
- **Trusting instrumentation-on throughput numbers as baseline evidence** — report `012` already shows severe distortion in medium/large regimes. Pair attribution runs with instrumentation-off runs.
- **Over-reading `buffer_reserve` timers as allocator truth** — the current counters include reserve-call behavior and pre-sizing effects, not pure reallocation cost. Use end-to-end deltas plus `perf` or flamegraphs before drawing offload conclusions.
- **Building a broad abstraction before proving a narrow boundary** — `dsa-ffi`, `iax-ffi`, `accel-codec`, and `accel-middleware` are mostly intent stubs today. M001 should wire one real path first.
- **Using bytes-only workloads as if they covered realistic protobuf shape** — report `013` argues that realistic schema traversal can change the offload-readiness story. Add proto-shaped workloads before making strong claims.

## Open Risks

- The repo currently lacks the actual same-repo accelerated Tonic path; the FFI and codec/middleware crates are still mostly placeholders, so the first integration slice carries real implementation risk.
- Lower-overhead split-endpoint measurement may itself change harness structure enough to affect comparability with existing report `012` results.
- Realistic protobuf-shaped workloads may reduce the apparent importance of copy/buffer movement, which could narrow or even falsify the expected DSA win story.
- Hardware-floor comparisons from `hw-eval` may not transfer cleanly to the Tonic path once submit/completion, wakeup, and buffer-management costs are included.
