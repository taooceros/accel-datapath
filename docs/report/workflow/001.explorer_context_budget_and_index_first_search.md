# Explorer context budget and index-first search

**Date**: 2026-04-01
**Scope**: root `AGENTS.md`, orchestrator workflow, and `@explorer` task design

## Executive Summary

`@explorer` is overrunning context not because repo discovery is inherently too large, but because the current guidance pushes it toward a single broad task that mixes:

- multi-domain search,
- selective reading,
- cross-language integration,
- and final synthesis.

The strongest contributing instructions are:

- root `AGENTS.md`: use `@explorer` for broad discovery and batch independent searches in one request,
- orchestrator workflow: split and parallelize broad discovery,
- `@explorer` role guidance: use it for uncertain scope and summarized repo maps.

What is missing is an explicit context-budget rule. There is no instruction to:

- separate discovery from synthesis,
- cap files per area,
- or require index-first pruning before reading.

The recommended fix is to make `@explorer` **index-first and staged**:

1. discover candidates using KB/search/index signals,
2. prune to a small shortlist,
3. read only the files needed for the answer,
4. synthesize after pruning or in a separate follow-up task.

## Observed failure mode

For a prompt like “map launcher, C++, Rust, benchmarking, docs, and integration paths across the repo,” the current guidance naturally leads to:

1. choose `@explorer` because the scope is broad and uncertain,
2. batch multiple domains because the policy recommends concurrent search,
3. read a growing set of “maybe relevant” files,
4. hold cross-domain facts until the final synthesis,
5. risk context pressure before enough summarization happens.

This is a workflow-shape problem, not only an implementation problem.

## Root cause in current guidance

### Root `AGENTS.md`

The following rules bias toward monolithic explorer jobs:

- “For broad codebase discovery, prefer parallel search or `@explorer`.”
- “When delegating to `@explorer`, batch independent searches in one request and ask it to run them concurrently.”

These are good latency defaults, but without a bounding rule they also encourage one wide search over many domains.

### Orchestrator prompt

The orchestrator prompt further reinforces this shape by encouraging:

- delegation for broad discovery,
- split-and-parallelize behavior,
- summarized maps rather than direct known-path reads.

Again, what is absent is a counterweight such as “candidate discovery only” or “respect a context budget before synthesis.”

## Recommended changes

### 1. Add an index-first explorer rule

Before reading files, `@explorer` should prefer:

- KB indexes and prior reports,
- file/path metadata,
- `glob`/`grep` hit lists,
- AST/LSP symbol discovery.

Explorer should return a compact candidate map first when scope spans multiple areas.

### 2. Separate discovery from synthesis

Broad tasks should default to:

- **phase 1**: candidate discovery only,
- **phase 2**: shortlist confirmation,
- **phase 3**: selective reads and synthesis.

This prevents the subagent from carrying every candidate file into final reasoning.

### 3. Add explicit bounds

Recommended policy knobs:

- max files inspected per area before returning for pruning,
- max domains per explorer invocation,
- no end-to-end synthesis until relevant files are shortlisted,
- prefer multiple bounded explorer jobs when launcher/code/docs all matter.

### 4. Move integration synthesis to orchestration when possible

If several bounded explorer jobs each return compact candidate sets, the orchestrator can merge them with much lower risk than asking one explorer session to retain all intermediate evidence.

## Concrete policy edits to make

Recommended root `AGENTS.md` additions:

1. Under **TOOL ROUTING** or **PARALLELISM**:
   - prefer index-first discovery for broad search tasks
   - use `@explorer` for candidate discovery before deep reading
2. Under **PARALLELISM**:
   - split broad cross-domain discovery into multiple bounded explorer jobs when feasible
3. Under **CONVENTIONS** or a new subsection:
   - separate discovery, pruning, and synthesis
   - cap files per area unless the user explicitly asks for exhaustive search
4. Under a new **ANTI-PATTERNS** section:
   - avoid combining repo-wide discovery, detailed reading, and final synthesis in one explorer task

## Expected outcome

If these changes are made, future broad repo-mapping work should shift from “search widely, read widely, synthesize late” to “search cheaply, shortlist early, read narrowly, synthesize after pruning.”

That should reduce context pressure, improve answer reliability, and still preserve the speed advantage of delegated search.
