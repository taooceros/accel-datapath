# Concurrent `current.md` cutover and canonical thread-file model

**Date**: 2026-04-14
**Scope**: root `AGENTS.md`, root `current.md`, `.agents/templates/thread_state.md`, `.agents/state/threads/`, and workflow guidance that previously treated `current.md` as the active run ledger

## Summary

This cutover retires the old single-active-thread `current.md` ledger and replaces it with a concurrent-agent model.

The active workflow now has these rules:

- `current.md` is a dashboard/index for live work, not the mutable source of truth
- each live thread has exactly one canonical thread file under `.agents/state/threads/`
- one live thread may be owned by only one agent session at a time, and one agent may hold only one live thread at a time
- `owner_session_id` records the current lease owner, while `previous_owner_session_id` preserves the most recent prior owner on resume or takeover
- a fixed 4-hour lease controls stale-thread takeover and resumed-session conflict handling

This was a clean cutover. The repo no longer presents the old ledger and the new concurrent model as co-equal options.

## What changed

### 1. `current.md` became a routing surface

`current.md` now exists as an index and dashboard view to help a future main-agent session find the right live thread quickly. Its dashboard entries may mirror the metadata needed for that match and for quick live-state visibility:

- `thread_id`
- `index_label`
- short summary text
- `match_hints`
- `owner_agent`
- `owner_session_id`
- status and lease timing
- canonical thread-file path
- next action

Those mirrored dashboard fields are still non-authoritative. They keep the live dashboard compact and useful while mutable per-thread state remains owned by the canonical thread file.

### 2. Canonical thread files became the authority

Each live thread now has one canonical thread file under `.agents/state/threads/`. That file owns mutable state such as:

- status
- ownership
- lease timestamps
- handoff fields
- blockers
- next actions
- related artifacts

If `current.md` and the thread file disagree, the canonical thread file wins. The repair step is to refresh the dashboard, not to treat mirrored dashboard fields in `current.md` as a second mutable source of truth.

### 3. Ownership is explicit for concurrent main-agent sessions

The concurrent model is not shared editing inside one live thread. Instead, it is one-thread-per-agent ownership across multiple live threads.

The core ownership fields are:

- `owner_agent`
- `owner_session_id`
- `previous_owner_session_id`
- `lease_acquired_at`
- `lease_expires_at`
- `last_updated`

On resume or takeover, the claiming session updates the canonical thread file first, moves the prior session id into `previous_owner_session_id`, writes the new `owner_session_id`, and refreshes lease timestamps before the dashboard entry is updated.

## Self-claim and auto-resume flow

Concurrent main-agent sessions do not require the user to provide a `thread_id` manually.

On session start, the agent reads `current.md` first and matches the incoming request against dashboard metadata:

- `index_label`
- `summary`
- `match_hints`
- `related_artifacts`

From there, the session follows one path:

1. If exactly one plausible live match exists and `resume_allowed` is true, resume that thread by claiming its canonical thread file.
2. If there is no plausible live match, create a new canonical thread file and then add or refresh the dashboard entry.
3. If multiple plausible live matches exist, ask one disambiguation question.

Matching on `owner_agent` alone is not enough. The request-to-dashboard match decides whether a session resumes an existing thread or creates a new one.

## 4-hour lease, takeover, and resumed-session behavior

The lease model is fixed and simple:

- `last_updated` is the most recent owner update time
- the default 4-hour lease expires 4 hours after `last_updated`
- takeover is allowed only after lease expiry or after an explicit handoff recorded with `handoff_to` and `handoff_reason`

This prevents two live sessions from silently believing they both own the same thread.

If an older session resumes after another session already claimed the thread, it must reread `current.md` and the canonical thread file. When the file shows a different active `owner_session_id`, the older session no longer owns that thread. It may only proceed through a valid stale-lease takeover or by self-claiming a different matching thread.

## Update ordering and dashboard hygiene

The cutover also fixed the write order:

1. update the canonical thread file first
2. refresh the `current.md` dashboard entry second

That order matters because the dashboard is now an index view of live state rather than the state container itself.

The older archival intent still survives in the new model. Completed or archived threads leave the live dashboard once their outcome has been captured in durable artifacts such as reports, remarks, or completed plan notes.

## Why the clean cutover happened

The old run ledger was tuned for a single active thread, offset-based navigation, and cheap in-file updates. That worked while one session owned the whole mutable picture, but it broke down once the repo needed concurrent main-agent sessions with self-claim and auto-resume.

The main problems were:

- `current.md` mixed routing metadata with mutable thread detail
- a single active section did not fit multiple concurrent live threads
- offset navigation optimized file jumping, but not ownership, takeover, or stale-session recovery
- resumed sessions had no canonical per-thread lease record to inspect

The clean cutover avoids split authority. There is now one active model:

- dashboard/index in `current.md`, which may mirror selected live-thread metadata for routing and visibility without becoming authoritative
- canonical mutable state in `.agents/state/threads/`
- durable historical detail in reports, remarks, and completed plan notes
