# Current.md run-ledger policy rollout

**Date**: 2026-04-01
**Scope**: root `AGENTS.md`, root `current.md`, and `docs/plan/2026-04-01/10-15.*`

> **Superseded note**
>
> This report documents the historical single-active-thread `current.md` ledger that was active on 2026-04-01.
> It is no longer the current policy.
> The active concurrent-agent replacement is documented in `docs/report/workflow/004.current_md_concurrent_cutover.md`.

## Summary

This change turns `current.md` from a loose status note into an explicit run ledger for the repo.

The new workflow makes three things first-class:

- `current.md` is read before other repo context on resume
- one thread is explicitly marked active at a time
- progress is tracked inside each thread with active, paused, and completed items

The current format is intentionally optimized for agent navigation rather than prose readability. The top-of-file TOC now exposes only:

- section status
- section name
- starting offset

That gives agents a cheap way to jump to the right section for reads and edits without scanning the whole file.

## Changes made

### 1. Added a dedicated `current.md` workflow section in `AGENTS.md`

Root `AGENTS.md` now treats `current.md` as an active run ledger instead of a passive resume note.

The policy now requires:

- reading `current.md` first on resume
- keeping an explicit overall goal in the active section
- maintaining exactly one active thread at a time
- updating `current.md` before non-trivial work continues
- marking individual items complete as soon as they finish
- recording concrete artifact paths alongside completed items when applicable

### 2. Restructured `current.md` around thread sections

`current.md` now uses a thread-oriented layout:

- a short TOC at the top
- one active section
- paused sections for interrupted work
- per-section active, paused, and completed items
- a short resume note

This makes short focus switches cheaper to represent. The old thread does not need to be rewritten or merged into a global status dump; it can simply move to paused while a new section becomes active.

### 3. Reduced TOC content to section metadata only

Earlier iterations used anchor links and then a global completed section. Both were removed.

The final TOC format is a Markdown task-list entry such as:

- `- [x] Active — workflow and run-tracking policy (offset: 9)`
- `- [ ] Paused — tonic-only profiling and experiment design (offset: 29)`

This keeps the TOC compact and makes the offset the primary navigation primitive.

### 4. Moved completion tracking into each section

Completed items now live inside their own thread section rather than in one shared completed list.

That preserves local context:

- what the thread was trying to do
- what is still active or paused
- what was already finished
- which artifacts came out of that thread

This is a better fit for interrupted multi-session work than a global completion bucket.

## Why this structure

The repo already depends on resumable, handoff-friendly workflows. A flat `current.md` note was enough for a single narrow thread, but it was weak in three cases:

1. when focus switched for a few sessions
2. when multiple sub-steps finished at different times
3. when an agent needed a fast way to locate the correct section to edit

The new structure addresses those directly:

- the TOC gives a low-cost index
- the active-section rule prevents ambiguous “current goals”
- per-section completion tracking keeps history local to the thread
