# M008 cleanup conventions and integrated proof

**Milestone:** M008 — Hardware Rust `bon` / `snafu` cleanup  
**Slice:** S06 — Document cleanup conventions and integrated proof  
**Status:** Final conventions report; ordinary-host proof commands are executable through the final proof collector  
**Reader:** A maintainer or future agent deciding where hardware Rust cleanup may continue after M008.  
**Post-read action:** Make a scoped `bon` or `snafu` cleanup decision without breaking proof artifacts, ordinary-host verification, prepared-host claims, or the no-payload diagnostic contract.

## Purpose

M008 was a cleanup milestone, not a hardware-semantics rewrite. Its durable outcome is a set of conventions for where builders and structured errors help the Rust DSA/IAX stack, where they should not be forced, and which evidence must remain stable for future maintainers.

This report consolidates the actual S02-S05 outcome:

- `MemmoveValidationConfig` and the normalized `hw-eval` benchmark config are the approved builder-backed surfaces.
- `idxd-rust` sync, async, and direct error types plus bounded `hw-eval` entrypoint failures are the approved SNAFU-backed surfaces.
- `idxd-sys`, proof-binary private CLI structs, request construction, benchmark hot loops, report records, and broad downstream cleanup remain explicit non-targets.
- Ordinary-host checks can prove contracts, schemas, classifications, and redaction. Real DSA/IAX throughput or hardware-success claims require a prepared-host run with the documented launcher and devices.

## `bon` conventions tied to R016

R016 is satisfied for M008 by applying builders only to hardware Rust configuration surfaces where named defaults and explicit setters reduce ambiguity.

Use a `bon` builder when all of these are true:

1. The type is a stable configuration boundary, not a raw hardware command.
2. Defaults matter and must be visible to tests.
3. Multiple optional knobs would otherwise become positional arguments.
4. Normalization or validation belongs at construction time.
5. Existing callers can keep compatibility wrappers or migrate deliberately.

The approved examples are:

- `MemmoveValidationConfig`: builder defaults, explicit retry setter, device-path normalization, empty-device validation, and compatibility constructors all remain tested.
- `hw-eval` internal benchmark config: builder-backed normalization covers accelerator, device selection, parsed sizes, iteration count, concurrency, software-only mode, pinning, cold-cache flag, and JSON mode without changing CLI flags.

Do not add builders just to use the library. Builders are not required for:

- `AsyncMemmoveRequest::new`, because explicit source and destination ownership/capacity are part of the safety and recovery contract.
- `idxd-sys` descriptor/completion helpers, because they are raw ABI writers with unsafe preconditions.
- Proof-binary private CLI structs, because verifier stability matters more than private construction ceremony.
- Benchmark report structs, because serialized field names are artifact contracts.
- Hot-loop parameter bundles, unless a future slice proves the change does not perturb benchmark behavior.

## `snafu` conventions tied to R017

R017 is satisfied for M008 by using SNAFU where structured context, source chaining, and recovery classification improve maintainability without hiding raw hardware facts.

Use SNAFU when the caller has meaningful context to attach:

- validation phase, status, retry count, requested size, and device path;
- async lifecycle, worker, direct, or validation classification;
- request-recovery state and safe buffer return;
- direct retry budget/count and completion snapshot metadata;
- `hw-eval` CLI token, full size list, accelerator, device path, operation, CPU core, or report serialization context.

Retain raw/std or simple status values when that is the clearer boundary:

- `idxd-sys::WqPortal::open` returns the OS open/mmap failure directly.
- `idxd-sys` affinity and topology helpers remain lightweight raw helpers.
- ENQCMD accepted/rejected status and completion status bytes remain hardware signals, not domain errors.
- Unsafe descriptor/submission helpers keep their safety contract visible.
- Hot benchmark loops may continue to expose status/size/concurrency metadata without converting every operation into a fallible abstraction.

The practical rule is: attach context at the layer that knows the operation intent, and preserve the original source or raw status when the lower layer is the hardware boundary.

## Do-not-force boundaries

Future cleanup should treat these as intentional exclusions, not unfinished M008 work:

- no broad `idxd-sys` builder or domain-error migration;
- no builder around raw DSA/IAX descriptors, completions, portals, or unsafe submit helpers;
- no builder for `AsyncMemmoveRequest` or allocation-convenience async memmove constructor;
- no forced migration of proof-binary private CLI structs;
- no report-struct builder that changes JSON names or graphing assumptions;
- no benchmark-matrix redesign under the name of cleanup;
- no broad downstream `accel-rpc` or Tonic cleanup beyond compatibility repairs;
- no change to payload redaction rules, stdout/artifact equality, expected-failure classification, or prepared-host claim criteria without a versioned contract update.

## Artifact and verifier contract

The proof surface is contract-first. Producers, verifiers, and downstream consumers may evolve only when their field changes are explicit and tested together.

Stable artifact families:

| Artifact family | Required stability |
|---|---|
| Sync validation proof | Keep success/failure verdict, device path, requested size, retry count, phase, status/error kind, message, stdout/artifact parity, and expected-failure classification. |
| Async validation proof | Keep sync fields plus lifecycle, worker, direct, retry, completion, and validation classification fields. |
| Tokio benchmark proof | Keep schema version, backend, suite, runtime, requested size, iteration/concurrency, per-row metrics, failure classes, direct/validation metadata, claim eligibility, and stdout/artifact parity. |
| Downstream async proof | Keep consumer/binding identity, composition, operation count, device path, requested size, phase, lifecycle/worker/direct/validation classifications, message, and verifier-friendly final lines. |
| `hw-eval` benchmark report | Keep `metadata`, `latency`, and `throughput` top-level objects and the existing benchmark names, stat fields, throughput fields, device/topology metadata, and software/hardware mode semantics. |

The no-payload invariant remains mandatory. Diagnostics may include metadata such as phase, status, retry count, device path, requested size, token, launcher status, output directory, artifact path, completion result, bytes completed, fault address, CPU core, accelerator, operation, and OS source. They must not include copied source or destination contents, buffer dumps, or fields whose purpose is to serialize those contents.

## Ordinary-host proof commands

Ordinary-host proof is valuable because it exercises the contract without requiring privileged DSA/IAX devices. It can prove buildability, schema stability, verifier behavior, expected-failure classification, redaction, and software-only report compatibility. It cannot prove that a real hardware submission succeeded.

The final ordinary-host collector is:

```bash
bash idxd-rust/scripts/collect_m008_final_proof.sh
bash idxd-rust/scripts/check_m008_final_proof_manifest.sh target/m008-final-proof/latest/manifest.json
```

The collector is expected to record these classes of checks:

| Check class | Ordinary-host claim |
|---|---|
| Documentation guard | The final report exists and mentions R016, R017, ordinary-host limits, and prepared-host expectations. |
| `idxd-rust` contract tests | Builder defaults, SNAFU-backed errors, async recovery, verifier schema, and no-content redaction remain host-free testable. |
| `idxd-rust` bin check | Proof binaries still compile against the builder/error APIs. |
| `hw-eval` contract tests | Config parsing, software-only JSON, missing-device diagnostics, pin warnings, and raw `idxd-sys` boundary tests remain host-free testable. |
| Software benchmark verifier | Software backend proof can pass while remaining non-claim-eligible for hardware performance. |
| Hardware proof verifiers on ordinary host | Missing launcher/device states are acceptable only when classified as expected failures with phase and launcher status metadata. |
| Downstream proof checks | `tonic-profile` compatibility tests preserve downstream artifact semantics and no-content redaction. |

The manifest checker rejects hard failures, missing records, malformed manifests, missing logs, forbidden content-dump labels, and inconsistent summary counts.

## Prepared-host expectations

A prepared-host run is required for any claim that DSA/IAX hardware completed work successfully or for any performance conclusion about the accelerator path.

A prepared host must provide:

- configured DSA/IAX work queues at the documented device paths;
- the launcher path expected by verifier scripts, including the privileges needed for work-queue access;
- CPU/device affinity behavior compatible with the benchmark harness;
- enough runtime permissions for proof binaries and `hw-eval` to open devices and emit artifacts;
- the same no-content redaction and artifact validation used on ordinary hosts.

On a prepared-host run, the same final proof collector should turn hardware verifier records from expected failures into claim-eligible hardware passes. Software-only benchmark passes remain useful sanity checks, but they are never hardware-performance proof.

## Future-work guardrails

Before adding future cleanup in this area, answer these questions:

1. Is this a stable config boundary, or am I adding a builder to raw hardware plumbing?
2. Does this error layer know more context than the lower-level source, or would SNAFU hide the raw cause?
3. Will any JSON field, verifier final line, classification string, exit-code split, or stdout/artifact equality rule change?
4. Can the ordinary-host contract test catch the change without DSA/IAX hardware?
5. If a hardware claim is being made, did the prepared-host proof actually produce a claim-eligible hardware pass?
6. Did the change preserve the no-payload invariant in displays, stderr, JSON, logs, reports, and manifests?

If the answer is uncertain, treat the change as a new migration slice rather than incidental cleanup.

## Reader-test result

A cold reader can now route future changes: apply builders only at normalized config surfaces, use SNAFU only where caller context improves diagnostics, keep raw hardware boundaries raw, preserve proof schemas and redaction, use ordinary-host verification for contracts, and require prepared-host evidence for hardware-success or performance claims.
