**Status:** Draft inventory in progress; `idxd-rust` completed by S01/T02; `idxd-sys` and `hw-eval` boundaries completed by S01/T03; artifact and downstream compatibility obligations mapped by S01/T04
**Milestone:** M008 — Hardware Rust `bon`/`snafu` cleanup
**Scope:** `idxd-rust`, `idxd-sys`, and `hw-eval`
**Primary requirements:** R008, R016, R017

## Purpose

This report is the durable migration contract for M008. It will inventory the hardware-side Rust cleanup targets before implementation, classify where `bon` builders and `snafu` errors should be applied, and preserve the diagnostic, artifact, and downstream compatibility obligations that later slices must not rediscover or accidentally break.

The cleanup is allowed to make aggressive API and test churn when that produces a cleaner hardware Rust surface. That churn is still bounded by the requirement to preserve truthful hardware diagnostics, expected-failure classification, artifact semantics, and the no-payload-logging contract.

## Inventory scope

### In scope

- `idxd-rust` public library APIs, proof binaries, verifier scripts, and host-free contract tests.
- `idxd-sys` raw hardware boundary, but only where added context materially improves diagnostics without hiding the raw ABI/MMIO source.
- `hw-eval` benchmark/evaluation harness configuration, CLI parsing, report generation, JSON output, and hardware-open/pin/serialize failure paths.
- Narrow downstream compatibility edits or checks for consumers affected by hardware crate public API changes.

### Out of scope / do-not-force boundaries

- Vendored `accel-rpc/tonic/**` cleanup.
- Broad `accel-rpc` application cleanup outside narrow compatibility fixes.
- `codemogger`, unrelated tooling, and C++ `dsa-stdexec` cleanup.
- Rewriting hardware behavior, completion semantics, batching policy, MOVDIR64 support, or Tonic integration.
- Adding abstractions solely to use `bon` or `snafu` where the raw boundary is clearer.

## Decision vocabulary

Use these classifications consistently in the inventory tables below.

### Config candidates

- `bon now` — add a `bon` builder in M008 because named construction/defaults materially improve the API.
- `defer` — a builder may be useful, but the surface is too unstable or too large for this milestone.
- `do not force` — keep existing construction because a builder would add ceremony or obscure a raw/simple boundary.

### Error candidates

- `snafu now` — migrate to `snafu` because structured context/source chaining improves diagnostics.
- `retain raw/std` — keep `std::io::Result`, raw status, or another low-level shape because that is the clearest boundary.
- `defer` — revisit after higher-priority migrations land.
- `do not force` — do not add a domain error layer where it would obscure hardware behavior or expand scope.

### `idxd-rust` request-buffer recovery expectations

- `AsyncMemmoveRequest::new` must remain synchronous and must reject zero-length requests or insufficient destination spare capacity before runtime dispatch.
- `AsyncMemmoveRequestError::into_parts` must return the underlying `MemmoveError`, original `Bytes` source, and original `BytesMut` destination without logging or formatting payload bytes.
- `AsyncMemmoveError::into_request` must recover the owned request for owner-shutdown, request-channel, direct-registration/backpressure/monitor failures, and memmove failures when the request is still safe to return.
- Accepted direct async operations are not cancelled by dropping the awaiting receiver; completion ownership stays in the runtime until completion or cleanup.
- Retry-continuation backpressure must preserve a completion snapshot and recover buffers; post-copy mismatch after destination length advancement must not expose unsafe extra bytes and may return no request when recovery is no longer safe.

Guarding tests: `idxd-rust/tests/async_memmove_contract.rs::{rejects_zero_length_owned_requests_before_runtime_dispatch,rejects_destination_size_mismatch_before_runtime_dispatch,handle_use_after_explicit_shutdown_is_a_lifecycle_error,backpressure_exhaustion_reports_retry_budget_without_payload_bytes,dropped_direct_receiver_does_not_remove_accepted_operation_before_completion,retry_continuation_backpressure_reports_snapshot_and_recovers_buffers,post_copy_mismatch_preserves_destination_length_safety}` and `idxd-rust/tests/tokio_handle_contract.rs::explicit_owner_shutdown_refuses_new_submissions_without_losing_accepted_work`.

## `idxd-sys` raw-boundary decisions completed in S01/T03

`idxd-sys` remains the canonical raw DSA/IAX UAPI and MMIO boundary. Later M008 implementation slices should not add `bon` builders or broad `snafu` error enums here just to satisfy library adoption requirements. The crate's value is transparent mapping of bindgen-generated structures, alignment wrappers, unsafe submission preconditions, raw completion status, and OS `io::Error` sources.

Bounded context rule: if a downstream caller needs richer diagnostics, add context at the caller boundary that knows the operation intent (`idxd-rust` validation, async direct submission, `hw-eval` benchmark open, CPU pinning), while preserving the original `std::io::Error` as the source.

Guarding checks:

- `cargo test -p idxd-sys` for bindgen layout and helper-write contracts.
- `cargo check -p hw-eval` for the `hw-eval` façade and local portal use.
- `cargo test -p idxd-rust --test memmove_contract --test async_memmove_contract --no-run` when `idxd-sys` signatures or descriptor helpers change.

## `hw-eval` cleanup boundaries completed in S01/T03
