# M008 migration contract: hardware Rust `bon` / `snafu` cleanup

**Status:** Draft skeleton for S01 inventory
**Milestone:** M008 — Hardware Rust `bon`/`snafu` cleanup
**Scope:** `idxd-rust`, `idxd-sys`, and `hw-eval`
**Primary requirements:** R008, R016, R017

## Purpose

This report is the durable migration contract for M008. It will inventory the hardware-side Rust cleanup targets before implementation, classify where `bon` builders and `snafu` errors should be applied, and preserve the diagnostic, artifact, and downstream compatibility obligations that later slices must not rediscover or accidentally break.

The cleanup is allowed to make aggressive API and test churn when that produces a cleaner hardware Rust surface. That churn is still bounded by the requirement to preserve truthful hardware diagnostics, expected-failure classification, artifact semantics, and the no-payload-logging contract.

## Inventory scope

### In scope

- `idxd-rust` public library APIs, proof binaries, verifier scripts, and host-free contract tests.
- `idxd-sys` raw hardware boundary, but only where added context materially improves diagnostics without hiding the raw ABI/MMIO source.
- `hw-eval` benchmark/evaluation harness configuration, CLI parsing, report generation, JSON output, and hardware-open/pin/serialize failure paths.
- Narrow downstream compatibility edits or checks for consumers affected by hardware crate public API changes.

### Out of scope / do-not-force boundaries

- Vendored `accel-rpc/tonic/**` cleanup.
- Broad `accel-rpc` application cleanup outside narrow compatibility fixes.
- `codemogger`, unrelated tooling, and C++ `dsa-stdexec` cleanup.
- Rewriting hardware behavior, completion semantics, batching policy, MOVDIR64 support, or Tonic integration.
- Adding abstractions solely to use `bon` or `snafu` where the raw boundary is clearer.

## Decision vocabulary

Use these classifications consistently in the inventory tables below.

### Config candidates

- `bon now` — add a `bon` builder in M008 because named construction/defaults materially improve the API.
- `defer` — a builder may be useful, but the surface is too unstable or too large for this milestone.
- `do not force` — keep existing construction because a builder would add ceremony or obscure a raw/simple boundary.

### Error candidates

- `snafu now` — migrate to `snafu` because structured context/source chaining improves diagnostics.
- `retain raw/std` — keep `std::io::Result`, raw status, or another low-level shape because that is the clearest boundary.
- `defer` — revisit after higher-priority migrations land.
- `do not force` — do not add a domain error layer where it would obscure hardware behavior or expand scope.

## Artifact contracts to preserve

Later tasks must confirm exact producers, consumers, and tests for each artifact field. Any field rename/removal must have an explicit replacement and verifier/test update.

### Proof and verifier fields

- `ok`
- `verdict`
- `device_path`
- `requested_bytes`
- `page_fault_retries`
- `final_status`
- `phase`
- `error_kind`

### Async/direct benchmark fields

- lifecycle failure kind
- worker failure kind
- direct failure kind
- retry budget
- retry count
- completion result/status/bytes/fault address
- validation phase/error kind
- `schema_version`
- backend
- suite
- claim eligibility
- result rows
- failure class

### No-payload-logging constraint

Diagnostics, displays, logs, reports, and artifacts may include lengths, paths, phases, statuses, kinds, retry counts, and completion metadata. They must not include source or destination payload bytes.
