# S05 hw-eval / idxd-sys cleanup report

**Milestone:** M008 — Hardware Rust `bon` / `snafu` cleanup  
**Slice:** S05 — bounded `hw-eval` / `idxd-sys` cleanup  
**Status:** Final with fresh T04 proof evidence  
**Reader:** A future M008/S06 maintainer or hardware-evaluation contributor landing cold on the S05 changes.  
**Post-read action:** Decide where to add future `bon` / `snafu` cleanup without breaking `hw-eval` benchmark JSON compatibility, no-payload diagnostics, or the intentionally lean `idxd-sys` raw boundary.

## Summary

S05 applied the M008 migration contract narrowly. `hw-eval` gained structured entrypoint diagnostics where the harness has enough context to explain failures: CLI size parsing, normalized benchmark configuration, work-queue open failures, non-fatal CPU pin warnings, and final JSON report serialization. `idxd-sys` stayed lean and raw: its ABI helpers, work-queue open result, and submission status surface remain `std` / raw hardware boundaries, guarded by host-free tests rather than new production abstractions.

No structural decision diverged from D051, D052, or the S01 migration contract. The implementation follows D051 by avoiding broad `snafu` / `bon` adoption in `idxd-sys`, and follows D052 by limiting `hw-eval` churn to failure/context improvements rather than benchmark architecture redesign.

## Legitimate targets and anti-targets

Legitimate S05 targets:

- Internal `hw-eval` benchmark configuration normalization with `bon`, while keeping Clap as the external CLI parser.
- `hw-eval` CLI/config, WQ-open, pin-warning, and report-serialization diagnostics with `snafu` where the harness can attach user-facing context.
- Host-free contract tests for malformed CLI inputs, software-only JSON compatibility, non-fatal pin warnings, missing hardware WQ diagnostics, and `idxd-sys` raw-boundary behavior.

Do-not-force anti-targets:

- No `bon` builders for `idxd-sys` descriptors, completions, WQ portals, or raw submission helpers.
- No broad `snafu` error enum in `idxd-sys`; raw `std::io::Result`, unsafe preconditions, and raw completion/submission statuses remain visible.
- No benchmark-matrix redesign, no conversion of every hot-loop panic into fallible control flow, and no JSON report field rename/removal.

## No-payload verification

S05 diagnostics may mention metadata: size values, malformed size tokens, accelerator kind, device path, operation name, OS error source, requested CPU core, completion statuses, IAX error codes, invalid flags, batch/concurrency, and report serialization class.

S05 diagnostics must not include source or destination buffer bytes. The CLI contract tests assert that software-only JSON output, malformed-size stderr, missing-WQ stderr, and pin-warning stderr do not expose the benchmark payload sentinel (`0xAB` / `171`). The report schema remains compatible with existing `metadata`, `latency`, and `throughput` consumers.
