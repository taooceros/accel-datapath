# Explorer index-first workflow

## Goal

Reduce `@explorer` context blowups by changing repo guidance from broad read-and-summarize discovery to staged, index-first discovery.

## Problem

- Current guidance encourages broad parallel `@explorer` tasks for uncertain scope.
- The same task often mixes discovery, selective reading, synthesis, and ambiguity analysis.
- Without explicit caps, `@explorer` can accumulate too many candidate files and intermediate notes before compressing them.

## Scope

- update root `AGENTS.md` guidance for `@explorer`
- prefer index-first discovery using KB/search metadata before file reading
- split discovery and synthesis into separate phases
- add bounded exploration rules for multi-domain explorer tasks
- keep direct local reads for known-path work unchanged

## Proposed changes

1. Add an **index-first** rule under tool routing / parallelism:
   - prefer KB, `grep`, `glob`, AST, and LSP metadata before reading files
   - use `@explorer` to return candidate sets first, not full synthesis by default
2. Add a **staged workflow** rule for `@explorer`:
   - phase 1: candidate discovery
   - phase 2: shortlist/prune
   - phase 3: selective detail reads and synthesis
3. Add **bounded search** rules:
   - cap files inspected per area unless explicitly expanded
   - cap number of domains per explorer invocation
   - require narrow follow-up jobs for cross-domain synthesis
4. Add an **orchestrator merge** rule:
   - prefer multiple bounded explorer jobs over one monolithic search when a task spans launcher/code/docs/benchmarks
5. Add a short **anti-patterns** section:
   - avoid combining discovery + deep reading + final synthesis in one explorer task for broad repo mapping

## Result

- repo guidance should steer future explorer use toward index-first, bounded, staged search
- broad repo mapping should shift from one context-heavy explorer task to multiple compact discovery passes with orchestration merge
