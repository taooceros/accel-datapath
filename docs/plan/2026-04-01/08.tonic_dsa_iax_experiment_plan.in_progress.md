# Plan: Tonic DSA/IAX Profiling and Async-Overhead Experiment

**Date**: 2026-04-01

## Goal

Design a repo-grounded experimental program that:

1. profiles tonic end to end to identify which payload-path stages are worth offloading to Intel DSA/IAX,
2. quantifies the software overhead added by the async/runtime path when DSA/IAX-style completion and batching are integrated,
3. separates raw hardware opportunity from integration overhead,
4. and defines an automation stack that can run the full experiment matrix reproducibly.

## Why this project is non-trivial

This is not a simple “plug in DSA and compare throughput” task.

- Prior repo analysis already indicates that the best tonic offload candidates are **compression/decompression** and **copy/copy+CRC** at the codec/body boundary, while Tokio, Tower orchestration, hyper/h2 state machines, and prost serialization are mostly **measurement targets**, not good offload targets.
- Prior repo benchmarking also shows a broader research risk: once hardware cost is batched down, **framework overhead can dominate total per-op cost**.
- `accel-rpc/tonic-profile/` is currently only a stub, and `accel-rpc/async-bench/` currently measures only a minimal Tokio spawn/join path.
- `hw-eval/` already provides a useful raw-hardware lower-bound harness and JSON-oriented reporting shape, but it does not yet answer the tonic-specific integration questions.

So the right first milestone is **decomposition and crossover analysis**, not transport replacement or deep integration.

The main architectural assumption is:

- **Target first**: codec/body hooks, compression/decompression, copy, copy+CRC, buffer movement.
- **Measure but do not offload first**: Tokio wake/poll/scheduling, Tower layering, hyper/h2, request orchestration, prost encode/decode.

## Primary research questions

1. In tonic's request/response path, what fraction of CPU time and tail latency is spent in:
   - protobuf encode/decode,
   - buffer allocation/growth/reuse,
   - payload copy,
   - compression/decompression,
   - gRPC framing,
   - Tower/hyper/h2 orchestration,
   - Tokio scheduling and completion delivery?
2. For which stages does raw DSA/IAX capability provide enough headroom to matter after integration overhead is included?
3. At what message sizes, payload types, and concurrency levels does offload beat software?
4. How much async-framework overhead is introduced by submission, completion notification, task wakeup, cross-thread movement, and batching control?
5. Can we derive a practical decision rule for “offload vs stay on CPU” per stage and workload regime?

## Hypotheses to test

1. **IAX compression/decompression** is the strongest first offload candidate for medium-to-large, compressible payloads.
2. **DSA copy/copy+CRC** is only worthwhile for payloads that retain a real copy/checksum budget after buffer pooling and software cleanup.
3. For small messages, **async completion overhead and runtime churn** can erase hardware gains even when raw DSA/IAX is faster than software for the isolated primitive.
4. The dominant blockers may turn out to be **buffer management and framework overhead**, not the raw transform itself.

## High-level project phases

## Phase 0 — Build the measurement substrate

### Objective

Turn `accel-rpc/tonic-profile/` from a stub into a controlled tonic measurement harness and expand `async-bench/` into a proper runtime-overhead microbenchmark suite.

### Deliverables

- `tonic-profile` baseline harness with unary RPC first, streaming second.
- Structured JSON output aligned as much as possible with `hw-eval` metadata/result conventions.
- A single experiment-manifest format that drives all benchmark runners.
- Result directories and post-processing scripts that join tonic, async, and raw-hardware results.

### Minimum harness features

For `tonic-profile`:

- client and server roles in one crate,
- configurable payload size,
- configurable payload generator,
- unary mode first, then client/server/bidi streaming,
- optional software compression,
- explicit CPU pinning,
- explicit NUMA/device metadata capture,
- warmup and steady-state windows,
- throughput + latency + CPU accounting output,
- optional internal phase timing hooks around encode/decode/compress/decompress/body assembly.

For `async-bench`:

- spawn/join baseline,
- oneshot completion latency,
- mpsc send/recv,
- wakeup path with same-thread vs cross-thread completion,
- completion-to-poll delay,
- batching-control overhead,
- polling-heavy vs event-driven variants.

## Phase 1 — Software-only tonic decomposition

### Objective

Measure where time goes in tonic **before** real accelerator integration.

### Rationale

This is the tonic equivalent of the repo's successful layer-removal methodology in `docs/report/benchmarking/006.stdexec_overhead_results.md`: isolate costs by controlled subtraction instead of relying on speculative per-function timing.

### Experimental variants

Start with unary RPC over loopback and then widen.

1. **Full baseline**
   - tonic + Tokio + hyper/h2 + prost + default buffering
   - compression off
2. **Compression baseline**
   - same as above, compression on
3. **Pooled-buffer software variant**
   - keep software path but reduce allocation/growth noise
4. **Copy-minimized software variant**
   - remove avoidable copies where possible without accelerator integration
5. **Middleware-placement control**
   - compare transform placement at codec/body vs middleware when practical
6. **Stub-offload shape variant**
   - preserve queueing/completion shape while replacing real hardware with a mock/instant-complete path

### Metrics

- RPC throughput
- p50/p95/p99 latency
- CPU utilization per endpoint
- cycles per RPC
- allocations / bytes allocated
- encoded payload size and compressibility
- per-stage timing totals
- flamegraphs / perf call stacks for representative points

### Expected outcome

An attribution table that answers: “If we improve stage X perfectly, how much end-to-end gain is even available?”

## Phase 2 — Async-framework overhead characterization

### Objective

Quantify the software tax of integrating accelerator-style submission/completion into the async stack.

### Key question

If raw DSA/IAX saves N ns on the transform, does Tokio/task/completion machinery spend N or more giving that benefit back?

### Benchmark families

1. **Submission-side overhead**
   - enqueue request to worker / completion path
   - batch assembly overhead
   - descriptor bookkeeping surrogate cost
2. **Completion-side overhead**
   - oneshot completion
   - channel-based completion
   - manual poll + wake callback path
   - completion to task resumption delay
3. **Placement/topology overhead**
   - same-thread completion
   - cross-thread completion
   - pinned vs unpinned runtime
   - single-thread runtime vs multi-thread runtime
4. **Batching sensitivity**
   - 1, 2, 4, 8, 16, 32, 64 operations per completion batch

### Output

Per-primitive overhead curves and a reusable “software overhead budget” that can be added to `hw-eval` raw lower bounds when reasoning about integrated offload.

## Phase 3 — Raw-hardware floor and crossover preparation

### Objective

Use `hw-eval/` to establish what DSA/IAX can do in isolation, and align that with tonic candidate stages.

### What to reuse from existing repo work

- `hw-eval/src/main.rs` already exposes latency and throughput modes with metadata.
- `hw-eval/plot_results.py` already provides a JSON-to-graphs pattern.
- Existing software baselines in `hw-eval` can anchor memcpy/CRC comparisons.

### Gaps to close conceptually

- Treat DSA copy/copy+CRC as an immediate lower-bound input.
- Do **not** assume current `hw-eval` fully answers IAX compression/decompression yet; compression-specific raw-floor work may need to be added later.

### Core analysis

For each candidate stage:

`integrated_offload_cost ≈ raw_hw_cost + submission/control_cost + completion/wakeup_cost + extra_copy_or_buffer_cost`

Offload only becomes plausible when that integrated cost beats the software stage cost in the relevant workload regime.

## Phase 4 — Controlled offload integration experiments

### Objective

Integrate only the most promising candidates, in priority order, with strong controls.

### Priority order

1. **IAX compression/decompression at codec compression boundary**
2. **DSA copy+CRC at codec/body or middleware payload transform boundary**
3. **DSA copy-only for unavoidable large payload movement**

### Integration principles

- Integrate at the codec/body boundary first.
- Keep the transport/runtime unchanged as much as possible.
- Preserve software-control variants for A/B comparison.
- Keep offload eligibility explicit and measurable.

### Comparison set per candidate

1. full software baseline,
2. software-optimized control,
3. mock-offload control with same async shape,
4. real offload integration.

This prevents “better buffering” or “different queue shape” from being mis-labeled as “hardware acceleration.”

## Phase 5 — Synthesis and decision policy

### Objective

Produce a practical policy for when to offload.

### Final outputs

- crossover plots by size, concurrency, payload type, and RPC mode,
- attribution tables showing where end-to-end gains come from,
- async-overhead budget tables,
- recommendation matrix:
  - offload now,
  - software-optimize first,
  - do not pursue.

## Experimental matrix

The initial matrix should stay deliberately bounded.

### Workload dimensions

- **RPC type**: unary first; then client streaming, server streaming, bidi streaming
- **payload size**: 64 B, 256 B, 1 KB, 4 KB, 16 KB, 64 KB, 256 KB, 1 MB
- **payload type**:
  - incompressible random bytes,
  - moderately compressible structured bytes,
  - highly compressible synthetic bytes,
  - protobuf-like structured messages
- **compression mode**: off, software-on, candidate offload-on
- **concurrency**: 1, 8, 32, 128, 512, 2048
- **runtime placement**:
  - single-thread runtime,
  - multi-thread runtime,
  - pinned client/server threads,
  - same-NUMA vs cross-NUMA where possible
- **offload mode**:
  - none,
  - mock,
  - DSA,
  - IAX

### Bounded first-pass matrix

Do **not** run the full Cartesian product first.

Start with:

- unary RPC,
- loopback or same-host client/server,
- compression off and software-on,
- 4 payload sizes: 256 B, 4 KB, 64 KB, 1 MB,
- 4 concurrency levels: 1, 32, 128, 512,
- pinned single-NUMA runs first.

Widen only after the harness and analysis pipeline are stable.

## Instrumentation plan

### In-process measurements

Add explicit timestamps/counters around:

- request construction,
- prost encode,
- codec buffer growth/reserve,
- compression/decompression,
- gRPC frame assembly,
- body handoff,
- receive-side frame decode,
- prost decode,
- completion notification path for any offload integration.

### External measurements

- `perf stat` for cycles/instructions/cache misses/branch misses/context switches
- `perf record` + flamegraphs for representative points
- allocator statistics if available
- CPU placement / NUMA metadata in every run record

### Methodological rule

Do not trust only micro-timestamps. Use **variant deltas** and **layer-removal style comparisons** as the primary source of truth, and use timestamps/flamegraphs to explain the deltas.

## Automation architecture

## Principle

Use one manifest-driven experiment runner and one common result schema.

## Proposed structure

- `accel-rpc/tonic-profile/`
  - benchmark harness binary
- `accel-rpc/async-bench/`
  - Criterion microbenches plus machine-readable summaries
- `hw-eval/`
  - raw hardware lower-bound runner
- `tools/` or `scripts/`
  - orchestration, environment capture, result collation, plotting

## Proposed automation components

1. **Experiment manifest**
   - YAML or JSON file describing matrix entries
   - includes payload size, concurrency, compression mode, runtime config, offload mode, repetitions, warmup, pinning, and output directory

2. **Top-level orchestrator script**
   - reads manifest
   - runs `tonic-profile`, `async-bench`, and `hw-eval` in the correct order
   - uses `launch` for hardware-facing binaries when required
   - stamps environment metadata
   - writes per-run JSON artifacts

3. **Environment capture script**
   - CPU model
   - kernel version
   - Rust version
   - git commit
   - NUMA topology
   - DSA/IAX device path and queue mode

4. **Result collation script**
   - merges JSON from tonic, async, and raw hardware runs
   - computes derived metrics
   - emits a unified analysis-ready JSON/CSV bundle

5. **Plot/report script**
   - reuse the style of `hw-eval/plot_results.py`
   - generate crossover, attribution, and overhead-budget plots

## Suggested first script set

- `tools/run_tonic_matrix.py`
- `tools/run_async_matrix.py`
- `tools/run_hw_floor.py`
- `tools/collect_env.py`
- `tools/merge_experiment_results.py`
- `tools/plot_tonic_offload_results.py`

These names are proposals for the implementation phase; the important point is a manifest-driven, reproducible workflow rather than ad hoc shell commands.

## Result schema requirements

Every run should record:

- benchmark family (`tonic_profile`, `async_bench`, `hw_eval`)
- timestamp
- git revision
- machine metadata
- pinned cores / NUMA
- runtime mode
- RPC mode
- payload generator and size
- concurrency and batching config
- compression and offload mode
- latency summary
- throughput summary
- CPU/cycles/cache summary
- per-stage counters/timers when available

The schema should mirror `hw-eval`'s current metadata/result separation as closely as practical so that tooling remains consistent.

## Threats to validity

1. **Allocation effects can masquerade as offload wins**
   - control with pooled-buffer software variants.
2. **Loopback/network/TLS effects can hide payload-stage changes**
   - start with same-host decomposition before broader end-to-end claims.
3. **Compressibility can artificially inflate IAX gains**
   - use multiple payload classes.
4. **NUMA/device placement can dominate the result**
   - always record locality and run pinned baselines.
5. **Async completion overhead can erase hardware savings**
   - explicitly measure completion and wakeup costs.
6. **Mock-vs-real mismatch**
   - use mock to isolate software cost, but validate final claims on real hardware paths.
7. **Overly broad early matrix**
   - keep the first-pass matrix intentionally small.

## Success criteria

The project succeeds if it produces all of the following:

1. a stable tonic decomposition harness,
2. a reproducible async-overhead microbenchmark suite,
3. raw-hardware lower-bound data aligned with tonic candidate stages,
4. crossover maps showing where offload wins, loses, or ties,
5. a recommendation matrix that clearly separates:
   - “worth integrating now,”
   - “software optimization first,”
   - “not promising.”

## Candidate-specific go/no-go rules

Proceed with deeper integration only if:

- the candidate stage is a material fraction of target workload cost,
- raw hardware shows meaningful headroom over software,
- integrated offload improves throughput or tail latency materially, or reduces CPU materially at iso-throughput,
- and the gain survives controls for pooling, locality, and runtime placement.

Stop early if:

- prost + runtime orchestration dominate after software cleanup,
- completion overhead is comparable to or larger than the saved stage cost,
- gains only appear on unrealistic synthetic payloads,
- or software-only controls remove most of the candidate budget.

## What not to do first

- Do not start by changing Tokio, hyper, or h2 internals.
- Do not start by building a new accelerator-native transport.
- Do not treat tonic interceptors as payload-path hooks.
- Do not use spawn/join-only async results as the final framework-overhead story.
- Do not assume current `hw-eval` fully answers IAX compression readiness.
- Do not run an uncontrolled full-matrix sweep before the harness is validated.

## Implementation order

1. Define the common result schema and experiment manifest.
2. Implement unary `tonic-profile` baseline with JSON output and pinning metadata.
3. Expand `async-bench` to cover submission/completion/wakeup microbenchmarks.
4. Add tonic decomposition variants and internal counters.
5. Wire manifest-driven automation and result collation.
6. Run the bounded software-only matrix.
7. Compare against `hw-eval` lower bounds.
8. Choose the first real offload candidate.
9. Integrate one candidate with software and mock controls.
10. Generate crossover maps and the final recommendation matrix.

## Expected output

This project should eventually produce:

- one or more `docs/report/benchmarking/` reports on tonic decomposition and async overhead,
- automation scripts for repeatable runs,
- a tonic profiling harness that replaces the current stub,
- and a grounded argument for which tonic stages should or should not be offloaded to Intel DSA/IAX.
