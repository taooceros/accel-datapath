# Explorer aggressive memory writes

## Goal

Require explorer runs to checkpoint findings aggressively into session memory so useful state survives compaction pressure and resumed runs can hand back cleanly without re-discovery.

## Problem

- Explorer now stops on prior compaction, but session memory can still be too sparse if findings are only written late.
- Short exploratory rounds can discover useful candidates, rejections, and files-read state that should be persisted immediately.
- Without eager checkpointing, resumed runs may lack enough state to explain what was already learned.

## Scope

- update root `AGENTS.md` explorer policy
- update `.agents/workflows/explorer.md`
- update `.agents/templates/explorer_discovery.md`

## Planned changes

1. Require explorer to write findings into session memory after each meaningful search step.
2. Require checkpointing before any action likely to grow context further.
3. Clarify the minimum state that should be persisted eagerly.
4. Keep the policy session-scoped and execution-only, not repo knowledge.

## Completion notes

- Root `AGENTS.md` now requires aggressive checkpointing of explorer findings into session memory after meaningful search steps.
- `.agents/workflows/explorer.md` now defines a dedicated checkpointing section with minimum update triggers and compact-write guidance.
- `.agents/templates/explorer_discovery.md` now instructs delegated explorer tasks to checkpoint candidates, rejections, unknowns, file reads, and round boundaries eagerly.
