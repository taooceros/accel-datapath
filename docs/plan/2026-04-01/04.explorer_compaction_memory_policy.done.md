# Explorer compaction memory policy

## Goal

Tighten explorer and orchestrator guidance so bounded discovery survives 128k-context pressure: when explorer runs are compacted or resumed, they should stop, hand back current findings, and suggest parallel decomposition instead of re-expanding search.

## Problem

- Explorer tasks can still loop after compaction by re-globbing to rebuild lost context.
- Existing explorer templates describe budgets, but not explicit session-state accounting or compaction handling.
- The orchestrator policy does not yet require a session-scoped explorer memory record for resumed runs.

## Scope

- update root `AGENTS.md` explorer policy for orchestrator-side session memory and compaction handling
- update `.agents/workflows/explorer.md` with explicit session-state and compaction rules
- update `.agents/templates/explorer_discovery.md` so delegated prompts carry hard-stop and status requirements

## Planned changes

1. Add root-policy guidance that explorer state is session-scoped, not repo knowledge.
2. Require the orchestrator to pass explorer session status for resumed runs.
3. Add a hard rule that compaction/resume triggers immediate return instead of renewed broad discovery.
4. Require explorer outputs to include round/candidate/compaction/stop-condition status.
5. Require compaction-triggered handoff to recommend parallel sub-searches for follow-up.

## Completion notes

- Root `AGENTS.md` now requires session-scoped explorer memory for resumed work and treats prior compaction / resumed-with-missing-context as a hard handback condition.
- `.agents/workflows/explorer.md` now defines explorer session state, missing-context detection, compaction response, and required status reporting.
- `.agents/templates/explorer_discovery.md` now carries session-state fields, hard-stop rules, compaction-triggered immediate return behavior, and explicit status output.
