# Plan: Tonic-Only Profiling Experiment

**Date**: 2026-04-01

## Goal

Design a clear, reproducible experiment plan that focuses **only on tonic** for now.

The purpose of this phase is:

1. to profile tonic's end-to-end request/response path,
2. to identify which tonic stages deserve deeper investigation later for DSA/IAX offload,
3. to quantify the software overhead inside the tonic/Tokio/hyper/h2/prost stack,
4. and to define exactly how the experiments should be run and automated.

## Scope

### In scope

- tonic request/response profiling
- tonic codec/body boundary profiling
- software-only compression/copy/buffer experiments inside tonic
- async/runtime overhead attribution as it affects tonic
- experiment harness design
- experiment automation design
- run procedure, setup, metadata capture, and result format

### Out of scope for this phase

- real DSA integration
- real IAX integration
- accelerator transport replacement
- final offload implementation

Hardware/offload analysis is only used here as a **guide for where to look carefully in the tonic profiles**.

## Repo-grounded basis

This plan is grounded in the current repo state:

- `docs/report/architecture/002.tonic_component_analysis.md`
- `docs/report/architecture/003.tonic_interception_points.md`
- `docs/research_plan.md`
- `accel-rpc/README.md`
- `accel-rpc/Cargo.toml`
- `accel-rpc/tonic-profile/src/main.rs`
- `accel-rpc/async-bench/benches/async_overhead.rs`

The repo's current conclusions already narrow the search space:

- **Likely interesting payload-path stages**: compression/decompression, copy/copy+CRC, buffer movement, buffer lifecycle.
- **Important to measure but not offload first**: Tokio scheduling, Tower orchestration, hyper/h2 state machines, prost encode/decode, request routing.

That means this phase should answer:

> inside tonic, where is time actually spent, how stable is that attribution across workloads, and which stage is large enough to justify deeper follow-up?

## Main research questions

1. For a tonic RPC, how much time is spent in:
   - prost encode,
   - codec buffer reservation/growth,
   - compression/decompression,
   - frame assembly/parsing,
   - body movement/copying,
   - Tower/hyper/h2 orchestration,
   - Tokio scheduling and wakeups,
   - prost decode?
2. How do those costs change with:
   - payload size,
   - payload compressibility,
   - unary vs streaming,
   - concurrency,
   - runtime topology?
3. How much of the apparent cost is actually:
   - allocation,
   - buffer policy,
   - task wakeup/notification,
   - pipeline placement,
   rather than the transform itself?
4. Which tonic stages are large and stable enough to justify the later DSA/IAX integration phase?

## Working hypotheses

1. For small messages, framework and buffering overhead will dominate.
2. For medium and large messages, compression and payload movement become more visible.
3. A meaningful fraction of “payload work” may actually be buffer allocation/copy policy rather than raw transform cost.
4. Unary and streaming RPCs will show different attribution patterns because buffering and batching behavior differ.

## Success criteria for this phase

This phase succeeds if it produces:

1. a stable tonic profiling harness,
2. a reproducible software-only experiment matrix,
3. attribution tables for major tonic stages,
4. `cargo flamegraph` and `perf` data for representative runs,
5. a shortlist of tonic stages worth deeper follow-up in the next phase.

## What the final answer from this phase should look like

At the end of this phase, we want a report that can say:

- “For small unary RPCs, X/Y/Z dominate; do not look at offload yet.”
- “For large compressible payloads, compression is N% of end-to-end cost; investigate this next.”
- “Copy cost disappears after pooling, so copy offload is not the first follow-up.”
- “Streaming shifts cost from scheduling to buffer management.”

## Experimental philosophy

This phase should use **decomposition** rather than premature integration.

The key rule is:

> first explain where tonic spends time in pure software, then decide what deserves deeper follow-up.

Do not begin with hardware hooks. Instead, build software variants that remove or isolate candidate costs.

## Experiment design overview

The experiment has four layers:

1. **End-to-end tonic baseline runs**
2. **Internal tonic phase instrumentation**
3. **Variant-based subtraction experiments**
4. **Async-overhead microbenchmarks that explain tonic behavior**

These four layers should be analyzed together.

## Phase A — Build the tonic profiling harness

## Objective

Turn `accel-rpc/tonic-profile/` from a stub into the primary tonic profiling harness.

## Required harness capabilities

### Functional capabilities

- one binary crate with `server`, `client`, and `selftest` modes
- unary RPC first
- later: client streaming, server streaming, bidi streaming
- configurable payload size
- configurable payload generator
- configurable RPC count and concurrency
- optional software compression on/off
- optional internal instrumentation on/off
- warmup duration
- measurement duration
- JSON output

### Measurement capabilities

- throughput
- latency summary: p50, p95, p99, max
- per-stage cumulative timers
- request counts
- bytes sent/received
- CPU/core metadata
- runtime mode metadata
- allocator/copy counters when available

### Proposed CLI shape

The exact implementation may evolve, but the harness should target a CLI like:

```bash
cd accel-rpc
cargo run -p tonic-profile --release -- \
  --mode selftest \
  --rpc unary \
  --payload-size 4096 \
  --payload-kind structured \
  --compression off \
  --concurrency 128 \
  --requests 200000 \
  --warmup-ms 3000 \
  --measure-ms 10000 \
  --server-core 4 \
  --client-core 5 \
  --json-out results/tonic/unary_4k_structured_c128.json
```

For split client/server mode:

```bash
cd accel-rpc
cargo run -p tonic-profile --release -- \
  --mode server \
  --rpc unary \
  --bind 127.0.0.1:50051 \
  --compression off \
  --server-core 4

cargo run -p tonic-profile --release -- \
  --mode client \
  --rpc unary \
  --target 127.0.0.1:50051 \
  --payload-size 4096 \
  --payload-kind structured \
  --compression off \
  --concurrency 128 \
  --requests 200000 \
  --warmup-ms 3000 \
  --measure-ms 10000 \
  --client-core 5 \
  --json-out results/tonic/unary_4k_structured_c128.json
```

## Instrumentation plan inside tonic

Instrumentation should be placed at the boundaries already identified in prior repo analysis.

### Outbound path

Measure cumulative time and counts for:

1. application request creation
2. prost encode
3. codec buffer reserve/grow
4. compression
5. gRPC header/frame assembly
6. body handoff to transport

### Inbound path

Measure cumulative time and counts for:

1. body accumulation
2. frame parse
3. decompression
4. prost decode
5. response handoff to application

### Runtime/control-path attribution

Track:

- task count / spawned work if relevant
- wakeup or completion counts if visible
- batching-related counters if used later in streaming mode

### Important rule

Do not rely on instrumentation alone. Pair it with **variant-based subtraction**, **`perf` analysis**, and **`cargo flamegraph` call-stack visualization**.

- 1
- 8
- 32
- 128
- 512

### Runtime mode

- single-thread Tokio runtime
- multi-thread Tokio runtime

## First-pass bounded matrix

Do **not** start with the full Cartesian product.

Start with:

- unary only
- single-host loopback
- compression off/on
- payload sizes: 256 B, 4 KB, 64 KB, 1 MB
- payload kinds: random, structured
- concurrency: 1, 32, 128, 512
- runtime modes: single-thread and multi-thread

This first-pass matrix is already enough to reveal whether the attribution is stable or workload-sensitive.

## Phase C — Tonic software variants for subtraction experiments

## Goal

Use controlled software variants to isolate where tonic time comes from.

## Required variants

### Variant 0: full baseline

- tonic default path
- compression off

### Variant 1: compression enabled

- same as baseline
- tonic software compression on

### Variant 2: pooled-buffer variant

- reduce allocation/buffer growth noise
- same functionality as baseline

### Variant 3: copy-minimized variant

- remove avoidable buffer copies where practical
- same transport/runtime

### Variant 4: instrumentation-off control

- same as baseline but without internal phase timing
- used to quantify instrumentation overhead

### Variant 5: placement-control variant

- where applicable, compare transformation at codec/body boundary versus middleware boundary

## Why these variants matter

These variants let us answer:

- Is compression expensive enough to matter?
- Is buffer growth the real problem?
- Is copy visible after pooling?
- Does instrumentation distort results?
- Does placement change cost attribution?

## Phase D — Async-overhead side measurements for tonic interpretation

## Goal

Explain tonic behavior with small focused microbenchmarks.

`async-bench` should be expanded to measure the overheads most likely to appear in tonic:

- Tokio spawn/join
- oneshot send/recv
- mpsc send/recv
- same-thread wakeup
- cross-thread wakeup
- single-thread runtime vs multi-thread runtime overhead

These are not the main experiment outputs, but they help explain tonic-level observations.

## Experiment setup

## Machine setup requirements

Use a fixed machine and record:

- CPU model
- core count
- SMT on/off
- NUMA topology
- kernel version
- Rust toolchain version
- git commit hash

## Placement rules

- pin the server to a fixed core
- pin the client to a fixed core
- keep client/server on the same NUMA node for the first pass
- do not vary placement and workload at the same time in early runs

## Run-mode rules

- use release builds only
- use a warmup period before measurement
- use fixed-duration or fixed-request steady-state windows
- run at least 3 repeats per configuration
- store raw JSON per repeat

## Noise-control rules

- no unrelated workloads on the machine
- stable CPU frequency policy if possible
- same build for all compared runs
- same loopback path and same machine for baseline comparison
- instrumentation-overhead control run always included

## Automation plan

## Principle

Use one manifest-driven runner for tonic experiments.

## Proposed files

- `tools/run_tonic_profile_matrix.py`
- `tools/collect_tonic_env.py`
- `tools/merge_tonic_results.py`
- `tools/plot_tonic_profile_results.py`
- `experiments/tonic_profile_matrix.json` or `.yaml`

## Manifest fields

Each matrix entry should include:

- rpc mode
- payload size
- payload kind
- compression mode
- concurrency
- runtime mode
- warmup duration
- measurement duration
- repeat count
- client core
- server core
- instrumentation on/off
- variant name

## Orchestrator behavior

The matrix runner should:

1. load the manifest,
2. create output directories,
3. stamp environment metadata,
4. run each configuration N times,
5. capture stdout/stderr and JSON,
6. skip or retry failed runs,
7. write a merged summary file.

## Output schema

Each tonic run should emit:

- metadata
  - timestamp
  - git commit
  - machine info
  - cores
  - runtime mode
  - payload size/kind
  - compression mode
  - concurrency
  - variant
- end-to-end metrics
  - throughput
  - p50/p95/p99/max latency
  - request count
  - bytes transferred
- internal attribution
  - encode time
  - decode time
  - compression time
  - decompression time
  - buffer reserve/grow time
  - frame assembly/parse time
- optional external metrics
  - cycles
  - instructions
  - LLC misses
  - context switches

## Analysis plan

## Primary plots

1. throughput vs concurrency
2. latency vs payload size
3. compression-on vs compression-off delta
4. per-stage attribution stacked bars
5. runtime mode comparison: single-thread vs multi-thread
6. variant deltas: baseline vs pooled-buffer vs copy-minimized

## Primary tables

1. top contributors by workload point
2. variance across repeats
3. instrumentation overhead
4. compression share of total cost
5. buffer-management share of total cost

## Decision rules

After this phase, a tonic stage becomes a next-step candidate only if:

- it is a material fraction of total cost,
- it remains visible across repeated runs,
- its cost is not explained away by instrumentation or allocation artifacts,
- and its importance persists in at least one realistic workload regime.

## Threats to validity

1. instrumentation overhead distorts attribution,
2. loopback hides some transport effects,
3. synthetic payloads misrepresent real compressibility,
4. allocation effects are mistaken for transform cost,
5. multi-thread runtime noise obscures stage attribution.

Each threat should be addressed with a control:

- instrumentation-off runs,
- multiple payload kinds,
- pooled-buffer variant,
- single-thread and multi-thread comparison,
- repeated runs with pinned placement.

## Implementation order

1. implement unary `tonic-profile` with JSON output,
2. add payload size/kind/compression/concurrency controls,
3. add client/server core pinning and metadata capture,
4. add internal phase timers,
5. implement baseline matrix runner,
6. run first-pass bounded matrix,
7. add pooled-buffer and copy-minimized variants,
8. collect `perf` and `cargo flamegraph` data for representative points,
9. expand to streaming modes,
10. write the profiling report and shortlist next-step candidates.

## Expected output

This plan should produce:

- a tonic-only profiling harness,
- a reproducible experiment workflow,
- a clear attribution of tonic software cost,
- and a grounded answer about which tonic stages deserve deeper analysis in the hardware-integration phase.
