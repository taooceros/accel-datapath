## Goal

Update the tonic-only profiling plan into a more explicit **characterization plan** that is directly grounded in the repo's literature intake and first-pass profiling results.

The immediate purpose is to produce a measurement campaign that can answer, with evidence rather than intuition:

1. which parts of the tonic request/response path dominate under which workload regimes,
2. which apparent hot spots are actually allocation, buffer policy, or runtime-placement artifacts,
3. and which payload-path stages remain large enough after software cleanup to justify later DSA/IAX follow-up.

This plan tightens `docs/plan/2026-04-01/09.tonic_only_profiling_experiment_plan.in_progress.md` rather than replacing its broad direction.

## Why this refinement is needed now

Since the earlier plan was written, the repo gained two kinds of evidence that change how the work should be sequenced:

- literature that justifies a **bucketed cost-decomposition view** of RPC stacks,
- bounded-matrix profiling that already shows the dominant bucket changes with workload regime.

The key implication is that Tonic should not be characterized with a single "baseline vs optimized" story. It should be characterized **by regime** across:

- runtime / queueing behavior,
- serialization / deserialization,
- copy and buffer lifecycle,
- compression / decompression,
- framing / transport glue,
- and end-to-end latency tails.

## Main characterization question

For a tonic RPC, what is the end-to-end cost split across runtime/queueing, encode/decode, copy/buffer lifecycle, compression, framing, and transport glue, and how does that split change with workload regime?

## Characterization principles

1. **Characterize by bucket, not by monolithic throughput alone.**
   Every result should be interpretable in terms of queueing/runtime, serialization, copy/buffer lifecycle, compression, or framing/transport work.

2. **Characterize by regime, not by one representative point.**
   The bounded matrix already shows that small, medium, and large payloads can prefer different runtime/concurrency choices.

3. **Use subtraction and controls before claiming offload opportunity.**
   If pooling or copy minimization collapses a hot path, then that path is not yet a strong hardware target.

4. **Separate throughput from tail-latency interpretation.**
   Compression already shows cases where throughput drops while p99 improves.

5. **Prefer codec/body-boundary instrumentation over metadata-level hooks.**
   Characterization should follow the real byte path, not interceptor-visible control metadata.

## Research questions

1. For unary tonic RPCs, how much cumulative time and CPU share are spent in:
   - prost encode/decode,
   - buffer reservation/growth/reuse,
   - payload copies,
   - compression/decompression,
   - frame assembly/parsing,
   - body accumulation/handoff,
   - Tower/hyper/h2 orchestration,
   - Tokio scheduling and wake behavior?
2. How does that split change with:
   - payload size,
   - payload kind / compressibility,
   - concurrency,
   - runtime mode,
   - client/server placement,
   - unary vs streaming mode?
3. After software controls are applied, which buckets remain large and stable enough to motivate later DSA/IAX integration?
4. Which conclusions are robust across placement and payload realism, and which are artifacts of the current localhost setup?

## Working hypotheses

1. **Tiny unary RPCs are dominated by control-path and queueing/runtime effects.**
2. **Medium and large uncompressed RPCs are dominated by copy, allocation, and buffer growth rather than protobuf or HTTP/2 semantics.**
3. **Compression is regime-sensitive rather than globally good or bad.**
4. **A meaningful fraction of apparent payload work disappears once pooling and copy-minimization controls are applied.**
5. **Streaming will shift attribution away from per-request scheduling and toward buffer management, batching, and body accumulation.**

## Success criteria

This phase succeeds when the repo can produce a report that:

1. shows throughput and p50/p95/p99/max latency for matched workload regimes,
2. reports internal stage timers for the codec/body/compression path,
3. explains representative hot paths with `perf` / flamegraph evidence,
4. uses control variants to distinguish transform cost from allocation/copy policy,
5. and ends with a ranked shortlist of stages that still justify offload follow-up.

## Output shape the characterization should support

The final report from this phase should be able to say things like:

- "At 256 B unary / low concurrency, runtime and control-path overhead dominate; payload offload is not yet the lever."
- "At 4 KiB–64 KiB uncompressed, copy plus allocation dominates, and pooled-buffer controls remove N% of the apparent cost."
- "At 64 KiB structured compressed, throughput falls but p99 improves, so compression must be evaluated on both axes."
- "After software cleanup, only stage X retains enough stable budget to justify DSA/IAX integration."

## Measurement buckets

Every experiment and result table should map work into these buckets:

1. **Runtime / queueing / scheduling**
   - Tokio runtime mode
   - wakeup and task-resumption effects
   - client/server placement sensitivity

2. **Serialization / deserialization**
   - prost encode
   - prost decode

3. **Copy / buffer lifecycle**
   - `BytesMut` growth
   - buffer reserve/reallocation
   - body accumulation
   - explicit or implicit payload copies

4. **Compression / decompression**
   - software compression
   - software decompression
   - compression-ratio-sensitive behavior

5. **Framing / transport glue**
   - gRPC frame header handling
   - body handoff
   - hyper/h2-visible transport glue where it appears in profiles

## Workload dimensions

### Phase 1 dimensions: unary decomposition

- payload sizes: `256 B`, `4 KiB`, `64 KiB`, `1 MiB`
- payload kinds: `random`, `structured`, `repeated`
- compression: `off`, `on`
- concurrency: `1`, `32`, `128`, `512`
- runtime mode: `single`, `multi`
- placement mode:
  - same-host loopback, split client/server processes,
  - same-NUMA pinned cores first,
  - stricter split-core repeat for representative points

### Phase 2 dimensions: unary refinement

Expand only after Phase 1 is interpretable:

- payload sizes: add `64 B`, `1 KiB`, `16 KiB`, `256 KiB`
- topology: vary same-core vs split-core vs cross-NUMA on selected matched points
- repeat count: widen beyond 3 where tails are unstable

### Phase 3 dimensions: streaming extension

Only after unary attribution is stable:

- server streaming
- client streaming
- bidi streaming
- message count per stream
- yield-threshold / batching-sensitive settings when exposed

## Metrics to record

### End-to-end metrics

- throughput
- p50 latency
- p95 latency
- p99 latency
- max latency
- request count / error count
- bytes sent / received

### Stage-attribution metrics

- encode timer total
- decode timer total
- compress timer total
- decompress timer total
- buffer reserve / grow timer total
- body accumulation / handoff timer total
- optional counts for reserve/grow/copy events where visible

### System / runtime metrics

- CPU utilization by endpoint if available
- cycles / instructions / IPC for representative points
- memory-bound / backend-bound indicators from `perf stat`
- runtime mode, pinning, NUMA, kernel, Rust version, git hash

## Instrumentation plan

### Primary instrumentation sites

Follow the codec/body hooks already identified in repo analysis:

- outbound encode path
  - message encode
  - buffer reserve/grow
  - compression
  - frame-header write
  - body handoff
- inbound decode path
  - body accumulation
  - frame parse
  - decompression
  - prost decode
  - application handoff

### Secondary attribution support

- `perf stat` on representative matched points
- `perf record` / flamegraph captures on representative matched points
- `async-bench` microbenchmarks to explain runtime-side effects that do not show cleanly inside tonic timers

### Instrumentation rules

1. timers must be toggleable so instrumentation-overhead controls remain possible,
2. timer output must be cumulative and count-backed,
3. representative call-stack captures must be collected for both compressed and uncompressed regimes,
4. internal timers do not replace subtraction experiments.

## Control variants

The refined plan should explicitly compare these software variants:

1. **Baseline**
   - tonic default path
   - instrumentation on/off pair

2. **Compression-on baseline**
   - same as baseline with tonic software compression enabled

3. **Pooled-buffer variant**
   - reduce allocation and growth noise while keeping semantics unchanged

4. **Copy-minimized variant**
   - remove avoidable copies without changing transport/runtime semantics

5. **Placement-control variant**
   - compare codec/body-boundary transform placement against middleware placement where practical

6. **Runtime-control variant**
   - same workload under single-thread and multi-thread runtime on matched pinning

These controls exist to answer:

- whether a hot path is transform cost or policy overhead,
- whether instrumentation is distorting results,
- whether placement changes attribution materially,
- whether runtime behavior or payload work is the actual bottleneck.

## Characterization sequence

### Phase A — Close unary attribution gaps

**Objective:** finish the missing pieces from the existing bounded matrix so unary characterization becomes trustworthy.

Tasks:

1. add internal phase timers to `tonic-profile`,
2. run instrumentation-on / instrumentation-off controls,
3. implement pooled-buffer and copy-minimized software variants,
4. rerun matched unary points chosen from the bounded matrix,
5. recompute attribution using matched regimes rather than cherry-picked best points.

Expected output:

- a unary attribution table with stable stage buckets,
- explicit instrumentation-overhead numbers,
- subtraction evidence for allocation/copy-policy effects.

### Phase B — Refine runtime and placement interpretation

**Objective:** explain when runtime topology matters and when payload movement dominates.

Tasks:

1. expand `async-bench` beyond spawn/join to oneshot, mpsc, same-thread wake, cross-thread wake,
2. rerun representative unary points under matched single-thread vs multi-thread runtimes,
3. rerun selected points under stricter split-core and, if needed, cross-NUMA placement,
4. align async-microbench results with tonic-level observations.

Expected output:

- a runtime-overhead side report that explains the observed runtime crossover,
- stronger claims about where runtime policy matters and where it does not.

### Phase C — Extend to streaming

**Objective:** determine whether the unary conclusions survive when message accumulation and batching behavior change.

Tasks:

1. add server-streaming, client-streaming, and bidi modes to the harness,
2. choose a bounded streaming matrix rather than the full unary matrix,
3. collect stage timers and representative `perf` evidence,
4. compare streaming attribution against unary attribution.

Expected output:

- a streaming-vs-unary comparison focused on buffering, batching, and body accumulation.

### Phase D — Prepare offload-readiness ranking

**Objective:** turn characterization into a decision rule for later DSA/IAX work.

Tasks:

1. rank buckets by remaining stable cost after software cleanup,
2. map those buckets to codec/body or middleware insertion points,
3. compare remaining budgets against raw-hardware lower bounds from `hw-eval/`,
4. define explicit "do not offload yet" and "candidate for follow-up" thresholds.

Expected output:

- an offload-readiness shortlist supported by measured residual software budgets.

## Representative profiling points

For each major characterization pass, collect `perf stat`, `perf record`, and flamegraph evidence for matched points such as:

1. small / low concurrency / compression off,
2. medium / medium concurrency / compression off,
3. large / high concurrency / compression off,
4. structured compressible / compression on,
5. one point where runtime mode changes the winner,
6. one point where pooling or copy minimization materially changes attribution.

## Threats to validity and controls

1. **Loopback bias**
   - keep first-pass localhost for decomposition,
   - do not overclaim network-wire conclusions from it.

2. **Synthetic payload bias**
   - keep payload kind explicit (`random`, `structured`, `repeated`),
   - do not generalize compression conclusions without naming the payload regime.

3. **Instrumentation overhead**
   - always include instrumentation-off controls,
   - report the delta.

4. **Placement confounding**
   - do not vary topology and workload dimensions simultaneously in early runs,
   - rerun representative points under stricter pinning before broad claims.

5. **Allocation mistaken for transform cost**
   - require pooled-buffer and copy-minimized controls before calling copy or compression a stable target.

6. **Best-point cherry-picking**
   - compare matched regimes, not independently optimized points, when making attribution claims.
