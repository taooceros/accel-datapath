# Plan: Tonic Interception Analysis

**Date**: 2026-03-30

## Goal

Produce a repo-grounded report that explains where tonic components can be intercepted, what each hook can and cannot change, and which hooks are the right fit for `accel-rpc`'s accelerator-oriented work.

## Steps

1. Review the existing tonic architecture report and the `accel-rpc` workspace docs.
2. Map the generated-code boundaries against tonic runtime boundaries.
3. Identify interception points at codegen, interceptor, Tower layer, transport, and codec/body boundaries.
4. Rank those hooks by usefulness for metadata policy, observability, and payload-path acceleration.
5. Write the analysis under `docs/report/architecture/` and sync the KB index.

- `docs/report/architecture/002.tonic_component_analysis.md`
- `accel-rpc/tonic/tonic-build/src/client.rs`

- `accel-rpc/tonic/tonic/src/codec/{encode.rs,decode.rs,compression.rs}`

## Expected Output

A report that explains:

- the interception taxonomy inside tonic,
- the generated vs handwritten control flow,
- which hooks are metadata-only versus payload-capable,
- and the recommended insertion points for `accel-codec`, `accel-middleware`, `dsa-ffi`, and `iax-ffi`.
