# Direct Tokio completion-record contract plan

**Date:** 2026-04-28
**Status:** Planned and executed for M007/S01/T01

## Goal

Write the normative design contract for the first direct Tokio IDXD memmove implementation in `idxd-rust`. The contract must be useful to the S02 implementer: it should say how a request becomes operation-owned descriptor/completion state, how ENQCMD submission and backpressure are handled, how a Tokio completion monitor makes progress, and how failures are reported without exposing buffer contents.

## Reader and post-read action

The reader is an internal engineer implementing M007/S02. After reading the report, they should be able to implement the direct submit-now / complete-later memmove path and write host-free tests that prove futures resolve from completion-record observation rather than from the current synchronous worker wrapper.

## Scope

In scope:

- Define the direct completion-driven async memmove architecture.
- Preserve the M006 public request/result constraints: `AsyncMemmoveRequest::new(source: Bytes, destination: BytesMut)`, explicit destination ownership, no allocation convenience constructor, and no public copy-back helper.
- Require dynamically created per-request completion records for v1.
- Require ENQCMD-oriented direct submission with yield/backoff on rejection.
- Specify monitor-driven future signaling, page-fault retry continuation, cancellation/drop semantics, shutdown behavior, and failure taxonomy.
- Name S02 host-free proof obligations.

Out of scope:

- Implementing the runtime path.
- Adding the mechanical verifier script or guard test; those are planned for later S01 tasks.
- Benchmarking, Tonic integration, preallocated completion-record registries, worker batching, or MOVDIR64/MOVDIR64B serialized submission designs.

## Source grounding

The report is grounded in:

- The current async wrapper, which still sends owned requests to a worker thread that calls synchronous `DsaSession::memmove_uninit` before replying.
- The synchronous memmove state machine, which already performs request validation, descriptor fill, completion reset, submission, completion snapshot/classification, page-fault retry, and post-copy validation.
- The validation module, especially `CompletionSnapshot`, `CompletionAction`, `MemmoveRetry`, and typed `MemmoveError` variants.
- The low-level IDXD primitives, especially aligned descriptor/completion wrappers, `submit_enqcmd`, `reset_completion`, `touch_fault_page`, and the current blocking/spinning helpers that are not suitable as direct Tokio monitor primitives.
- C++ prior art showing operation-owned descriptor/completion storage and a polling loop that notifies completed operations.

Codemogger was attempted first for code discovery with query `idxd-rust direct async completion record ENQCMD memmove`, but the local index returned no hits. Execution therefore fell back to direct reads of the known source inputs and targeted exact searches.

## Completion notes

The design report is intentionally narrow. It records that S01 does not add runtime observability, but it requires S02 to expose diagnostics for submission phase, monitor phase, retry count, completion snapshot/status, lifecycle/shutdown classification, and no payload bytes. It also records the local contradiction that the current async implementation is worker-backed and synchronous internally; S02 must replace that target behavior rather than preserve it as the direct async model.
