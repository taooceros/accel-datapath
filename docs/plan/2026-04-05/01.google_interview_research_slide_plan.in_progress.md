## Slide design

### Design goals

- keep the talk conversational and interview-friendly rather than defense-style
- emphasize first-principles motivation, measured insight, and current direction
- avoid dense mechanism-heavy slides unless they directly support the main point
- keep one main takeaway per slide
- use simple diagrams/cards/callouts rather than large tables except for the single main result slide
- surface the main evidence earlier rather than spending too long on setup
- keep claim boundaries precise: measured DSA/stdexec result first, broader generalization only as hypothesis

### Story spine

1. accelerator workloads are naturally asynchronous, but current callback-style APIs are painful to use
2. async abstractions therefore seem like the natural software answer
3. modern accelerators are much faster than the era many async APIs were designed for
4. earlier RDMA work showed that MMIO submission cost could dominate, and batching could amortize it
5. this project asks whether Intel DSA/IAX show the same batching-driven regime change
6. results indicate that they do: once submission is amortized, software/control-path overhead becomes the bottleneck
7. this changes the research question from "how do I use accelerators nicely?" to "are current async frameworks fundamentally suitable for this performance regime?"
8. current work extends that question into richer software stacks such as Tonic/RPC

### Planned slide sequence

1. **Title / thesis**
   - introduce the central claim: batching can make hardware cheap enough that software becomes the bottleneck
   - components to show:
     - title
     - one-sentence subtitle
     - speaker name / date
     - one highlighted thesis callout
2. **The first-principles question**
   - ask whether today’s async APIs are still efficient enough for modern fast accelerators
   - components to show:
     - one large question at the top
     - short bullets for: asynchronous devices, async abstractions, efficiency concern
     - one small takeaway card on why this matters for systems design
3. **Why async looked like the right answer**
    - contrast callback-heavy accelerator APIs with the desired composable async model
    - components to show:
      - left card: current callback/polling style API shape with a tiny pseudo-API example (`submit`, callback, `poll_completion`)
      - right card: desired C++ `stdexec` / Rust `async` style API examples showing composition
      - explicit labels that name the “better API” examples as *C++ sender/receiver (`stdexec`)* and *Rust `async/.await`*
      - short bottom callout: “programmability pressure pushes us toward async abstractions”
4. **Why this question matters now**
   - combine the hardware-regime change and the earlier RDMA batching clue into one slide
   - components to show:
      - two-column comparison: older async regime vs modern accelerator regime
      - RDMA/MMIO precedent in the same slide
      - one visual/cartoon of expensive submission amortized across many ops
      - a takeaway callout connecting regime change to the current project
5. **What I built to test that idea**
   - state the DSA hypothesis and briefly show the benchmarking/abstraction-comparison method
   - components to show:
      - hypothesis sentence
      - explicit comparison dimensions:
        - mock hardware path
        - real DSA path
        - full async framework baseline
        - stripped framework variants
      - short credibility bullets:
        - 8 DSA operations
        - message-rate benchmarking
        - abstraction-layer comparison
6. **Main result**
    - show that batching exposes the same bottleneck shift and summarize measured throughput / per-op deltas
    - components to show:
      - one main result callout sentence
      - one compact table with *matched* baseline / direct / reusable numbers
      - one short note separating the matched `c=2048` point from the hotter `c=32` ceiling result
      - three takeaway cards:
        - measured delta
        - interpretation
        - implication for smaller offloads
7. **What the result does — and does not — say**
    - bound the claim carefully so the talk does not sound broader than the evidence
    - components to show:
      - left card: what the measured result supports
      - right card: what it does not prove
      - one emphasized sentence that this is a measured control-path result, not a universal anti-async claim
8. **How this changed the research direction**
    - reframe the project around async framework suitability, not only accelerator programmability
    - components to show:
      - one large reframed research question
      - two cards:
        - broader systems hypothesis (RDMA / io_uring / NVMe / RPC)
        - project-level implication (stdexec as measurement vehicle, not final destination)
      - short bottom takeaway line
9. **Current status in Tonic/RPC**
     - connect the low-level result to the present direction: payload-path decomposition, copy/CRC/compression focus, and offload crossover analysis
     - components to show:
      - one simplified pipeline graphic: codec → payload transforms → framing → runtime
      - color-coded fit labels:
        - make clear which labels are architecture-guided and which are already measurement-informed
        - codec / buffer handling: architecture-guided
        - copy + CRC: best current target
        - compression / decompression: best current target
        - framing: mixed / case-dependent
        - runtime / tokio control flow: measurement-informed as important
      - two cards:
        - measured now (bounded profiling findings)
        - architecture-guided next steps (what to offload, when offload helps, where software overhead dominates)
