# Plan: FleetBench-Inspired Two-Level Tonic Characterization

**Date**: 2026-04-13
**Status**: in progress

## Goal

Evolve `tonic-profile` from a bytes-payload decomposition harness into a **two-level characterization harness**:

1. **High-level decomposition** for understanding which Tonic/RPC stage owns cost: encode/decode, copy/buffer lifecycle, compression/decompression, framing/transport glue, runtime/queueing, endpoint directionality, and tails.
2. **Low-level CPU/code-path characterization** for understanding why the CPU behaves that way: instruction count, IPC, topdown retiring/backend/frontend/memory-bound behavior, cache/memory indicators, branch behavior, flamegraph hot symbols, and realistic protobuf-shape sensitivity.

The key design principle is that these two levels must be explicitly mapped to each other. Stage timers identify **where** the cost belongs; CPU evidence explains **why** the machine spends cycles there.

## Background

This plan follows:

- characterization refinement plan: `docs/plan/2026-04-12/02.tonic_characterization_plan.in_progress.md`
- first Phase-A results: `docs/report/benchmarking/012.tonic_characterization_refinement_results.md`
- FleetBench intake: `docs/report/benchmarking/013.fleetbench_rpc_characterization_intake.md`
- current bytes-only harness: `accel-rpc/tonic-profile/src/main.rs`

The prior Phase-A run showed that the current Tonic timers are useful diagnostically but too intrusive for final throughput in larger/high-concurrency regimes. It also showed that bytes-only payloads emphasize copy/buffer/compression behavior but do not fully exercise realistic protobuf schema traversal.

FleetBench contributes the missing workload-realism ideas:

- generated protobuf program shapes instead of one flat `bytes` payload,
- asymmetric request/response shape selection,
- pre-generated message buffers,
- closed-loop outstanding-RPC generation,
- request/server delay distributions,
- peer / connection topology controls,
- and CPU-oriented benchmarking with external counters/profiles.

## Core thesis

We need **both** views:

- **Decomposition** gives higher-level RPC understanding: which stage or design decision is responsible.
- **CPU characterization** gives lower-level system understanding: whether that stage is instruction-heavy, memory-bound, branch-sensitive, allocator-heavy, scheduler-sensitive, or transform-compute-heavy.

Neither view is sufficient alone:

- Stage timers without CPU evidence can mislabel a bucket without explaining machine behavior.
- CPU profiles without stage decomposition can show hot symbols without telling whether the actionable lever is codec design, buffer lifecycle, compression policy, runtime policy, or endpoint asymmetry.

## Target output from this plan

The final report after implementation should be able to say things like:

- “The Fleet/P10-like response-heavy shape increases decode/encode stage time and shifts perf from pure memory movement toward generated prost traversal; copy remains visible only after response sizes exceed X.”
- “At 64 KiB bytes payloads the profile is memory-bound and led by `memmove`/`BytesMut`; at comparable serialized proto shape the profile adds instruction-retiring generated encode/decode work.”
- “Pooled buffering improves 4 KiB byte mode but helps less in string-heavy proto-shape mode, so the buffer-control conclusion is workload-shape sensitive.”
- “A p99 regression under poisson server delay matches configured service-time variance and should not be blamed on transport framing.”
- “The remaining offload candidate is copy/buffer movement for large response-heavy regimes after software pooling, not generic protobuf traversal or control-path scheduling.”

## Workstreams

## Workstream A — Add FleetBench-inspired protobuf shape mode

### Objective

Add a Tonic-native `unary-proto-shape` mode that exercises realistic protobuf field traversal instead of only encoding a single bytes field.

### Requirements

Add a second unary RPC shape beside the current bytes echo path:

```text
--rpc unary-bytes        # current EchoRequest { bytes payload }
--rpc unary-proto-shape  # new nested protobuf request/response mode
```

Add shape controls:

```text
--proto-program small | moderate | repeated | string-heavy | fleet-response-heavy
--proto-message-policy cycle | random
--response-program same | small | moderate | repeated | string-heavy | fleet-response-heavy
```

### Initial schema subset

Do not copy all FleetBench-generated proto files at first. Implement a compact Tonic-native subset that preserves the important shape classes:

| Shape | Purpose | FleetBench inspiration |
|---|---|---|
| `small` | fixed overhead / small schema traversal | P6-like small request/response |
| `moderate` | nested + repeated but not huge | P0/P2-like moderate shape |
| `string-heavy` | prost string/bytes traversal and compressibility | P3/P7-like string-heavy request |
| `repeated-heavy` | repeated fields / packed fields / vector traversal | P2/P5-like repeated-heavy shape |
| `fleet-response-heavy` | asymmetric response-heavy program | P10 / Fleet-like response-heavy side |

### Implementation notes

- Add new messages to `accel-rpc/tonic-profile/proto/profile.proto` or a sibling proto file.
- Keep the current bytes echo path intact for matched comparisons.
- Pre-generate a small pool of request and response messages per shape, analogous to FleetBench’s `3 message variants * 4 setter variants` pattern.
- Add metadata for:
  - request shape,
  - response shape,
  - request serialized bytes,
  - response serialized bytes,
  - message policy (`cycle` vs `random`),
  - shape class (`small`, `moderate`, `string-heavy`, etc.).

### Acceptance criteria

- `cargo build --release -p tonic-profile` passes.
- `unary-bytes` behavior remains backward compatible for existing runs.
- `unary-proto-shape` can run at least one selftest and one split client/server test.
- JSON output reports request/response serialized sizes and shape metadata.

## Workstream B — Add FleetBench-style load shaping

### Objective

Separate “how many tasks are running” from “what load is being generated.” The current `--concurrency` knob is useful, but FleetBench shows that closed-loop outstanding RPCs, connections per peer, and delay distributions should be explicit.

### New controls

```text
--load-mode closed-loop | fixed-window
--max-outstanding-rpcs N
--connections-per-peer N
--peer-count N
--client-delay-dist constant=0 | uniform=0:50 | poisson=20:max=200 | empirical=0,0,10,100
--server-delay-dist constant=0 | uniform=0:50 | poisson=20:max=200 | empirical=0,0,10,100
```

### Semantics

- `closed-loop`: each completed RPC schedules the next RPC until the measurement window ends.
- `fixed-window`: current behavior; workers loop until deadline or request target.
- `max-outstanding-rpcs`: total in-flight budget for closed-loop mode.
- `connections-per-peer`: number of channels/stubs per peer.
- `peer-count`: initially local endpoints only; useful metadata before true multi-peer support.
- delay distributions:
  - client delay = inter-RPC delay after completion and before next send,
  - server delay = service-time delay before response completion.

### Acceptance criteria

- JSON reports load mode and delay distribution descriptions.
- Delay distribution sample mean is recorded or reproducible from metadata.
- Configured delay is not counted as payload-path cost in report interpretation.

## Workstream C — Lower-overhead and endpoint-aware decomposition

### Objective

Make stage decomposition usable for matched comparison without distorting throughput as much as report 012 observed.

### Tasks

1. Replace hot-path shared atomic updates with lower-overhead accumulation where practical:
   - endpoint-local counters,
   - task/thread-local counters,
   - or batched per-phase flush.
2. Add split-process server-side stage snapshot output:
   - client report should include client-side stages,
   - server report should include server-side stages,
   - merged analysis should show direction-specific attribution.
3. Keep instrumentation-off control mandatory for throughput comparisons.

### Acceptance criteria

- Instrumentation-on/off overhead is explicitly measured for tiny, medium, and large matched points.
- Endpoint-specific stage snapshots exist for at least one split client/server run.
- Report labels stage data as diagnostic if overhead remains high.

## Workstream D — Define the high-level-to-low-level mapping layer

### Objective

Create a standard interpretation table that maps decomposition buckets to CPU evidence.

### Mapping table

| High-level bucket | CPU question | Evidence | Low-level signature | Decision rule |
|---|---|---|---|---|
| Serialization / deserialization | Is prost walking fields/nested messages/string fields? | encode/decode timers, prost/generated-message flamegraph frames, instructions/RPC, IPC, branch misses | retiring-heavy or branch-sensitive generated encode/decode work | If shape changes timers and generated frames without proportional byte-size change, classify as schema traversal. |
| Copy / buffer lifecycle | Is cost byte movement or growth/reallocation? | body/buffer timers, `memmove`, allocator, `BytesMut`, `RawVec`, backend/memory-bound topdown, LLC/cache metrics | low IPC, high memory/backend-bound, hot copy/grow symbols | If pooling/copy-min changes this, classify as software buffer-policy before offload. |
| Compression / decompression | Is transform compute dominating? | compression timers, gzip/miniz/zstd frames, IPC/topdown, compression ratio | high retiring and hot compression frames, or random-payload cost explosion | Only consider offload for compressible regimes where transform remains stable after controls. |
| Framing / transport glue | Is fixed per-RPC h2/hyper/tonic overhead visible? | frame/body handoff timers, h2/hyper/tonic flamegraph frames, instructions/RPC at tiny payload | fixed per-RPC overhead, weak payload-size scaling | Treat as control path unless it scales with bytes or batching. |
| Runtime / queueing | Is scheduling/load shape driving throughput or tails? | instrumentation-off throughput, p50/p95/p99, tokio scheduler/waker frames, async-bench, context switches, load metadata | high tail variance, runtime-mode sensitivity, scheduler frames | Treat as runtime policy if changing load shape shifts results without matching stage shift. |
| Application/service delay | Is observed latency configured by workload model? | client/server delay metadata, outstanding RPCs, latency histogram | p99 tracks configured delay distribution, stage timers small | Do not attribute configured delay to transport/payload stages. |
| Direction asymmetry | Is one endpoint or direction dominant? | split client/server stage snapshots, request/response serialized sizes, endpoint flamegraphs | server-heavy or client-heavy hot paths aligned with schema sizes | Require endpoint split before claiming direction-specific offload target. |

### Required report pattern

Each representative point should include a paired row:

```text
High-level bucket: <bucket>
Decomposition evidence: <stage timers / variant deltas>
CPU evidence: <IPC/topdown/flamegraph/hot symbols>
Mapping decision: <why CPU evidence supports/qualifies bucket attribution>
Action: <software cleanup, more measurement, or offload follow-up>
```

## Workstream E — Paired experiment matrix

### Objective

Run paired CPU and decomposition experiments so the two views can validate each other.

### Initial matrix

Keep the first run bounded:

| Axis | Values |
|---|---|
| RPC mode | `unary-bytes`, `unary-proto-shape` |
| proto shape | `small`, `moderate`, `string-heavy`, `fleet-response-heavy` |
| concurrency / outstanding | `1`, `32`, `128` |
| runtime | `single`, `multi` |
| compression | off; on only for `string-heavy` and `fleet-response-heavy` |
| buffer policy | default, pooled |
| instrumentation | off for CPU run, on for decomposition run |
| placement | selftest smoke first, then split client/server for representative points |

### CPU run

- instrumentation off,
- representative matched points only,
- `perf stat -d`, flamegraph, and optionally `perf record`,
- report IPC, topdown, memory/cache, hot symbols, throughput, and tails.

### Decomposition run

- instrumentation on,
- same matched configuration,
- lower-overhead counters where possible,
- report stage timers, counts, bytes, endpoint direction, and instrumentation overhead.

### Acceptance criteria

A report can compare bytes vs proto-shape mode and answer:

> Does realistic protobuf shape shift the offload-readiness ranking from copy/buffer first toward serialization/schema traversal in any regime, or does copy/buffer remain dominant after shape realism is added?

## Risks and controls

1. **Schema copy risk**
   - Do not blindly copy the entire FleetBench generated proto tree into the repo.
   - Start with compact FleetBench-inspired shapes.

2. **Attribution distortion**
   - Keep instrumentation-off runs as the throughput baseline.
   - Use instrumentation-on only for decomposition unless overhead is proven small.

3. **Confusing configured delay with RPC overhead**
   - Always report delay distributions in metadata.
   - Exclude configured delay when interpreting payload-path stages.

4. **Endpoint conflation**
   - Selftest is acceptable for smoke tests.
   - Direction-specific offload claims require split client/server snapshots.

5. **Overclaiming FleetBench equivalence**
   - Call the new mode FleetBench-inspired, not FleetBench-equivalent, unless the exact generated schemas/distributions are imported and validated.
