# Plan: Pre-Advisor Tonic Characterization Priorities

**Date**: 2026-04-13
**Status**: in progress
**Priority**: high

## Goal

Prepare a bounded, advisor-ready follow-up for the active Tonic/gRPC characterization thread.

The immediate outcome for this pass is not a broad new implementation wave. It is a sharper pre-next-meeting package that:

1. states a clearer RPC taxonomy above raw byte-level profiling,
2. explains which workload dimensions matter beyond message size,
3. tightens the next measurement lane so the evidence is stronger,
4. shows how FleetBench-inspired realism complements Tonic-specific decomposition,
5. and states an opinionated controller-model question for accelerator integration.

## Why this plan is needed now

The recent advisor conversation changed the immediate priority.

The open issue is no longer just whether the repo can produce more profiling output. The issue is whether the work can be framed as a convincing systems question:

- what are the meaningful stages in the life of an RPC,
- which workload dimensions change the dominant cost,
- what should be measured before making offload-readiness claims,
- and how should accelerator control interact with a multithreaded async RPC runtime.

The current repo artifacts already contain most of the ingredients, but they are spread across measurement, literature, and architecture notes. This pass packages them into a single high-priority pre-meeting plan.

## Main pre-meeting question

For unary Tonic RPCs, which cost bucket dominates which workload regime after software controls, and what evidence is still required before claiming a later DSA/IAX follow-up is justified?

## Out of scope for this pass

This plan is intentionally bounded. It does not attempt to:

- finish the full FleetBench-inspired implementation roadmap,
- expand to a full streaming characterization pass,
- implement accelerator integration,
- or define the complete thesis roadmap.

## Evidence contract for this pass

The pre-meeting package should only make claims that are supported by one of the following:

1. existing measurement evidence from report `012`,
2. explicit repo-grounded planning from the two-level characterization plan,
3. local literature-backed framing from report `007`,
4. or clearly labeled proposed next-step measurements.

If a claim requires lower-overhead split client/server attribution that does not yet exist, it must be stated as a next measurement requirement rather than as a settled result.

## Working framing

### 1. Higher-level RPC taxonomy

The next meeting should not frame the problem only as `memmove` vs protobuf vs compression. The work should use a higher-level decomposition:

- request/response shape and endpoint asymmetry,
- runtime / queueing / scheduling,
- serialization / deserialization,
- copy / buffer lifecycle,
- compression / decompression,
- framing / transport glue,
- configured service delay vs measured RPC-path cost,
- and tail behavior.

This preserves the repo's bucketed Tonic characterization while answering the advisor's request for a clearer life-of-an-RPC model.

### 2. Workload dimensions beyond message size

The bounded next-pass matrix should add dimensions beyond payload bytes alone:

- protobuf schema shape,
- request/response asymmetry,
- compressibility,
- concurrency and outstanding RPC pressure,
- runtime mode,
- connection / peer topology where relevant,
- and optional client/server delay distributions for queueing interpretation.

The point is not to make the matrix huge. The point is to avoid a size-only story.

### 3. Why FleetBench helps

FleetBench helps because it contributes the workload-realism mechanisms missing from the current bytes-only path:

- realistic protobuf program shapes rather than a single flat bytes payload,
- asymmetric request/response shape selection,
- closed-loop outstanding-RPC generation,
- delay distributions,
- and explicit connection/topology controls.

FleetBench should not replace Tonic decomposition. The repo should use a **two-level** method:

1. **CPU/code-path characterization** for realistic gRPC/protobuf execution behavior,
2. **Tonic stage decomposition** to map that behavior back to actionable buckets.

## Pre-meeting deliverables

Before the next advisor meeting, this thread should produce three concrete artifacts:

1. **A one-page RPC taxonomy and regime note**
   - defines the high-level buckets,
   - names the workload dimensions beyond size,
   - and states which conclusions are already supported vs still provisional.

2. **A tightened next-pass measurement plan**
   - separates instrumentation-off CPU runs from instrumentation-on decomposition runs,
   - requires lower-overhead endpoint-local accounting for stronger claims,
   - and names a small representative regime matrix.

3. **A short controller-model note**
   - states where accelerator control should conceptually live,
   - names the competing options,
   - and frames the open design question for advisor feedback.

## Workstreams

### Workstream A — Lock the advisor-facing question and claim boundaries

#### Objective

Convert the current active thread into one bounded question with explicit claim limits.

#### Tasks

1. State the main pre-meeting question in the opening section of the follow-up note.
2. Distinguish already-supported findings from next-step hypotheses.
3. Mark split client/server attribution and lower-overhead timers as prerequisites for stronger endpoint-specific conclusions.

#### Success criteria

- The pre-meeting note opens with one question.
- Every major claim is either evidence-backed now or labeled as requiring the next pass.

### Workstream B — Tighten the taxonomy and bounded regime matrix

#### Objective

Replace a size-only framing with a small but more realistic workload model.

#### Tasks

1. Keep the existing high-level Tonic buckets.
2. Add a bounded set of realism dimensions: schema shape, asymmetry, load mode, outstanding RPCs, and compressibility.
3. Define a small representative regime matrix for the next pass.

#### Recommended initial matrix

| Axis | Initial values |
|---|---|
| RPC mode | `unary-bytes`, `unary-proto-shape` |
| shape | `small`, `moderate`, `string-heavy`, `fleet-response-heavy` |
| runtime | `single`, `multi` |
| concurrency / outstanding | `1`, one medium point, one higher-pressure point |
| compression | off by default; on only for structured/string-heavy representative points |
| attribution mode | instrumentation off for CPU runs, on for paired decomposition runs |

#### Success criteria

- The next-pass plan is not size-only.
- The matrix remains bounded enough for a pre-meeting update.

### Workstream C — Tighten the next measurement lane

#### Objective

Define what evidence should be gathered next and in what order.

#### Tasks

1. Treat current internal timers as diagnostic attribution mode outside the tiny single-thread point.
2. Prioritize lower-overhead endpoint-local accumulation.
3. Prioritize split client/server snapshots before stronger endpoint claims.
4. Pair each representative point with:
   - one instrumentation-off CPU run,
   - one matched decomposition run,
   - and one short interpretation row linking stage bucket to CPU behavior.

#### Required interpretation pattern

Each representative point should support a statement of the form:

```text
Bucket: copy / buffer lifecycle
High-level evidence: body_accum + encode/decode remain large
Low-level evidence: low IPC, memory-bound, memmove / BytesMut / allocator hot
Conclusion: software buffer movement dominates; software cleanup precedes hardware claim
```

#### Success criteria

- The measurement lane explicitly separates throughput evidence from decomposition evidence.
- The next-step measurement order is low-distortion first, attribution second.

### Workstream D — Write the literature-backed framing

#### Objective

Support the framing with a small, disciplined set of citations.

#### Tasks

1. Use cloud-scale RPC characterization to justify queueing/tax and workload-mix framing.
2. Use the protocol-buffers accelerator paper to justify protobuf as its own cost bucket.
3. Use the FleetBench intake to justify realistic shape and load-shaping dimensions.
4. Keep the literature role limited: it should justify the buckets and workload axes, not replace Tonic-specific evidence.

#### Success criteria

- Each framing claim maps to a local literature artifact.
- The note clearly states the repo's remaining gap: no existing source replaces a Tonic-specific two-level measurement pass.

### Workstream E — State an opinionated controller-model question

#### Objective

Turn the advisor's controller discussion into a concrete design question.

#### Working position

The current Tonic stack should remain the control plane. Accelerator-eligible work should be limited to measured data-path boundaries such as copy/buffer movement and compression-like transforms, not general runtime or protocol-state logic.

The open controller-model question for the next meeting should be phrased as:

> If each accelerator has its own enqueue/control boundary, where should fairness and priority policy live: a dedicated controller thread, cooperative shared control among worker threads, or a thinner software admission layer on top of hardware queue concurrency?

#### Input sources for this question

- the advisor discussion,
- `dsa-stdexec` controller-like queue/submission patterns,
- `hw-eval` raw work-queue behavior,
- and the current Tonic runtime orchestration path.

#### Success criteria

- The follow-up note contains one explicit controller-model question.
- It presents 1–2 plausible control options and one working preference.
