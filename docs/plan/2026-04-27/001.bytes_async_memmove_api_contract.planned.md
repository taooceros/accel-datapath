# Bytes-based async memmove API contract plan

Status: planned
Date: 2026-04-27

## Reader and post-read action

Reader: an internal executor or reviewer preparing the M006 async memmove migration.

After reading this plan, the reader should be able to review the companion architecture contract and understand why S02 must migrate the async public API to owned `bytes` buffers without adding allocation conveniences or copy-back helpers.

## Expected executor skills

- `codemogger` for targeted discovery of the current async memmove surface before trusting stale planning notes.
- `unsafe-checker` for framing the `BytesMut` spare-capacity and future unsafe helper invariants.
- `write-docs` for structuring a reader-oriented contract that a fresh executor can implement from.

## Goal

Write a normative, implementation-aware API contract for the M006 async memmove migration. The contract must name the bytes-based public request/result shape, preserve existing failure observability, and separate first-version inline hardware submission from future batching or borrowed zero-copy work.

## Local source delta noted during planning

The current async implementation still uses `Vec<u8>` in `AsyncMemmoveRequest`, exposes `copy_exact`, `copy_into`, `source_bytes`, and `destination_bytes`, and still provides `AsyncDsaHandle::memmove_into` as a borrowed copy-back convenience. The current implementation also routes async work through a single worker-owned session and classifies owner shutdown separately from worker/channel failures and wrapped `MemmoveError` failures.

The architecture contract treats those current public helpers as stale migration surfaces to remove or keep private in S02. It preserves the diagnostic taxonomy already present in the current tests.

## Scope

In scope:

- Document `AsyncMemmoveRequest::new(source: Bytes, destination: BytesMut)` as the only canonical public constructor.
- Document `AsyncMemmoveResult { destination: BytesMut, report: MemmoveValidationReport }` as the explicit result shape.
- Document validation semantics for zero-length sources, too-small destinations, oversized destination tails, and validation-before-enqueue.
- Document `BytesMut` initialized-length and spare-capacity rules, including the invariants required before any future unsafe-backed borrowed zero-copy helper is accepted.
- Document the first implementation path as inline `enqcmd` hardware submission, allowing multiple submitting threads under the `enqcmd` model.
- Explicitly defer software aggregation, batching, and `movdir64` support.
- Provide S02 verification targets for `idxd-rust/tests/async_memmove_contract.rs` and `idxd-rust/tests/tokio_handle_contract.rs`.
- Add a mechanical verifier for the contract document required by the slice-level checks.

Out of scope:

- No Rust implementation changes in this task.
- No generic public `Buf` / `BufMut` request type.
- No allocation convenience API under a new name.
- No public borrowed-destination `memmove_into` helper in the post-migration surface.
- No requirement to implement software aggregation, batching, or `movdir64` support in M006.
